## Summary

IronCache's central runtime decision is work-stealing (tokio) vs thread-per-core / shared-nothing (glommio, monoio). The thread-per-core lineage traces to Seastar/ScyllaDB and is now the default for the highest-performance caches: Dragonfly shards its in-memory DB into N parts (k = HASH(KEY) % N, N <= thread count), binds each shard to exactly one thread, and forbids mutexes entirely — "Inter-thread interactions occur only via passing messages from thread to thread." Datadog's Glommio post makes the load-bearing argument crisply: with a single thread of execution per core, "locks are never necessary" because all updates to a shard are naturally serialized; the cost is that any cross-shard work must be explicit message passing, and a context switch (~5us) is more expensive than an io_uring I/O (<4us). For a Redis-wire cache whose dominant ops are single-key GET/SET, the keyspace partitions cleanly per core, so shared-nothing is the natural fit and the strongest lever for "max throughput per core."

The runtime tradeoff is concrete. tokio (1.52.3, MSRV 1.71) uses a multi-threaded work-stealing scheduler over epoll/kqueue (readiness model: two user/kernel transitions per I/O). It is the mature, portable, ecosystem-dominant default — but work-stealing forces every shared structure to be Send+Sync and re-introduces synchronization (atomics, cross-core cache-line bouncing) that thread-per-core eliminates. monoio (0.2.4, ByteDance) and glommio (0.9.0, Datadog, MSRV 1.70) are thread-per-core io_uring runtimes using a completion model. ByteDance's own benchmark claims monoio reaches ~2x tokio peak at 4 cores and ~3x at 16 cores, with the caveat that at single-core / very-few-connections monoio latency is worse than tokio (io_uring vs epoll). Glommio uses three io_uring rings per thread (main, latency, poll) to isolate NVMe polling. The key downside of both: io_uring is Linux-only (kernel 5.6+ for the basic ops; monoio falls back to epoll/kqueue below that), futures are !Send so the programming model differs from tokio, and the ecosystem (drivers, libraries) is far thinner.

io_uring itself is the I/O substrate worth borrowing. The io-uring crate (0.7.12) is the low-level binding; tokio-uring exists but requires kernel 5.11+. The high-value features map to specific kernels: registered/fixed buffers and SQPOLL (kernel-side submission polling, no submit syscall) for hot-path zero-syscall I/O; multishot accept (5.19) and multishot recv with ring-provided buffers (6.0) to amortize SQE submission across many connections — exactly the shape of a cache accepting thousands of short-lived or long-lived connections. The design tension is buffer ownership: completion-model I/O requires buffers to outlive the kernel operation, which forces owned-buffer APIs (IoBuf/IoBufMut) rather than borrowed &mut [u8], shaping the entire networking layer.

On the data structure: if IronCache goes shared-nothing, the per-shard store can be a plain single-threaded hashbrown HashMap with no synchronization at all — the biggest win. If it instead uses a shared concurrent map (simpler connection model, work-stealing), the candidates are dashmap 7.0.0-rc2 (sharded RwLock<HashMap>, CachePadded to avoid false sharing, default shards = available_parallelism()*4 rounded to power of two), scc 3.4.8 (finer-grained bucket locks, best for write-heavy), papaya 0.2.4 (lock-free, open-addressing Swiss-table-style with an 8-bit metadata array, incremental resize, hyaline reclamation via seize 0.5 — built for read-heavy), and flurry (Java CHM port, lock-free reads, but allocator pressure). For a cache the workload is read-heavy with eviction churn, so papaya/scc beat a coarse RwLock<HashMap>, but a partitioned single-thread map beats all of them by avoiding atomics entirely. Memory reclamation matters: epoch-based reclamation (crossbeam-epoch) can stall reclamation behind a slow thread, whereas hyaline/seize gives more predictable GC — relevant because a cache must bound memory tightly.

Recommendation for IronCache: adopt the shared-nothing / shard-per-core model (borrow from Seastar/ScyllaDB/Dragonfly) as the architecture, with io_uring (borrow) as the Linux I/O path. Choose between monoio and glommio as the thread-per-core runtime via a prototype bake-off, and keep a tokio+epoll fallback path for portability and pre-5.6 kernels (adapt). Reject a single global RwLock<HashMap> and reject work-stealing as the primary model. Per-shard storage should be a single-threaded map; a shared concurrent map (papaya for read-heavy) is the fallback only if the connection-affinity model proves too costly. The open risk is multi-key/cross-shard commands (MGET, MSET, transactions, pub/sub fan-out) which shared-nothing makes harder — Dragonfly's coordinator-fiber pattern is the reference solution.

## Mechanisms: borrow, adapt, or reject

| Mechanism | System | Stance | What it does | Rationale for IronCache |
| --- | --- | --- | --- | --- |
| Thread-per-core / shared-nothing sharding | Seastar / ScyllaDB / Dragonfly | **borrow** | Pin one thread per core, partition the keyspace into N shards (k = HASH(KEY) % N), and let exactly one thread own each shard. All cross-shard work is explicit message passing; no shared mutable state, so no locks/atomics on the hot path. | For a Redis-wire cache the dominant ops (GET/SET) are single-key and partition cleanly per core. Eliminating atomics and cross-core cache-line bouncing is the single biggest lever for 'max throughput per core', and it is the proven architecture of the fastest caches (Dragonfly) and databases (ScyllaDB). |
| Work-stealing multi-threaded scheduler | tokio | **reject** | A pool of worker threads each with a local run queue; idle workers steal tasks from busy workers' queues to balance load. Tasks must be Send so they can migrate across threads. | Work-stealing requires every shared structure to be Send+Sync, forcing locks/atomics and re-introducing the cache-coherency cost shared-nothing avoids. It optimizes for uneven CPU-bound task graphs, not for uniform per-key cache ops. Keep tokio only as a portability fallback runtime, not the primary model. |
| io_uring completion-model I/O | Linux io_uring (io-uring crate 0.7.12) | **borrow** | Submission/completion ring buffers shared with the kernel; submit batches of I/O and reap completions, avoiding a readiness syscall per op. Completion model means buffers must outlive the in-flight op (owned-buffer APIs). | Halves syscalls vs epoll and enables zero-syscall hot paths; it is the substrate behind monoio/glommio. The owned-buffer constraint shapes the networking layer but is acceptable for the Linux fast path. |
| SQPOLL (kernel submission polling) | Linux io_uring | **adapt** | A kernel thread polls the submission queue, so userspace can submit I/O without any io_uring_enter syscall, trading a busy kernel thread for lower latency. | Attractive for ultra-low-latency single-tenant deployments, but it burns a core and hurts efficiency-per-core under low load. Make it an opt-in flag, not the default, given IronCache's efficiency goal. |
| Multishot accept + multishot recv with ring-provided buffers | Linux io_uring (kernel 5.19 / 6.0) | **adapt** | A single accept SQE posts a CQE per new connection; a single recv SQE repeatedly posts CQEs pulling buffers from a registered provided-buffer pool. Amortizes SQE submission across many connections/reads. | Ideal for a cache fronting thousands of connections, but require kernel 6.0; gate behind feature detection and fall back to one-shot ops on older kernels. Adopt registered/fixed buffers regardless for zero-copy hot paths. |
| Per-shard single-threaded hashbrown HashMap | std/hashbrown (no concurrency wrapper) | **borrow** | Under shared-nothing, each shard's store is an ordinary single-threaded SwissTable HashMap with no locks, no atomics, no reclamation overhead. | This is the payoff of shared-nothing: the store needs zero synchronization, beating any concurrent map. Pair with an intrusive LRU/eviction structure also owned by the shard thread. |
| Sharded RwLock<HashMap> concurrent map | dashmap 7.0.0-rc2 | **reject** | Array of CachePadded<RwLock<HashMap>> shards (default available_parallelism()*4, power-of-two); a key hashes to a shard and takes that shard's RwLock. CachePadded avoids false sharing. | Only relevant if we abandon shard-per-core. Even then a coarse RwLock per shard serializes writers within a shard and re-introduces lock contention under eviction churn; papaya/scc dominate it. Reject as the primary store; acceptable only for low-contention side metadata. |
| Lock-free open-addressing map with hyaline reclamation | papaya 0.2.4 + seize 0.5 | **adapt** | Swiss-table-style open addressing with an 8-bit metadata array for cache-efficient probing, atomic table-pointer swaps for incremental resize, and hyaline-based memory reclamation (seize) so readers never block writers. | The best shared-map fallback for IronCache's read-heavy profile if the connection-affinity model proves too costly. Adapt because a cache also needs bounded memory and TTL/eviction, which papaya does not provide out of the box. |
| Aggressive bucket-lock sharding | scc 3.4.8 | **reject** | Concurrent HashMap that shards locks at bucket granularity, far finer than dashmap, giving strong write-heavy throughput. | Write-heavy specialization is not IronCache's primary need (caches are read-heavy), and it still pays atomics/locks that shard-per-core avoids. Keep as a benchmark baseline only. |
| Epoch-based reclamation (EBR) | crossbeam-epoch | **adapt** | Defers freeing memory until all threads have advanced past the epoch in which it was unlinked, giving safe lock-free reclamation. | Workable but a stalled/slow thread can pin an epoch and stall reclamation, growing memory unboundedly — bad for a memory-bounded cache. Prefer hyaline/seize where lock-free reclamation is needed; under shard-per-core, reclamation is moot for the main store. |
| seqlock + cache-line padding for hot read-mostly state | general (crossbeam CachePadded, std AtomicU64) | **borrow** | A seqlock lets readers retry-read versioned data without taking a lock; CachePadded aligns hot atomics to a cache line to prevent false sharing between cores. | Useful for shared read-mostly state that must cross shards (config, stats counters). Cheap, standard, and complements shared-nothing for the rare genuinely-shared datum. |

## Implications for IronCache

- Adopt shard-per-core as the core architecture: pin one OS thread per core, partition the keyspace by k = HASH(KEY) % N (N = core count), and have each thread exclusively own its shard's HashMap + eviction structures so the GET/SET hot path takes zero locks and zero atomics.
- Make the runtime a swappable trait/abstraction so a prototype bake-off between monoio (0.2.4) and glommio (0.9.0) is cheap, with tokio (1.52.3) + epoll retained as the portability fallback for non-Linux and pre-5.6 kernels.
- Treat io_uring as Linux fast path only; require kernel 5.6+ for basic ops and feature-detect at startup for multishot accept (5.19) and multishot recv + ring-provided buffers (6.0), falling back to one-shot ops or epoll when absent.
- Build the networking layer around owned buffers (IoBuf/IoBufMut style) and registered/fixed buffers from day one, since the io_uring completion model forbids borrowing &mut [u8] across an in-flight op.
- Solve cross-shard / multi-key commands (MGET, MSET, MSETNX, transactions, pub/sub fan-out) with a Dragonfly-style coordinator: pin each connection to one thread's fiber that dispatches per-shard sub-commands and reassembles results, providing atomicity without a global lock.
- Bind each client connection to a single thread for its lifetime; if a request touches a remote shard, route via a bounded SPSC/MPSC message channel rather than reaching into another thread's map.
- If a shared concurrent map is ever needed (e.g. abandoning connection affinity), default to papaya 0.2.4 (read-heavy, hyaline reclamation) not dashmap's RwLock<HashMap>; reserve dashmap/scc as benchmark baselines.
- Cache-line-pad (CachePadded) any genuinely shared atomics (global stats, epoch counters) and use seqlocks for read-mostly shared config to avoid false sharing across cores.
- Make SQPOLL an opt-in latency mode, not default, because a dedicated kernel poll thread harms throughput-per-core under low/moderate load — directly counter to IronCache's efficiency goal.
- Document the kernel-version support matrix prominently in the README, since the single-static-binary promise collides with io_uring feature fragmentation across 5.6 / 5.11 / 5.19 / 6.0.

## Research papers and primary sources

- **The Impact of Thread-Per-Core Architecture on Application Tail Latency** (ANCS 2019). Empirically compares thread-per-core (shared-nothing) against work-stealing/shared designs, showing thread-per-core can improve tail latency substantially by eliminating cross-core synchronization and migration. [source](https://helda.helsinki.fi/items/2c5bb1c8-9c33-4530-8a40-1be2a5b66b5e) Relevance: Provides the academic backing for choosing shard-per-core over work-stealing for a latency-sensitive cache; the '71% tail-latency' figure Datadog cites traces here.
- **Scalable Kernel TCP Design and Implementation for Short-Lived Connections (MegaPipe / affinity-accept lineage)** (ASPLOS 2012). Per-core connection acceptance and CPU-affine connection handling remove accept-path contention and improve locality for many short-lived connections. [source](https://dl.acm.org/doi/10.1145/2150976.2150984) Relevance: Directly informs IronCache's accept/connection model: bind connections to the accepting core and use multishot accept to scale connection setup.
- **Efficient Lock-Free Durable Sets / Hyaline: Fast and Transparent Lock-Free Memory Reclamation** (PODC 2019 / OOPSLA). Hyaline reclamation uses reference-counted retirement batches that are robust to stalled threads, avoiding the unbounded-garbage problem of epoch-based reclamation. [source](https://arxiv.org/abs/1905.07903) Relevance: Justifies papaya/seize's reclamation choice over crossbeam-epoch for any lock-free path in a memory-bounded cache.
- **MICA: A Holistic Approach to Fast In-Memory Key-Value Storage** (NSDI 2014). Partitions data across cores with exclusive (EREW) access, uses kernel-bypass NIC queues steered per-core, achieving tens of millions of ops/sec by avoiding cross-core sharing. [source](https://www.usenix.org/conference/nsdi14/technical-sessions/presentation/lim) Relevance: Canonical proof that shard-per-core + per-core NIC steering is the path to maximum KV throughput per core — the IronCache north star.
- **Scaling Memcached at Facebook** (NSDI 2013). Real-world memcached scaling: lock granularity, slab allocation, and connection handling tradeoffs under read-heavy production load. [source](https://www.usenix.org/conference/nsdi13/technical-sessions/presentation/nishtala) Relevance: Grounds expectations about read-heavy cache workload mixes and the operational cost of lock contention that shard-per-core removes.

## Open questions

- Does shard-per-core (monoio/glommio) actually beat tokio+epoll+papaya for IronCache's specific GET/SET mix on representative hardware, or only at high core counts? Needs an in-house bake-off, since vendor benchmarks are not apples-to-apples.
- How costly is the cross-shard coordinator path for MGET/MSET/transactions/pub-sub, and at what fraction of multi-key traffic does shared-nothing stop winning vs a shared concurrent map?
- monoio vs glommio: which has the more maintainable API, better Linux-kernel feature coverage (multishot, fixed buffers), and a thinner unsafe surface for a production cache? Both have small ecosystems.
- How do we keep a true single-static-binary promise while supporting both io_uring (Linux 5.6+/6.0) and an epoll/kqueue fallback, including non-Linux dev machines? Compile-time vs runtime selection?
- What is the per-shard memory bookkeeping cost for tight memory bounds, and does owning eviction (LRU/LFU/W-TinyLFU) per shard skew hit rates vs a global eviction policy?
- Should connection-to-shard routing rebalance on hot keys (shard imbalance), and if so how, without re-introducing cross-core sharing? (ScyllaDB's 'hot shard' problem.)
- Does SQPOLL or fixed-buffer registration interact badly with many shards each owning a ring (memlock limits, kernel thread count)?

## Proposed issues (seeds for the tracker)

- **[decision, M1]** DECISION: Shard-per-core / shared-nothing as IronCache's core concurrency model. Adopt thread-per-core with per-core keyspace sharding (k = HASH(KEY) % N) and no shared mutable hot-path state, following Seastar/ScyllaDB/Dragonfly, in preference to a work-stealing + shared-concurrent-map design. Record the rationale and the rejected alternatives.
- **[research, M0]** RESEARCH: Runtime bake-off — monoio vs glommio vs tokio+epoll on GET/SET workload. Build a minimal echo/KV harness on each runtime and measure throughput-per-core and tail latency at 1/4/16 cores with realistic connection counts, to validate (or refute) the ~2x/3x thread-per-core claims for our workload before committing.
- **[design, M2]** DESIGN: Runtime abstraction layer to keep monoio/glommio/tokio swappable. Specify a thin runtime/IO trait so the cache core is decoupled from the concrete runtime, enabling the bake-off and preserving a tokio+epoll fallback for portability and pre-5.6 kernels.
- **[design, M2]** DESIGN: io_uring fast path with registered buffers, multishot accept/recv, kernel feature detection. Design the Linux I/O layer around the io-uring crate (0.7.12): owned/fixed buffers, multishot accept (5.19) and multishot recv with ring-provided buffers (6.0), with runtime feature detection and graceful fallback to one-shot ops.
- **[design, M2]** DESIGN: Cross-shard coordinator for multi-key and transactional commands. Specify a Dragonfly-style per-connection coordinator fiber that fans multi-key commands (MGET/MSET/transactions/pub-sub) out to shard-owning threads via bounded channels and reassembles results with defined atomicity guarantees.
- **[decision, M1]** DECISION: Per-shard single-thread HashMap vs shared concurrent map fallback. Decide that the primary per-shard store is an unsynchronized hashbrown HashMap, and document papaya 0.2.4 (read-heavy, seize/hyaline reclamation) as the designated shared-map fallback if shard affinity is abandoned; reject RwLock<HashMap> and scc as primary.
- **[research, M0]** RESEARCH: Memory reclamation strategy for any lock-free paths (seize/hyaline vs crossbeam-epoch). Evaluate whether any IronCache component needs lock-free structures, and if so compare hyaline (seize 0.5) vs epoch-based reclamation for predictable, memory-bounded GC suitable for a cache.
- **[task, M1]** TASK: Kernel-version support matrix and single-binary fallback strategy. Document the io_uring feature/kernel matrix (5.6 basic, 5.11 tokio-uring, 5.19 multishot accept, 6.0 multishot recv) and define how a single static binary selects io_uring vs epoll/kqueue at runtime.
- **[non-goal, M1]** NON-GOAL: General-purpose work-stealing task scheduling on the hot path. State explicitly that IronCache does not use work-stealing for hot-path command processing; work-stealing is out of scope except possibly for background/maintenance tasks.
- **[research, M0]** RESEARCH: Hot-shard / load-imbalance mitigation under shard-per-core. Investigate how to detect and mitigate skewed shard load (hot keys) without re-introducing cross-core sharing, drawing on ScyllaDB's hot-shard handling.
