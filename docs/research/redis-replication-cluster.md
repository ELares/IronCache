## Summary

Redis replication is leader-follower and asynchronous. The master streams a byte-ordered command log to replicas, tagged by a (replication ID, offset) pair. On reconnect, a replica sends PSYNC <replid> <offset>; if the requested offset still lives in the master's in-memory replication backlog (a fixed-size circular buffer, compiled default 1mb, repl-backlog-ttl 3600s), the master serves a partial resync, otherwise a full resync (background RDB snapshot + buffered command stream, optionally diskless: repl-diskless-sync yes since 7.0, repl-diskless-sync-delay 5s). PSYNC2 (Redis 4.0) added a secondary replication ID (master_replid2): a promoted replica retains the old master's replid as its secondary, so the old master's other replicas and a restarted node can partial-resync across a failover instead of doing an expensive full sync. Because replication is async, acknowledged writes can be lost on failover. WAIT numreplicas timeout (since 3.0) and WAITAOF numlocal numreplicas timeout (since 7.2) let a client block until N replicas ack receipt (WAIT) or until the write is fsynced to the local/replica AOF (WAITAOF), but Redis explicitly states neither makes it a strongly consistent store — both are best-effort and writes can still be lost during failover or restart. This is the central design tension IronCache must confront: Redis chose availability/latency (async + best-effort durability) over consensus.

Redis Cluster is a shared-nothing, client-routed sharding scheme. The keyspace is statically partitioned into exactly 16384 hash slots; a key maps via HASH_SLOT = CRC16(key) & 0x3FFF using the XMODEM CRC16 (poly 0x1021, init 0x0000). Hash tags ({...}) let related keys co-locate by hashing only the substring between the first { and the first non-empty } so multi-key ops and transactions stay on one slot. Nodes form a full-mesh cluster bus on port = client_port + 10000 (default 16379) speaking a binary gossip protocol for membership, failure detection (PFAIL→FAIL escalation requires a majority of masters to agree within cluster-node-timeout, default 15000ms), and config propagation via monotonic currentEpoch/configEpoch. Clients are routed via -MOVED (permanent: update slot map) and -ASK (transient during migration: send ASKING once). Failover is a Raft-like leader election: a replica with the freshest offset requests FAILOVER_AUTH_REQUEST votes and needs a majority of masters. Defaults that matter: cluster-require-full-coverage yes (any unassigned/down slot makes the whole cluster reject writes), cluster-migration-barrier 1 (auto replica rebalancing to orphaned masters), cluster-replica-validity-factor 10, cluster-allow-reads-when-down no. Sentinel is the non-cluster HA path: 3+ Sentinels monitor a master, a quorum agrees on ODOWN (objectively down) but a majority of all Sentinels must elect a leader to actually fail over; it is eventually consistent with "last failover wins" merge semantics and an inherent acknowledged-write-loss window.

Redis Enterprise's active-active CRDB is the strongest borrowable idea for multi-region. Each region holds a full member CRDB with bidirectional async replication; commands are silently swapped for CRDT implementations so concurrent writes converge without app-side conflict logic. Strings/bitfields use last-write-wins keyed on an OS wall-clock timestamp (the only place OS time arbitrates); numeric counters (INCR/INCRBY) become conflict-free PN-counter-style accumulators (capped at 59 bits to avoid concurrent overflow); sets use an OR-Set / observed-remove (add-wins) policy where a remove can only cancel adds it has already observed. This trades RAM/metadata overhead and per-type semantics for true multi-master availability.

For IronCache's single→multi-node story, the borrowable core is: (1) offset-based async replication with a circular backlog and partial-resync-on-reconnect — cheap, simple, proven; (2) the smart-client + redirection routing model (MOVED/ASK) so the data plane stays dumb and clients cache the slot map; (3) CRC16/16384-slot hashing and hash tags for wire compatibility (Redis clients already speak this). The mechanisms to adapt or reject: 16384 is wire-compatible but a poor partition primitive — IronCache should keep CRC16 slot math for compatibility but layer a cleaner internal sharding/rebalancing engine. The big strategic decision is durability/consistency: Redis deliberately ships async + WAIT-as-best-effort and explicitly disclaims strong consistency. IronCache, if it wants a differentiator, can offer an opt-in Raft-replicated mode per shard (synchronous quorum commit) alongside the fast async default — giving users a real "no acknowledged write loss" tier that Redis OSS structurally cannot. The gossip full-mesh bus scales to ~1000 nodes (Redis's own recommendation) and is a reasonable M2 target but should be a pluggable membership layer.

## Mechanisms: borrow, adapt, or reject

| Mechanism | System | Stance | What it does | Rationale for IronCache |
| --- | --- | --- | --- | --- |
| Offset-based async replication with circular backlog | Redis replication | **borrow** | Master assigns a monotonic byte offset to its command stream and keeps a fixed-size in-memory ring buffer (repl-backlog-size, default 1mb). Replicas track (replid, offset) and request only the missed range on reconnect. | Cheap, simple, and proven; the offset+backlog primitive is exactly what IronCache needs for its default fast async tier. IronCache should make the backlog size adaptive and possibly disk-spillable so partial resync survives longer outages than a 1mb in-RAM ring allows. |
| PSYNC2 secondary replication ID | Redis replication | **borrow** | A promoted replica keeps the old master's replication ID as a secondary; other replicas and a gracefully-restarted node can partial-resync against it instead of a full RDB transfer. | Avoids thundering-herd full syncs after every failover, which is a major operational pain in large fleets. IronCache should generalize this to a short chain of historical (epoch, offset) checkpoints rather than just one secondary ID. |
| WAIT / WAITAOF best-effort acknowledgement | Redis replication | **adapt** | Client blocks until N replicas ack receipt (WAIT) or fsync to AOF (WAITAOF), returning the count actually achieved; Redis explicitly disclaims strong consistency. | The client-driven, per-write durability knob is a good API, but 'best-effort that can still lose acknowledged writes' is a weak guarantee. IronCache should keep a WAIT-like command but back it with a real quorum-commit mode so a successful WAIT means the write survives any single-node failover. |
| 16384 fixed hash slots + CRC16/XMODEM + hash tags | Redis Cluster | **adapt** | Static partitioning of the keyspace into 16384 slots via CRC16(key)&0x3FFF, with {tag} braces to co-locate related keys on one slot. | Must keep CRC16 slot math and hash-tag parsing for Redis client wire compatibility (existing smart clients hardcode it). But 16384 fixed slots is a mediocre internal partition primitive; IronCache should treat slots as a compatibility surface and run a finer-grained, dynamically rebalanced internal shard map underneath. |
| MOVED / ASK redirection with smart clients | Redis Cluster | **borrow** | Server returns -MOVED (permanent, update slot map) or -ASK (transient during migration, send ASKING once) to route clients; clients cache CLUSTER SLOTS. | Keeps the data plane dumb and pushes routing intelligence to clients that already implement it. This is the lowest-friction path to multi-node for a Redis-compatible cache and avoids a proxy hop on the hot path. |
| Full-mesh gossip cluster bus (port+10000) | Redis Cluster | **adapt** | Every node opens N-1 TCP connections on the bus port (client+10000) and gossips membership, PFAIL/FAIL failure detection, and config epochs. | Full mesh is fine to ~1000 nodes (Redis's own ceiling) but O(N^2) connections and gossip overhead cap scale. IronCache should keep gossip-based membership but make it a pluggable layer (e.g. SWIM-style) so very large or cloud-managed deployments can swap it. |
| Epoch-based Raft-like failover election | Redis Cluster | **adapt** | On master FAIL, the replica with the freshest offset increments currentEpoch and requests FAILOVER_AUTH votes; needs a majority of masters; configEpoch resolves conflicting slot ownership. | The epoch/majority idea is sound but it is an ad-hoc consensus bolted onto an async log, which is why acknowledged writes can still be lost. IronCache should offer a real per-shard Raft group as an opt-in strongly-consistent tier, reusing the freshest-offset election heuristic for leader preference. |
| cluster-require-full-coverage = yes | Redis Cluster | **adapt** | If any of the 16384 slots is unassigned or its master is down with no failover, the entire cluster rejects writes (and reads unless allow-reads-when-down). | Defaulting to total unavailability on partial failure is surprising and operationally harsh. IronCache should default to serving the slots it can and returning a typed error only for affected keys (graceful partial availability), with strict mode opt-in. |
| Sentinel quorum-detect + majority-elect HA | Redis Sentinel | **reject** | 3+ external monitor processes; a quorum agrees a master is ODOWN, but a majority of all Sentinels must elect a leader to fail over; eventually consistent, 'last failover wins'. | A separate Sentinel fleet is operational overhead that fights IronCache's single-static-binary goal, and its eventual-consistency/last-failover-wins model still loses writes. IronCache should fold HA into the cluster membership/consensus layer of the same binary rather than ship a sidecar process. |
| Active-Active CRDB CRDTs (LWW strings, PN-counters, OR-Sets) | Redis Enterprise | **adapt** | Each region is a full member DB with bidirectional async replication; commands are swapped for CRDT versions so concurrent multi-master writes converge automatically (LWW on wall-clock for strings, accumulating counters capped at 59 bits, add-wins observed-remove for sets). | This is the best blueprint for multi-region multi-master availability and worth borrowing conceptually, but per-type CRDTs add metadata/RAM overhead that conflicts with IronCache's minimal-memory goal, and wall-clock LWW is clock-skew-sensitive. IronCache should make CRDT multi-master an opt-in per-keyspace mode, prefer Hybrid Logical Clocks over raw OS time, and keep the default mode async non-CRDT for memory efficiency. |

## Implications for IronCache

- Ship two durability tiers in one binary: a default fast async-replication mode (offset + ring backlog + partial resync, borrowed from Redis) and an opt-in per-shard Raft quorum-commit mode that makes a successful WAIT-equivalent mean 'survives any single-node failover'. This is a concrete differentiator over Redis OSS, which structurally can only offer best-effort.
- Keep CRC16/XMODEM slot math and {hash-tag} parsing exactly (crc16(key)&0x3FFF over 16384 slots) as a wire-compatibility surface so existing Redis smart clients route correctly, but decouple it from the internal physical shard map so IronCache can rebalance at finer granularity and migrate slots without the cluster-wide write freeze.
- Adopt MOVED/ASK + smart-client routing as the multi-node data path (no proxy hop), and publish a CLUSTER SLOTS / CLUSTER SHARDS compatible response so off-the-shelf clients work unmodified.
- Make the replication backlog adaptive and optionally disk-spillable rather than a fixed 1mb RAM ring, so partial resync survives longer outages and large fleets avoid full-resync storms; generalize PSYNC2's single secondary replid into a short ring of (epoch, offset) checkpoints.
- Default to graceful partial availability (serve reachable slots, typed error per affected key) instead of Redis's cluster-require-full-coverage=yes total shutdown; expose strict mode as opt-in.
- Fold HA into the same static binary's membership/consensus layer — do NOT require a separate Sentinel-style sidecar fleet; this preserves the easy single-binary setup promise.
- Offer CRDT-based active-active as an opt-in per-keyspace mode (PN-counters, OR-Sets, register LWW) for multi-region, but use Hybrid Logical Clocks instead of raw OS wall-clock for LWW, and keep it off by default because per-type CRDT metadata fights the minimal-memory goal.
- Use a Raft-like freshest-offset leader-preference heuristic for failover election, but back it with a real consensus group (per-shard) so configEpoch-style split-brain resolution is principled rather than ad hoc.
- Treat the gossip/membership protocol as a pluggable module (Redis full-mesh is fine to ~1000 nodes; SWIM or a hub topology may be needed beyond that or in cloud-managed envs).

## Research papers and primary sources

- **In Search of an Understandable Consensus Algorithm (Raft)** (USENIX ATC 2014). A consensus algorithm equivalent to multi-Paxos but decomposed into leader election, log replication, and safety for understandability and practical implementation. [source](https://www.usenix.org/system/files/conference/atc14/atc14-paper-ongaro.pdf) Relevance: The blueprint for IronCache's opt-in strongly-consistent per-shard replication tier — the thing Redis async replication + WAIT cannot provide. Redis Cluster's epoch-vote failover is a partial, ad-hoc approximation of this.
- **Dynamo: Amazon's Highly Available Key-value Store** (SOSP 2007). Eventually-consistent zero-hop DHT using consistent hashing, vector clocks for causality, sloppy quorums/hinted handoff, and gossip-based membership and failure detection. [source](https://www.allthingsdistributed.com/files/amazon-dynamo-sosp2007.pdf) Relevance: Reference design for the always-available, gossip-membership, eventually-consistent end of IronCache's spectrum; consistent hashing is the alternative to Redis's fixed 16384 slots for internal rebalancing, and vector clocks/HLC inform CRDT causal context.
- **Conflict-free Replicated Data Types (CRDTs)** (SSS 2011 (Shapiro, Preguica, Baquero, Zawirski)). Formal foundation for state- and operation-based replicated data types (counters, OR-Sets, LWW registers) that converge without coordination. [source](https://inria.hal.science/inria-00609399v1/document) Relevance: The theory behind Redis Enterprise active-active; directly informs IronCache's opt-in multi-master mode (PN-counters, observed-remove sets) and warns about the metadata cost that conflicts with the minimal-memory goal.
- **Don't Settle for Eventual: Scalable Causal Consistency for Wide-Area Storage with COPS** (SOSP 2011). Causal+ consistency at scale using explicit dependency tracking, offering a middle ground stronger than eventual but more available than linearizable. [source](https://www.cs.cmu.edu/~dga/papers/cops-sosp2011.pdf) Relevance: A candidate consistency model for IronCache's multi-region tier that is stronger than Redis Enterprise's wall-clock LWW yet far cheaper than full consensus; motivates Hybrid Logical Clocks over raw OS time.

## Open questions

- Can IronCache offer a per-shard Raft (quorum-commit) durable tier without giving up the per-core throughput crown? What is the measured write-latency/throughput delta vs async, and can batching/pipelining the Raft log hide it?
- Is wire-level Redis Cluster compatibility (16384 slots, CLUSTER SLOTS/SHARDS, MOVED/ASK) worth the constraint it imposes on the internal shard map, or should IronCache expose Redis routing only as a compatibility shim over a native consistent-hashing ring?
- How should slot/shard migration avoid Redis's cluster-wide write-freeze and -TRYAGAIN churn — incremental key streaming with per-key fencing, or snapshot+delta with a brief cutover?
- For the CRDT multi-master mode, what is the real per-key metadata overhead (causal context / version vectors), and can it be bounded enough to keep IronCache memory-competitive, or should multi-master be a fundamentally separate engine?
- Should HA membership use Redis-style full-mesh gossip, SWIM, or a managed control plane — and what is the node-count crossover where full mesh's O(N^2) connections become the bottleneck?
- Should LWW use Hybrid Logical Clocks to bound clock-skew anomalies, and what HLC epsilon is acceptable for a cache where occasional staleness is tolerable?
- Where does AI-driven innovation fit: e.g. learned/adaptive backlog sizing, predictive replica placement, or auto-tuning the consistency tier per keyspace based on observed access patterns?

## Proposed issues (seeds for the tracker)

- **[decision, M0]** DECISION: Default durability/consistency model (async best-effort vs opt-in quorum). Decide IronCache's default replication semantics and whether to commit to an opt-in synchronous quorum-commit tier. Redis OSS ships async + best-effort WAIT and explicitly disclaims strong consistency; decide if 'no acknowledged write loss on single-node failover' is a headline differentiator.
- **[research, M0]** RESEARCH: Per-shard Raft for an opt-in strongly-consistent tier. Evaluate Raft (ATC'14) as the consensus layer for an optional durable tier, including expected throughput/latency cost vs async and how to batch the log to preserve per-core throughput.
- **[design, M2]** DESIGN: Offset-based async replication with adaptive, disk-spillable backlog. Specify a (replid, byte-offset) replication stream and a partial-resync-on-reconnect backlog that is adaptive in size and can spill to disk, generalizing Redis's fixed 1mb RAM ring and PSYNC2 secondary-replid to a checkpoint ring.
- **[design, M1]** DESIGN: Redis-Cluster-compatible routing (16384 CRC16 slots, hash tags, MOVED/ASK). Define the wire-compatibility surface: CRC16/XMODEM key->slot mapping, {hash-tag} parsing, CLUSTER SLOTS/SHARDS responses, and MOVED/ASK redirection so existing Redis smart clients route unmodified.
- **[decision, M1]** DECISION: Internal shard map decoupled from the 16384 compatibility slots. Decide whether to run a finer-grained, dynamically-rebalanced internal shard map underneath the fixed 16384-slot compatibility layer, vs binding internal partitioning to Redis slots directly.
- **[design, M2]** DESIGN: Graceful partial availability instead of cluster-require-full-coverage shutdown. Specify serving reachable slots and returning typed per-key errors for unavailable slots by default, with Redis's all-or-nothing strict mode as opt-in.
- **[design, M2]** DESIGN: Online slot/shard migration without cluster-wide write freeze. Design incremental, fenced key migration that avoids Redis's migration-time write stalls and -TRYAGAIN churn while preserving correctness for multi-key/hash-tag operations.
- **[decision, M1]** DECISION: Fold HA into the single binary (reject external Sentinel sidecar). Decide that failure detection and failover live in IronCache's own membership/consensus layer rather than a separate Sentinel-style process, preserving the single-static-binary setup goal.
- **[research, M0]** RESEARCH: Opt-in CRDT active-active multi-master (counters, OR-Sets, HLC LWW). Study Redis Enterprise CRDB and CRDT theory to scope an optional per-keyspace multi-master mode; quantify metadata/RAM overhead against the minimal-memory goal and evaluate Hybrid Logical Clocks vs OS wall-clock for LWW.
- **[non-goal, M0]** NON-GOAL: Strong consistency / zero write loss in the default async mode. Explicitly state that the default fast tier, like Redis, is async and best-effort; strong durability is only the opt-in quorum tier. Prevents over-promising in the README/vision.
- **[research, M1]** RESEARCH: Membership protocol choice and node-count scaling (full-mesh gossip vs SWIM). Compare Redis full-mesh gossip (practical ceiling ~1000 nodes, O(N^2) connections) against SWIM-style or hub topologies and define a pluggable membership interface.
- **[epic, M0]** EPIC: Single-node to multi-node IronCache (replication, sharding, HA, multi-region). Umbrella epic tracking the path from a single static binary to a clustered, optionally strongly-consistent, optionally multi-master cache, sequencing the replication, routing, HA, and CRDT decisions above.
