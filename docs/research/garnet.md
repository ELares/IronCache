## Summary

Microsoft Garnet is the closest existing system to IronCache's stated goal: a RESP-wire-compatible remote cache-store engineered for high throughput-per-core, low and stable tail latency, larger-than-memory tiering, and single-node-to-cluster scale-out. Its core insight — and the thing IronCache should study hardest — is a clean layering: a "narrow-waist" storage API (Read / Upsert / Delete / atomic RMW with async callbacks) over the Tsavorite engine (a fork of FASTER), with the full rich RESP surface implemented on top of just those four primitives. This is exactly the boundary IronCache wants between its wire layer and its store. Garnet is C#/.NET, so the runtime (GC, SocketAsyncEventArgs, SslStream, thread pool) is not portable, but the algorithms and data structures are.

The storage engine is the deepest source of ideas. Tsavorite/FASTER combines a lock-free hash index (cache-aligned 64-byte buckets, 8 entries each, with a "tentative" bit for lock-free insertion) with a HybridLog: a single logical address space split into three regions by two moving markers. The mutable region (tail down to ReadOnlyAddress) supports in-place updates of the hot set with no allocation or log growth — this is what lets FASTER beat pure in-memory structures and is the single most important mechanism for IronCache's "max throughput per core." The read-only region (ReadOnlyAddress down to HeadAddress) is immutable and updated via read-copy-update (append a new version at the tail). The stable region (below HeadAddress) lives on disk, giving larger-than-memory storage. Concurrency is governed by an epoch-protection framework (global epoch counter, thread-local epochs, a drain list of trigger actions fired when all threads pass an epoch) rather than fine-grained latches — region-boundary shifts, page flushes, and memory reclamation are all deferred until safe. Locking, where needed, is at hash-bucket granularity (15 shared-lock bits + 1 exclusive bit) rather than per-record. FASTER reports up to 160M ops/sec on a single multi-core machine (YCSB).

Garnet's networking/threading model is its second major contribution and is highly portable as a design. Rather than Redis's single-threaded event loop, Garnet runs a shared-memory, thread-scalable server: connections are handled with async socket completions, and crucially TLS decryption, RESP parsing, and the storage operation all execute inline on the IO completion thread, avoiding a thread hand-off per request. I confirmed this in source: the non-TLS receive path calls OnNetworkReceiveWithoutTLS synchronously inside the SocketAsyncEventArgs.Completed callback in a do/while(!ReceiveAsync) loop, so when data is already buffered the same thread keeps parsing and serving with zero context switches. This "process on the thread that received the bytes" design plus batching is what gives Garnet better scaling across cores and across payload sizes than Redis/KeyDB, and far more stable p99/p99.9 than Dragonfly in MSR's published comparisons. IronCache, being natively threaded Rust with no GC, can implement the same shared-state-threaded model (e.g. io_uring or epoll with per-core acceptor + inline processing) without .NET's overheads.

Durability and ops features round out the picture: append-only-file (AOF / write-ahead log, off by default), non-blocking checkpoints in two flavors — Fold-Over (move ReadOnly marker to tail, persist incrementally within the hybrid log) and Snapshot (separate full in-memory snapshot file) — built on Concurrent Prefix Recovery (CPR, SIGMOD'19), which commits "all operations until time t per session" via async incremental checkpointing instead of a classic WAL. Cluster mode adds Redis-Cluster-compatible hash-slot sharding, gossip, replication, failover, and dynamic key migration (including a FastMigrate path that writes straight to the network buffer). Log compaction (Lookup strategy recommended; Scan discouraged due to transient memory blow-up) and "revivification" (free-record reuse) manage space.

For IronCache the takeaways are concrete. Borrow: the narrow-waist store API, the HybridLog mutable/read-only/stable tri-region with in-place hot-set updates, epoch-based deferred reclamation, and inline-on-IO-thread request processing. Adapt: tiered storage and checkpointing (Rust has no managed GC, so memory management and the object store for composite types must be redesigned around explicit ownership / arenas; F2's hot-log+cold-log split is a better blueprint than FASTER's single hybrid log for very skewed larger-than-memory workloads). Reject for v1: the .NET object store representation, AOF-per-op durability defaults, and full cluster mode — these are M2+ concerns. The defaults to note for parity decisions: port 6379, 16 GiB log, 90% mutable, storage tiering and read cache OFF by default, compaction None by default, AOF off by default.

## Mechanisms: borrow, adapt, or reject

| Mechanism | System | Stance | What it does | Rationale for IronCache |
| --- | --- | --- | --- | --- |
| Narrow-waist storage API under a rich RESP surface | Garnet | **borrow** | Implements the entire RESP command surface (strings, objects, bitmaps, HLL, geo, pub/sub) on top of just four storage primitives — Read, Upsert, Delete, and atomic Read-Modify-Write (RMW) — each taking async callbacks where the command layer injects logic (merge functions, expiry, conditional writes). | This is the single cleanest architectural idea for IronCache: it decouples the wire/command layer from the engine and makes RMW (INCR, APPEND, expiry, LPUSH) expressible without bespoke engine code. In Rust this maps naturally to a trait with associated callback closures / generic Functions parameter. Adopt the boundary verbatim; it is the key to 'easy to extend' and to keeping the hot path uniform. |
| HybridLog tri-region address space (mutable / read-only / stable) | FASTER/Tsavorite | **borrow** | A single logical 64-bit address space split by two moving markers: mutable region [ReadOnlyAddress, TailAddress) supports in-place updates of the hot set (no allocation); read-only region [HeadAddress, ReadOnlyAddress) is immutable and updated via read-copy-update (append new version at tail); stable region (< HeadAddress) spills to disk for larger-than-memory. | In-place update of the hot set is precisely how FASTER beats pure in-memory structures and is the mechanism most aligned with 'max throughput per core, minimal memory.' The tri-region marker scheme also gives larger-than-memory tiering for free. Borrow the structure; IronCache must re-implement record layout with explicit Rust ownership (no GC) and consider variable-length value handling. |
| Epoch-protection framework for lock-free concurrency | FASTER/Tsavorite | **borrow** | A global epoch counter plus thread-local epoch entries and a 'drain list' of trigger actions executed only when every active thread has advanced past a given epoch. Region-boundary shifts, page flushes/eviction, index resizing, and memory reclamation are deferred until epoch-safe, avoiding fine-grained latches on the hot path. | This is the concurrency backbone enabling lock-free reads/in-place updates and safe deferred reclamation — directly serving throughput-per-core. Rust analog: epoch-based reclamation (crossbeam-epoch) or a hand-rolled equivalent. Borrow the concept; crossbeam-epoch gives a battle-tested Rust foundation, but the trigger-action drain list is worth replicating for boundary shifts. |
| Lock-free hash index with tentative-bit insertion | FASTER/Tsavorite | **adapt** | Cache-line-aligned 64-byte hash buckets holding 8 entries each; insertion uses a two-phase CAS with a per-entry 'tentative' bit to avoid duplicate keys under concurrent insert without locking. Hash-bucket-level locking (15 shared + 1 exclusive bit) is used only when locking is required. | The cache-aligned bucket + tentative-bit pattern is excellent for throughput, but IronCache may prefer a more modern open-addressing or swiss-table-style index in Rust. Adapt: keep cache-line alignment and bucket-granularity locking, but evaluate against alternatives; the 64-byte/8-entry geometry is a strong default. |
| Inline request processing on the IO completion thread | Garnet | **borrow** | On socket receive, RESP parsing and the storage operation execute synchronously on the same thread that received the bytes (verified in TcpNetworkHandlerBase: OnNetworkReceiveWithoutTLS is called inline inside the SocketAsyncEventArgs.Completed callback, looping while data is buffered). No per-request hand-off to a worker pool; TLS decryption also runs there. | Eliminating the thread hand-off per request is a primary reason Garnet scales across cores and keeps p99 stable versus a single-threaded event loop. IronCache (Rust, no GC) can do the same with io_uring or epoll: per-core acceptor, process-on-receiving-thread. Borrow the principle; replace SocketAsyncEventArgs with io_uring/tokio's multi-thread runtime or a thread-per-core design. |
| Two stores: string main store + object store, unified log | Garnet | **adapt** | Raw strings live in a 'main store' optimized for inline values; composite types (list/hash/set/sortedset) live in a separate 'object store'. Their durability is bound by a single unified operation log so checkpoints/AOF are consistent across both. | The separation is sound (different memory layouts for scalars vs containers), and the unified log for cross-store consistency is worth keeping. But Garnet's object store holds managed .NET objects on the GC heap — not portable. Adapt: IronCache needs an explicit, arena/slab-backed representation for container types with manual memory management; keep the unified-log durability binding. |
| Dual checkpoint modes (Fold-Over + Snapshot) on CPR | FASTER/Tsavorite | **adapt** | Fold-Over moves the ReadOnly marker to the tail and persists state incrementally within the existing hybrid log (cheap, incremental). Snapshot writes a separate full in-memory snapshot file. Both are non-blocking, built on Concurrent Prefix Recovery, which defines a commit as 'all operations until time t, per client session.' | Non-blocking incremental checkpointing is valuable for a cache that also wants persistence, and Fold-Over reuses the log structure elegantly. Adapt: IronCache should offer at least Fold-Over-style incremental checkpoints, but CPR's per-session prefix semantics may be more than a cache needs; consider a simpler RDB-like snapshot for v1 and Fold-Over later. |
| F2 hot-log + cold-log split for skewed larger-than-memory | F2 (FASTER successor) | **adapt** | Splits storage into an in-memory hot-log component and an on-disk cold-log component with promotion/read-caching between them, specifically optimizing very skewed workloads larger than memory and reducing memory overhead vs single-hybrid-log FASTER. | For a cache, skewed access is the common case, so F2's two-component design is arguably a better blueprint than FASTER's single hybrid log when going larger-than-memory. Adapt as the model for IronCache's tiering design rather than copying FASTER's older single-log eviction; flagged as M2 research because it adds complexity. |
| AOF (write-ahead log) with tunable commit cadence | Garnet | **adapt** | Optional append-only file for durability, off by default; commit cadence configurable (0 = immediate per-op, -1 = manual COMMITAOF, >0 = periodic ms). Supports sharded sublogs (AofPhysicalSublogCount) for throughput and unsafe truncation at a size limit. | Sensible to offer for users who want persistence, and the sharded-sublog idea aids write scaling. But per-op fsync defaults destroy latency; IronCache should default AOF off (like Garnet) and, if offered, default to periodic group-commit. Adapt the model, choose safer/faster defaults. |
| Revivification (free-record reuse) and Lookup-based compaction | Garnet/Tsavorite | **borrow** | Revivification reclaims and reuses freed records in the mutable region to bound memory; log compaction reclaims dead records, with a Lookup strategy (hash-chain liveness check, no data loss) recommended and a Scan strategy explicitly discouraged because it builds a transient parallel index proportional to the keyspace. | A log-structured store needs space reclamation; revivification directly serves 'minimal memory,' and the documented Lookup-vs-Scan tradeoff is a useful design lesson (avoid algorithms with transient memory proportional to keyspace). Borrow the concepts and heed the warning about Scan-style compaction. |
| Cluster mode: hash-slot sharding, gossip, FastMigrate | Garnet | **adapt** | Redis-Cluster-compatible 16384 hash-slot sharding with gossip-based membership, replication, failover, and dynamic key migration; FastMigrate writes migrated keys directly to the network buffer and does not wait for per-batch acks. | IronCache wants single-node-to-multi-node, and Redis-Cluster slot compatibility maximizes client interop. Adapt for a later milestone (M2+): keep slot compatibility and the FastMigrate zero-copy idea, but cluster is explicitly out of scope for the first prototype. |
| C#/.NET runtime, GC, SslStream, managed thread pool | Garnet | **reject** | Garnet runs on .NET 8, relying on the CLR garbage collector, managed thread pool, SocketAsyncEventArgs, and SslStream for TLS; tuning exposes ThreadPoolMin/Max worker and IO-completion thread knobs. | The entire managed runtime is antithetical to IronCache's 'most efficient cache, single static binary' goal — GC pauses hurt tail latency, the CLR is not a single static binary, and managed object stores fight zero-copy. Reject the runtime wholesale; extract only the algorithms. Rust + a thread-per-core async stack achieves the same architecture without GC. |

## Implications for IronCache

- Adopt Garnet's narrow-waist store API as IronCache's internal boundary: define a Rust trait/engine interface of exactly Read, Upsert, Delete, RMW, parameterized by a Functions type carrying callbacks (merge/RMW logic, single-writer in-place update, read-copy-update, expiry). The whole RESP command set is built on these four — this keeps the hot path uniform and the engine testable in isolation.
- Build the engine around a HybridLog with in-place updates in a mutable hot region. This is the mechanism that lets FASTER beat in-memory structures; it is the most direct route to IronCache's 'max throughput per core, minimal memory.' Default mutable fraction ~90% (Garnet's default) and make it tunable.
- Use epoch-based reclamation (crossbeam-epoch or a hand-rolled global-epoch + thread-local + drain-list) instead of per-record locks. Defer all region-boundary shifts, page evictions, and frees to epoch-safe points. This is the lock-free concurrency backbone that a GC-less Rust implementation must provide explicitly.
- Process requests inline on the receiving thread. Choose a thread-per-core (or io_uring) network stack where RESP parsing and the store operation run on the thread that read the bytes, with no hand-off to a worker pool — this is the verified source-level reason Garnet scales across cores and keeps p99 stable. This is a hard architectural decision to lock in early.
- Default durability/persistence OFF, matching Garnet (EnableStorageTier=false, EnableAOF=false, CompactionType=None). Ship as a fast in-memory cache by default; tiering, AOF, checkpointing are opt-in. If AOF is offered, default to periodic group-commit, never per-op fsync.
- Separate scalar and container representations (main store vs object store) but redesign the container store with explicit Rust memory management (slabs/arenas), since Garnet's GC-heap object store is not portable. Bind both to one unified operation log for consistent checkpoints.
- For larger-than-memory, prefer F2's hot-log/cold-log two-component design over FASTER's single hybrid log — caches are skewed, and F2 is purpose-built for skew with lower memory overhead. Treat as M2 research.
- Heed Garnet's compaction warning: avoid space-reclamation algorithms whose transient memory is proportional to the keyspace (Garnet's Scan); prefer hash-chain-liveness Lookup-style compaction plus revivification for bounded memory.
- Use a cache-line-aligned bucketed hash index (64-byte / 8-entry with tentative-bit lock-free insert) as a strong baseline, but benchmark against a Rust swiss-table/open-addressing design before committing.
- Target RESP2/RESP3 and Redis-Cluster 16384-slot compatibility for client interop, but scope single-node first; cluster/replication/failover and FastMigrate-style zero-copy migration are M2+.
- Run a head-to-head benchmark harness against Garnet itself (plus Redis 7.2+, Valkey, KeyDB, Dragonfly) on the same Garnet methodology (8B keys/values, batch 4-4096, 1K and 256M key DBs, 1-128 sessions, measure p50/p99/p99.9) — note MSR published only graphs, so IronCache should publish absolute numbers to differentiate.
- Plan for a single static binary from day one (musl/static-linked Rust) — this is a concrete differentiator over Garnet, which requires the .NET runtime and is not a single static binary.

## Research papers and primary sources

- **FASTER: A Concurrent Key-Value Store with In-Place Updates** (SIGMOD 2018). Combines a lock-free cache-aligned hash index with a HybridLog (mutable in-memory + read-only in-memory + on-disk regions) enabling in-place updates of the hot set; up to 160M ops/sec single-machine. [source](https://www.microsoft.com/en-us/research/uploads/prod/2018/03/faster-sigmod18.pdf) Relevance: Foundational design of Tsavorite/Garnet's engine; the in-place-update + epoch-protection model is the core IronCache should borrow for throughput-per-core.
- **FASTER: An Embedded Concurrent Key-Value Store for State Management** (PVLDB Vol 11 (2018)). Journal version expanding on FASTER's embedded use, recovery, and state-management workloads. [source](http://www.vldb.org/pvldb/vol11/p1930-chandramouli.pdf) Relevance: Deeper treatment of recovery and embedding; informs IronCache's persistence and embeddability design.
- **Concurrent Prefix Recovery: Performing CPR on a Database** (SIGMOD 2019). Non-blocking checkpointing that commits 'all operations until time t per session' via asynchronous incremental checkpoints instead of a classic WAL. [source](https://www.microsoft.com/en-us/research/wp-content/uploads/2019/01/cpr-sigmod19.pdf) Relevance: Underpins Garnet's Fold-Over/Snapshot checkpoints; a reference for IronCache's optional non-blocking persistence.
- **Asynchronous Prefix Recoverability for Fast Distributed Stores (D-FASTER)** (SIGMOD 2021). Extends CPR to distributed commit as a lightweight metadata-only operation across nodes. [source](https://dl.acm.org/doi/pdf/10.1145/3448016.3458454) Relevance: Relevant if IronCache pursues distributed/replicated durability semantics in cluster mode.
- **From FASTER to F2: Evolving Concurrent Key-Value Store Designs for Large Skewed Workloads** (arXiv 2023 (2305.01516)). Two-component hot-log (in-memory) + cold-log (on-disk) design with promotion/read-caching, optimized for very skewed larger-than-memory workloads and lower memory overhead than FASTER. [source](https://arxiv.org/pdf/2305.01516) Relevance: Better blueprint than the single hybrid log for a cache's skewed, larger-than-memory tiering — directly informs IronCache's tiering design.

## Open questions

- Garnet/MSR publish only log-scale benchmark graphs, not absolute ops/sec or microsecond p99 figures — what are the real numbers on the F72s v2 setup, and how would a Rust thread-per-core design compare head-to-head?
- How much of Garnet's throughput advantage comes from the HybridLog engine vs the inline-on-IO-thread network model? Decomposing this determines where IronCache should invest first.
- Does the in-place-update / hybrid-log model interact well with smart compression (a stated IronCache goal)? In-place updates assume stable record size; compressed/variable-length values complicate in-place mutation — needs design study.
- Garnet's object store relies on the .NET GC heap; what is the right GC-less Rust representation for list/hash/set/sortedset that preserves O(1)-ish ops and zero-copy serialization without fragmentation?
- Is crossbeam-epoch sufficient, or does IronCache need FASTER's richer drain-list/trigger-action epoch framework for region-boundary shifts and index resizing?
- What is the minimal durability story for a cache: is CPR's per-session prefix recovery overkill, and is a simpler periodic snapshot + optional AOF enough for v1?
- How does F2's hot/cold split compare to a classic admission/eviction policy (e.g., TinyLFU/W-TinyLFU) for a cache workload — should IronCache combine log-structured storage with a modern admission policy rather than rely on hybrid-log markers alone?
- Garnet supports server-side stored procedures and custom commands in C#; what is IronCache's portable extensibility story (WASM modules? Rust dylib plugins? Lua)?

## Proposed issues (seeds for the tracker)

- **[design, M1]** Adopt a narrow-waist storage API (Read/Upsert/Delete/RMW) under the RESP layer. Specify a four-primitive engine interface with callback hooks, mirroring Garnet's Tsavorite narrow waist, so the full RESP surface (including RMW commands like INCR/APPEND/expiry) is built on a uniform hot path. Decouples wire/command layer from the store.
- **[design, M2]** Engine design: HybridLog with in-place hot-set updates. Design IronCache's log-structured store as a tri-region hybrid log (mutable/read-only/stable) with in-place updates in the mutable region, ported from FASTER but with explicit Rust record layout and ownership (no GC).
- **[decision, M1]** Decision: epoch-based reclamation (crossbeam-epoch) vs custom drain-list framework. Decide the lock-free concurrency backbone: evaluate crossbeam-epoch against a FASTER-style global-epoch + thread-local + trigger-action drain list, weighing maturity vs the need to coordinate region-boundary shifts and index resizing.
- **[decision, M1]** Decision: thread-per-core, process-on-receiving-thread network model. Lock in the threading model: replicate Garnet's verified inline-on-IO-thread processing (no per-request hand-off) using io_uring or a thread-per-core async stack in Rust, contrasted with Redis's single-threaded event loop.
- **[research, M0]** Research: benchmark IronCache architecture vs Garnet/Redis/Valkey/KeyDB/Dragonfly. Reproduce Garnet's methodology (2x F72s-class, 8B kv, batch 4-4096, 1K & 256M keys, 1-128 sessions, p50/p99/p99.9) and, unlike MSR, publish absolute throughput and latency numbers.
- **[research, M2]** Research: F2 hot-log/cold-log split vs single hybrid log for skewed larger-than-memory. Evaluate whether IronCache's larger-than-memory tier should follow F2's two-component (hot/cold) design rather than FASTER's single hybrid log, given caches' inherent access skew and F2's lower memory overhead.
- **[design, M2]** Design: GC-less representation for container types (list/hash/set/sortedset). Design slab/arena-backed Rust representations for Redis composite types to replace Garnet's GC-heap object store, preserving zero-copy serialization and bounded fragmentation, bound to a unified operation log with the scalar store.
- **[decision, M1]** Decision: default durability posture (tiering/AOF/checkpoint OFF by default). Match Garnet's safe defaults (storage tier off, AOF off, compaction None) so IronCache ships as a fast in-memory cache by default; if AOF is offered, default to periodic group-commit, never per-op fsync.
- **[research, M2]** Research: space reclamation via revivification + Lookup-style compaction. Adopt Garnet's revivification (free-record reuse) and hash-chain-liveness Lookup compaction; explicitly avoid Scan-style compaction whose transient memory is proportional to the keyspace.
- **[non-goal, M0]** Non-goal (v1): full cluster mode, replication, and failover. Declare cluster sharding/replication/failover and FastMigrate-style key migration out of scope for the first prototype; preserve Redis-Cluster 16384-slot wire compatibility for later, single-node first.
- **[non-goal, M0]** Non-goal: managed runtime / GC — single static Rust binary instead. Explicitly reject Garnet's .NET/GC runtime: IronCache must be a single statically linked Rust binary with no GC pauses, extracting only Garnet's algorithms, not its runtime.
- **[research, M2]** Research: interaction of in-place updates with smart value compression. Study how the hybrid-log in-place-update model (which assumes stable record size) interacts with IronCache's smart-compression goal, since compressed/variable-length values complicate in-place mutation.
