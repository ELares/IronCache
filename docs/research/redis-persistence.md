## Summary

Redis offers two persistence engines and a spectrum of durability that is weaker than most users assume. RDB is a point-in-time binary snapshot produced by fork()+copy-on-write: BGSAVE forks a child that walks the (frozen-at-fork) heap and writes a compressed RDB to a temp file, then atomically renames it. Save points are time/change thresholds; the documented default (when uncommented) is `save 3600 1 300 100 60 10000` (snapshot after 3600s if >=1 change, 300s if >=100, 60s if >=10000). The headline weakness is the COW memory-blowup: every 4KB page the parent mutates during the snapshot is duplicated, so under a fully-write-heavy workload RSS can nearly double; operators are told to keep ~50-100% headroom. fork() itself is O(heap) because the kernel copies page tables — Redis's own docs cite ~9-13 ms/GB on physical Linux, but up to 239 ms/GB on old Xen EC2, meaning fork is a multi-hundred-millisecond stall on a large heap and the single biggest source of Redis tail latency.

AOF is a write-ahead command log replayed on restart. `appendfsync` has three modes — `always` (fsync every write, slowest, ~single-write durability), `everysec` (the default; a background thread fsyncs once per second), and `no` (rely on the kernel's ~30s writeback). The honest durability story is the gap: `everysec` is widely described as "lose at most 1 second," but the source's flushAppendOnlyFile logic postpones writes for up to 2 seconds when a background fsync is still in flight on a busy disk, so the real worst case is ~2 seconds of loss. `always` fsyncs per event-loop iteration (per batch of commands), not literally per command, and still cannot survive a power loss between the write() and the fsync on some filesystems. AOF rewrite (BGREWRITEAOF, also fork-based) compacts the log; auto-triggers at `auto-aof-rewrite-percentage 100` once past `auto-aof-rewrite-min-size 64mb`.

The two most architecturally interesting modern features are (1) the hybrid RDB-preamble / multi-part AOF, and (2) diskless replication. Since 4.0, `aof-use-rdb-preamble yes` lets an AOF rewrite emit a compact RDB image as its prefix followed by an incremental command tail — fast load + ongoing durability. Redis 7.0 generalized this into multi-part AOF: instead of one file there is a directory (`appenddirname appendonlydir`) holding at most one BASE file (RDB or AOF format), one or more INCR files, and HISTORY files, all tracked by a plaintext manifest (`file <name> seq <n> type <b|i|h>`). This made AOFRW crash-safe (atomic manifest swap instead of in-place buffer copying) and decoupled the base snapshot from the live append stream.

Diskless replication (`repl-diskless-sync yes`, default since 7.0) has the fork child stream the RDB straight to replica sockets, skipping the master's disk entirely; `repl-diskless-sync-delay 5` batches multiple replicas into one fork. On the replica, `repl-diskless-load` defaults to `disabled` (spool to disk first) because the two non-default modes are dangerous: `swapdb` doubles memory by holding old+new datasets, and `on-empty-db` only diskless-loads into an empty instance. WAIT (since 3.0) and WAITAOF (since 7.2) expose tunable durability per-command, but Redis explicitly documents that WAIT "does not make Redis a strongly consistent store" — it confirms in-memory replica acks, not disk, and acked writes can still be lost on failover.

For IronCache the lessons are sharp. fork()+COW is the root cause of Redis's worst latency and memory pathologies (compounded catastrophically by Transparent Huge Pages, where a single byte write copies a whole 2MB page — hence Redis's hard recommendation to set THP to `never`). A from-scratch Rust cache should reject fork-based snapshotting outright in favor of a thread-based, bounded-memory snapshot (e.g. copy-on-write at the slab/page level under our own control, or an LSM/segment log) so snapshot memory overhead is configurable and predictable rather than "up to 2x and hope you set vm.overcommit_memory=1." We should borrow the multi-part-AOF manifest design (atomic, crash-safe, base+incremental) and the per-command durability knob (WAITAOF-style), but make the durability semantics explicit and honest in our README rather than burying the 2-second everysec caveat. Diskless replication is worth adapting (stream snapshot over the socket) but the replica-side memory-doubling trap must be designed out from day one.

Defaults verified against redis.conf at tags 7.4.0 and 8.8.0 (identical); behavior comments cross-checked against src/aof.c and the official persistence/latency/replication docs.

## Mechanisms: borrow, adapt, or reject

| Mechanism | System | Stance | What it does | Rationale for IronCache |
| --- | --- | --- | --- | --- |
| RDB snapshot via fork()+copy-on-write (BGSAVE) | Redis | **reject** | Forks a child process that inherits a COW view of the heap at fork time and serializes it to a compressed, checksummed RDB file, then atomically renames over the old one. Parent keeps serving; mutated pages are duplicated by the kernel. | fork()+COW is the single largest source of Redis tail latency (page-table copy is ~9-13 ms/GB, far worse on virtualized hosts) and the COW memory blowup (up to ~2x RSS, catastrophic with THP) is unpredictable and unconfigurable. IronCache should reject process fork entirely and implement snapshotting inside the process with our own bounded, page-granular COW over our slab allocator (or an LSM/segment-log design), giving a configurable memory ceiling and no page-table-copy stall. |
| AOF appendfsync everysec (background fsync thread) | Redis | **adapt** | Buffers writes and a background thread fsyncs once per second; if the prior fsync is still running, the main thread postpones the write up to 2 seconds, then forces it through. | The everysec model (decoupled background fsync) is a good throughput/durability tradeoff and worth keeping, but IronCache should be honest that the worst case is ~2s not 1s, expose the postpone bound as a tunable, and surface a per-write durability metric so users can see how stale the on-disk state is. |
| AOF appendfsync always | Redis | **adapt** | fsync per event-loop flush (effectively per batch of commands), giving near-per-write durability at large throughput cost. | Keep an 'always/strict' mode but implement it as group-commit (coalesce concurrent writers into one fsync) to avoid Redis's throughput collapse; also use fdatasync and consider io_uring batched fsync to amortize the syscall. |
| Multi-part AOF with manifest (base + incr + history, since 7.0) | Redis | **borrow** | Splits the AOF into a directory of one BASE snapshot file, one-or-more INCR command-log files, and HISTORY files, coordinated by an atomically-rewritten plaintext manifest, making AOF rewrite crash-safe. | This is exactly the right shape for a durable log: an atomic manifest pointer over immutable segments, base decoupled from incremental, old segments retired by manifest swap rather than in-place mutation. IronCache should adopt a near-identical segment+manifest layout (it also enables cheap point-in-time recovery and parallel load). |
| RDB-preamble hybrid AOF (aof-use-rdb-preamble) | Redis | **borrow** | An AOF rewrite writes a compact RDB image as the file prefix then appends subsequent commands, combining fast load (binary base) with ongoing command-level durability. | Binary base + textual/structured incremental tail is the standard log-structured recovery pattern and gives fast restart. IronCache should use a compact binary base segment plus an incremental record log, exactly this idea. |
| Diskless replication (repl-diskless-sync) + delayed batching | Redis | **adapt** | The snapshot child streams the RDB directly to replica sockets without writing the master's disk; repl-diskless-sync-delay waits a few seconds to fan one snapshot out to multiple joining replicas. | Streaming the snapshot over the socket (skip master disk) is excellent for full-sync of a fresh node and worth adopting. Adapt by sourcing the stream from our in-process snapshot (no fork) and by making the multi-replica fan-out a first-class feature with backpressure rather than a fixed time delay. |
| repl-diskless-load swapdb / on-empty-db | Redis | **reject** | Replica loads the incoming RDB straight from the socket; swapdb keeps the old dataset in RAM (doubling memory) until load completes, on-empty-db only does this when empty, disabled spools to disk first. | The memory-doubling swapdb trap and the abort-on-IO-error fragility are exactly the failure modes a new design should eliminate. IronCache should stream incoming snapshots into a separate arena and switch atomically with a hard, configurable memory cap and graceful abort, never silently doubling RSS. |
| WAIT / WAITAOF per-command durability barriers | Redis | **borrow** | WAIT blocks until N replicas ack a write in memory; WAITAOF blocks until the write is fsynced to N local and/or replica AOFs. | Per-command tunable durability (let the client choose its safety/latency point) is the right primitive and IronCache should expose an equivalent. But we should fix the honesty gap: clearly document that replica-memory-ack (WAIT) is not durability and is not CP. |
| stop-writes-on-bgsave-error (fail-closed on persistence failure) | Redis | **borrow** | By default rejects writes if the last background save failed, so users notice persistence is broken instead of silently losing data. | Fail-closed-on-persistence-failure is a good safety default that surfaces silent durability loss; IronCache should adopt an equivalent guard with a clear error and a metric/alert hook. |
| Transparent Huge Pages disablement requirement | Redis | **reject** | Redis tells operators to set THP to 'never' because post-fork COW on 2MB pages copies far more memory per write than 4KB pages, spiking latency and memory. | Requiring users to tune a kernel sysctl is an operational footgun and contradicts IronCache's easy-single-binary goal. By not using fork at all, IronCache makes THP irrelevant to snapshotting; if we ever madvise our own arenas we control page size explicitly rather than depending on host THP config. |

## Implications for IronCache

- Do not use fork()+COW for snapshotting. Implement an in-process snapshot via our own page/slab-granular copy-on-write (or an immutable LSM/segment log), so snapshot memory overhead is a configurable ceiling rather than 'up to 2x RSS and pray', and there is no page-table-copy stall that scales with heap size.
- Because we avoid fork, the entire THP-COW-blowup and vm.overcommit_memory=1 class of operational footguns disappears — a concrete win for the single-binary, zero-tuning setup goal. Document this as a positive differentiator.
- Adopt the multi-part-AOF shape directly: a directory of immutable segments (one binary base + N incremental logs) coordinated by an atomically-rewritten manifest. This gives crash-safe compaction, parallel load on restart, and a natural basis for point-in-time recovery.
- Use a binary RDB-preamble-style base segment plus a structured incremental record log for fast restart, with CRC checksums per segment (borrow rdbchecksum) and built-in compression on the base (consider zstd/lz4 over Redis's LZF; tie into IronCache's 'smart compression' goal).
- Offer three durability tiers like appendfsync but be honest in the README: strict (group-commit fsync per write batch), interval (background fsync every N ms, with the worst-case loss window stated explicitly as up to 2x the interval if disk is slow), and os-buffered. Expose a live 'durable_offset' / 'lag-behind-fsync' metric.
- Implement group-commit for the strict tier (coalesce concurrent writers into one fdatasync, batch via io_uring) to avoid Redis's throughput collapse under appendfsync always.
- Provide WAITAOF-equivalent per-command durability barriers (block until fsynced locally and/or replicated), but clearly separate 'replicated to memory' from 'durable on disk' in the API and docs to avoid Redis's WAIT confusion.
- Adopt diskless full-sync (stream the in-process snapshot over the socket to a joining node), and design the receive path with a separate arena + atomic switch under a hard memory cap, eliminating Redis's swapdb memory-doubling and abort-on-IO-error fragility.
- Fail-closed on persistence error by default (equivalent of stop-writes-on-bgsave-error) with a structured error and metric, so silent durability loss is impossible to ignore.
- Treat snapshot/compaction as a background task with explicit CPU and IO budgets (rate-limited, akin to rdb-key-save-delay) so persistence never starves the hot path on a single core — directly serving the max-throughput-per-core goal.

## Research papers and primary sources

- **Design and Implementation of Redis 7.0 Multi-Part AOF** (Alibaba Cloud Engineering Blog, 2022). Authoritative walkthrough by the feature's implementers of the base/incr/history segment model and the manifest that replaced in-place AOF rewrite buffer copying, removing AOFRW memory/CPU overhead and making rewrite crash-safe. [source](https://www.alibabacloud.com/blog/design-and-implementation-of-redis-7-0-multi-part-aof_599199) Relevance: Direct blueprint for IronCache's segment+manifest durable log layout; explains the failure modes (in-place rewrite, AOF buffer doubling) we are designing out.
- **Don't Stack Your Log On My Log** (USENIX INFLOW 2014). Log-structured application writes (like AOF) stacked on log-structured filesystems and flash FTLs cause write amplification and unpredictable latency. [source](https://www.usenix.org/system/files/conference/inflow14/inflow14-yang.pdf) Relevance: Informs IronCache's AOF/segment fsync and compaction design on SSDs/NVMe — avoid naive append logs that multiply with the FTL's own log.
- **The Linux implementation of a log-structured file system / RocksDB LSM background** (USENIX FAST (LSM-tree lineage)). LSM/segment-log designs achieve durability and compaction without process fork, using immutable on-disk runs plus an in-memory memtable and a WAL. [source](https://www.usenix.org/conference/fast) Relevance: Provides the alternative to RDB fork-snapshotting: an in-process, bounded-memory durable structure IronCache can use instead of fork()+COW.
- **Redis multi-part AOF implementation PR (#9788)** (redis/redis GitHub, 2021-2022). The actual code and review thread for the 7.0 multi-part AOF: manifest parsing, atomic manifest temp-file rename, history-file cleanup, and the type-marker scheme. [source](https://github.com/redis/redis/pull/9788) Relevance: Primary source for exact on-disk format and atomicity guarantees IronCache should replicate or improve upon.

## Open questions

- What is IronCache's snapshot mechanism if not fork()? Decide between (a) in-process page-granular COW over our own arena, (b) a versioned/MVCC memtable that can be iterated while writes continue, or (c) a full LSM/segment-log where the snapshot is implicit. Each has different memory-overhead and latency profiles to prototype.
- What is the exact worst-case memory overhead ceiling we promise during a snapshot, and how do we enforce it (back-pressure writes vs. throttle the snapshot vs. spill)? Redis's answer is 'unbounded up to ~2x'; ours must be a hard number.
- For the strict durability tier, what is the achievable single-core throughput with group-commit fdatasync vs io_uring batched fsync on NVMe, and how does it compare to Redis appendfsync always?
- Do we expose Redis-wire-compatible WAIT/WAITAOF semantics for compatibility, and if so how do we map our (potentially stronger) durability model onto commands whose Redis semantics are deliberately weak/non-CP?
- How do we make restart load fast on a large dataset — parallel segment load across cores, mmap of the base segment, lazy/on-demand value materialization? Redis single-threaded RDB load is a known cold-start bottleneck.
- What checksum/corruption-recovery policy do we adopt (Redis has aof-load-truncated yes and rdbchecksum)? Define behavior on a torn tail write vs. a corrupt base segment.
- For replication full-sync, what is the backpressure and memory model on the receiving node so we never hit Redis's swapdb doubling or abort-on-IO-error, and can we make snapshot transfer resumable?
- How does persistence interact with smart compression — do we persist compressed values as-is (cheap) or recompress with a stronger codec at base-segment build time (better ratio, more CPU)?

## Proposed issues (seeds for the tracker)

- **[decision, M1]** DECISION: Reject fork()+COW snapshotting; choose IronCache's in-process snapshot mechanism. Decide between in-process page-granular COW, MVCC/versioned memtable, or LSM/segment-log for point-in-time snapshots, with the explicit goal of a bounded, configurable memory overhead and no page-table-copy stall. Document why we reject Redis's fork-based BGSAVE.
- **[research, M0]** RESEARCH: Bound and enforce snapshot memory overhead (vs Redis's unbounded ~2x RSS COW blowup). Quantify the worst-case memory overhead of each candidate snapshot mechanism under write-heavy load and define the enforcement strategy (write back-pressure vs snapshot throttle vs spill) and the hard ceiling we promise.
- **[design, M2]** DESIGN: Segment + atomic manifest durable log (base + incremental + history). Specify a Redis-7.0-multi-part-AOF-style on-disk layout: immutable binary base segment, append-only incremental record logs, and an atomically-rewritten manifest, enabling crash-safe compaction and parallel restart load.
- **[design, M2]** DESIGN: Three durability tiers with honest, documented loss windows. Define strict (group-commit fsync), interval (background fsync every N ms, worst-case loss = up to 2x interval), and os-buffered tiers, plus a live durable_offset/fsync-lag metric — fixing Redis's '1 second' vs real-2-second everysec ambiguity.
- **[research, M0]** RESEARCH: Group-commit fdatasync vs io_uring batched fsync throughput on NVMe. Prototype-measure single-core write throughput for the strict durability tier using coalesced fdatasync and io_uring batched fsync, benchmarked against Redis appendfsync always.
- **[design, M2]** DESIGN: WAIT/WAITAOF-equivalent per-command durability barriers with honest semantics. Expose per-command durability/replication barriers, clearly distinguishing 'replicated to memory' from 'fsynced to disk', and document that replica-ack alone is not CP (the gap Redis's WAIT docs warn about).
- **[design, M2]** DESIGN: Diskless full-sync with bounded-memory atomic dataset switch on the receiver. Stream the in-process snapshot over the socket to a joining node; design the receive path with a separate arena, atomic switch, hard memory cap, and graceful IO-error abort to avoid Redis's swapdb doubling and abort-on-error fragility.
- **[non-goal, M0]** NON-GOAL: Process fork() for persistence and any host THP / vm.overcommit tuning requirement. State explicitly that IronCache will not use fork()-based snapshotting and will not require operators to disable Transparent Huge Pages or set vm.overcommit_memory=1, in service of the zero-tuning single-binary goal.
- **[task, M2]** TASK: Fail-closed-on-persistence-error guard with structured error and metric. Implement an equivalent of stop-writes-on-bgsave-error: when a snapshot/segment write or fsync fails, reject writes by default with a clear error and emit a metric/alert so silent durability loss is impossible.
- **[research, M0]** RESEARCH: Fast parallel restart load of large datasets. Investigate parallel segment load across cores, mmap of the base segment, and lazy value materialization to beat Redis's single-threaded RDB/AOF cold-start load time on large heaps.
- **[design, M2]** DESIGN: Corruption and torn-write recovery policy for base and incremental segments. Define checksumming (per-segment CRC) and recovery behavior for a torn incremental tail vs a corrupt base segment, analogous to Redis aof-load-truncated and rdbchecksum but with explicit, testable semantics.
- **[epic, M1]** EPIC: IronCache durability and persistence subsystem. Umbrella epic tracking the snapshot mechanism decision, segment+manifest log, durability tiers, diskless full-sync, and recovery semantics that together replace Redis RDB+AOF with a fork-free, bounded-memory, honestly-documented durability story.
