## Summary

The credibility problem for IronCache is twofold: proving it is "the most efficient cache in the world" without committing the classic benchmark-war sins, and proving Redis-wire correctness rigorously enough that operators trust a from-scratch reimplementation. On the benchmarking side, the standard tools are memtier_benchmark (the Redis-maintained generator), redis-benchmark/valkey-benchmark (simpler, single-process), and YCSB (cross-store, academically cited). memtier defaults are non-trivial and load-bearing for honesty: SET:GET ratio 1:10, pipeline=1, data-size=32B, key-pattern R:R (uniform random), 4 threads x 50 clients x 10000 requests, percentiles 50/99/99.9. Crucially, memtier DOES support Zipfian (key-pattern Z with --key-zipf-exp, default 1) and Gaussian (G, default stddev = range/6) distributions — a fact widely misreported — so realistic skew is achievable without YCSB. redis-benchmark's notorious trap is that without -r it hammers a single key, which masks cache-miss and contention behavior; this is the canonical example of a benchmark that flatters an in-memory store.

The published vendor numbers are a minefield of methodology asymmetries. Dragonfly's headline "25x faster than Redis" is measured single-instance on a c6gn.16xlarge with memtier configured to give Dragonfly 64 threads but Redis only 2 threads — comparing a multi-threaded store against a deliberately single-threaded one, and never against a Redis Cluster (Redis's actual horizontal-scaling story). Dragonfly's "30% less memory / 3x lower peak during snapshot" comes from io_uring async snapshotting vs Redis's fork()-based BGSAVE memory spike — a real architectural win but measured under update traffic during BGSAVE specifically. Valkey's 8.x numbers (e.g. ~1.19M RPS on c7g.4xlarge with I/O threads, vs ~360K for 7.2; ~678K SET with 6 io-threads vs 239K without) are more transparent but still depend heavily on io-threads, host vs Docker-NAT networking, and pinning. The deepest methodological landmine is coordinated omission (Gil Tene): closed-loop generators like default redis-benchmark/memtier under-report tail latency because when the server stalls, the client stops issuing requests, so the stall's victims are never timed. Open-loop / constant-rate generation (wrk2 style) plus HdrHistogram with expectedInterval correction is the honest fix. The Valkey perf team's own methodology stresses bare-metal, taskset pinning, loopback to remove the network as the bottleneck, and isolating CPU — the right primitives for a throughput-per-core metric.

For IronCache the efficiency thesis ("max throughput per core, minimal memory, smart compression") demands metrics that vendors usually avoid: throughput-per-core (not just aggregate RPS on a 64-core box), memory-at-fixed-hit-ratio (RSS to sustain a target hit ratio under a Zipfian working set — directly tests compression and overhead-per-key), and tail latency p99/p999/p9999 under open-loop load at a fixed offered rate. A defensible benchmark harness must publish: the exact tool+flags, hardware, pinning, warmup, distribution and zipf exponent, pipeline depth (and report pipelined vs non-pipelined separately because pipelining inflates RPS by amortizing syscalls), hit ratio, and CO-corrected histograms. Comparisons must be apples-to-apples on thread count, or explicitly framed as "per-core" to neutralize the Dragonfly-style thread asymmetry.

On correctness, the field has matured well past "run the unit tests." The Redis/Valkey TCL suite (tests/unit/, tests/integration/) is the de-facto conformance oracle, with a precise assertion vocabulary — assert_equal, assert_error, assert_encoding (checks OBJECT ENCODING: listpack/intset/quicklist/hashtable/skiplist/embstr/int), assert_type, assert_replication_stream. IronCache should run a per-command conformance suite either by porting these tests or by differential testing against a real redis-server (send identical command streams, diff RESP replies and OBJECT ENCODING). Distributed correctness is best proven Jepsen-style: the 2020 Redis-Raft analysis found 21 issues including split-brain, lost updates, stale reads, and total data loss on failover, using the Elle checker (black-box, cycle-detection over Adya dependency graphs, list-append workload, near-linear in history length and constant in concurrency — unlike Knossos which blows up exponentially). Elle detects G0/G1a/G1b/G1c/G-single/G2 and lost update. The most powerful modern technique is Deterministic Simulation Testing (FoundationDB Flow, TigerBeetle VOPR, now commercially via Antithesis): run the entire system single-threaded over simulated time/network/disk/RNG driven by one seed, inject partitions/crashes/clock-skew, check invariants, and replay any failure exactly from its seed. This is a strong fit for a green-field Rust codebase if concurrency and I/O are abstracted behind a deterministic runtime from day one.

The practical correctness gate stack for IronCache: (1) coverage-guided fuzzing of the RESP2/RESP3 parser (cargo-fuzz/libFuzzer or AFL++) — the parser is the attack surface and a frequent CVE source; (2) property-based tests (proptest/quickcheck/bolero) for data-structure invariants and a model-based oracle comparing IronCache against an in-memory reference model of each type; (3) a per-command conformance suite as a CI gate, ideally differential against pinned redis-server versions; (4) a DST harness as the architecture matures, with seeds archived in CI; (5) Jepsen+Elle for any clustering/replication mode before it ships. The cross-cutting decision is that determinism must be designed in early (single-threaded-per-shard or an abstracted async runtime), because retrofitting DST onto an arbitrary-threaded codebase is the expensive failure mode every DST adopter warns about.

## Mechanisms: borrow, adapt, or reject

| Mechanism | System | Stance | What it does | Rationale for IronCache |
| --- | --- | --- | --- | --- |
| memtier_benchmark as primary load generator | memtier_benchmark | **borrow** | Multi-threaded, multi-client generator with configurable SET:GET ratio, pipeline depth, data-size distributions, key patterns (uniform/sequential/Gaussian/Zipf), arbitrary --command support, cluster mode, and HDR histogram output (--hdr-file-prefix). | It is the Redis-ecosystem standard, supports Zipfian (--key-zipf-exp) and Gaussian skew natively, emits HdrHistogram files for honest tails, and can drive arbitrary commands. IronCache should adopt it as the canonical generator so numbers are directly comparable to Redis/Valkey/Dragonfly published runs. |
| redis-benchmark default single-key workload | redis-benchmark | **reject** | Simple generator that, without -r, repeatedly operates on one key, and reports throughput plus basic latency. | The single-key default produces inflated, unrealistic numbers (no contention/keyspace pressure) and it is closed-loop with weak tail reporting. Use it only as a smoke test, never as a headline benchmark; prefer memtier + open-loop tooling for published claims. |
| YCSB core workloads A-F with Zipfian | YCSB | **adapt** | Standardized cross-store workloads (50/50, 95/5, read-only, read-latest, scan, RMW) with uniform/zipfian/latest distributions, default zipfian constant 0.99. | Valuable for cross-database, academically-citable comparisons and for the read-latest distribution. But YCSB's JVM client is a throughput bottleneck and lacks pipelining; use it for distribution definitions and cross-store credibility, not for peak single-node throughput-per-core claims. |
| Open-loop / constant-rate load + coordinated-omission correction | wrk2 + HdrHistogram (Gil Tene) | **borrow** | Issues requests at a fixed offered rate regardless of server progress and records latency into an HdrHistogram with expected-interval correction so stalls are attributed to all delayed requests. | Required to report honest p99/p999/p9999. Without it, IronCache (and every competitor) systematically hides tail latency. IronCache's benchmark gate must include open-loop runs at fixed rates with CO-corrected histograms; this is a credibility differentiator. |
| Throughput-per-core and memory-at-fixed-hit-ratio metrics | Valkey perf methodology (adapted) | **borrow** | Bare-metal + taskset pinning + loopback to isolate CPU; measure aggregate RPS divided by cores, and RSS required to sustain a target hit ratio under a Zipfian working set. | These two metrics directly encode IronCache's thesis (max throughput per core, minimal memory, smart compression) and neutralize the Dragonfly-style thread-count asymmetry. They are the primary scoreboard, not aggregate RPS on a fat box. |
| Redis/Valkey TCL conformance suite + assert_encoding | Valkey/Redis tests | **adapt** | Per-command TCL tests with assertions assert_equal/assert_error/assert_type/assert_encoding (validates OBJECT ENCODING transitions like listpack->quicklist, intset->hashtable) and assert_replication_stream. | This is the de-facto Redis behavior oracle. Port/adapt it (or run it against IronCache via the Valkey harness) as a CI conformance gate. Adapt rather than borrow verbatim because IronCache's internal encodings differ; encoding assertions should map to IronCache's own OBJECT ENCODING reporting or be marked as compatibility-shimmed. |
| Differential testing against pinned redis-server | IronCache (proposed) vs Redis | **borrow** | Send identical fuzzed/random command streams to IronCache and a pinned real redis-server, diff RESP replies, error messages, and OBJECT ENCODING. | The strongest, lowest-effort way to prove wire/semantic compatibility for a reimplementation: the real server is the oracle. Catches subtle reply-shape and error-string mismatches that hand-written tests miss. |
| RESP2/RESP3 parser coverage-guided fuzzing | cargo-fuzz/libFuzzer or AFL++ | **borrow** | Feeds mutated byte streams into the protocol parser to find panics, OOB reads, integer overflow in bulk-length handling, and DoS via malformed multibulk. | The parser is the primary remote attack surface and a recurring CVE source in caches. In Rust this is cheap (cargo-fuzz) and catches the bugs unit tests cannot. Must be a CI fuzz-gate with a seed corpus from real RESP captures. |
| Property-based + model-based testing of data structures | proptest/quickcheck/bolero | **borrow** | Generates random operation sequences against each type (string/list/set/zset/hash/stream) and asserts invariants and equivalence to a simple in-memory reference model. | Property tests with a reference-model oracle catch encoding-transition and edge-case bugs (e.g. listpack overflow thresholds, ziplist->skiplist) far more thoroughly than examples. bolero can also run the same properties under libFuzzer for extended campaigns. |
| Deterministic Simulation Testing (Flow/VOPR/Antithesis style) | FoundationDB / TigerBeetle / Antithesis | **adapt** | Runs the whole system single-threaded over simulated time/network/disk/RNG from one seed, injects partitions/crashes/clock-skew/disk-faults, checks invariants, and replays any failure exactly. | Gold standard for distributed correctness and the highest-leverage long-term investment, but it constrains architecture (must abstract I/O/time/concurrency behind a deterministic runtime). Adopt for multi-node IronCache, and design the runtime for determinism in M1 so it is not a retrofit. For single-node M2, a lighter deterministic test harness suffices. |
| Jepsen + Elle for clustering/replication | Jepsen / Elle | **borrow** | Black-box consistency testing: drives a real cluster under faults, collects client-observed histories, and uses Elle's cycle detection to find G0/G1a/G1b/G1c/G-single/G2 and lost-update anomalies. | Any IronCache clustering/replication mode must pass a Jepsen test before claiming a consistency level, given Redis-Raft itself shipped 21 issues including split-brain. Elle is near-linear and black-box, so it works without instrumenting IronCache internals. |

## Implications for IronCache

- Publish a frozen, scripted benchmark harness (exact memtier flags, hardware, kernel, pinning, warmup, distribution + zipf exponent, pipeline depth, hit ratio) so every claim is reproducible; treat unreproducible vendor numbers as the anti-pattern to avoid.
- Make throughput-per-core and memory-to-sustain-a-target-hit-ratio (under a Zipfian working set) the headline metrics, not aggregate RPS on a high-core box — this directly encodes the efficiency thesis and immunizes IronCache against Dragonfly-style thread-count comparisons.
- Always report tail latency p99/p999/p9999 from open-loop, constant-rate load with HdrHistogram coordinated-omission correction; never headline a closed-loop p99. Ship CO-corrected histograms as build artifacts.
- Report pipelined and non-pipelined throughput as separate, explicitly-labeled numbers; pipelining amortizes syscalls and can inflate RPS by an order of magnitude, so mixing them is a benchmark-war trap.
- When comparing to Redis/Valkey/Dragonfly, fix thread counts symmetrically OR frame everything per-core, and always include a Redis Cluster comparison for any multi-node claim (since vendors omit it).
- Adopt the Redis/Valkey TCL conformance vocabulary as a CI gate: run a per-command suite and, where encodings differ, expose IronCache's own OBJECT ENCODING so assert_encoding-style checks remain meaningful; supplement with differential testing against pinned redis-server versions.
- Stand up parser fuzzing (cargo-fuzz on the RESP2/RESP3 decoder) and property/model-based tests for every data type from the first prototype; seed the fuzz corpus with real RESP captures.
- Decide the determinism stance in M1: abstract time/network/disk/RNG behind a runtime so Deterministic Simulation Testing is feasible later without a rewrite; archive failing seeds in CI for exact replay.
- Gate any clustering/replication mode behind a passing Jepsen+Elle run before publicly claiming a consistency level (e.g. linearizable, causal), and document the exact failure modes (split-brain, lost writes on failover) that the mode does and does not tolerate.
- Define explicit non-goals: e.g. IronCache will not claim a consistency guarantee it has not Jepsen-tested, and will not publish a headline throughput number without a CO-corrected tail-latency companion and per-core normalization.

## Research papers and primary sources

- **Benchmarking Cloud Serving Systems with YCSB** (SoCC 2010). Defines the YCSB framework: core workloads A-F, request distributions (uniform, zipfian default ~0.99, latest), and a tier-1 (performance) / tier-2 (scaling) methodology for cross-store comparison. [source](https://courses.cs.duke.edu/fall13/cps296.4/838-CloudPapers/ycsb.pdf) Relevance: Canonical reference for realistic key-distribution and read/write mixes; IronCache should cite it for distribution definitions and cross-store credibility.
- **Elle: Inferring Isolation Anomalies from Experimental Observations** (VLDB 2020 (arXiv 2003.10554)). Black-box transactional safety checker using cycle detection over Adya dependency graphs; detects G0/G1a/G1b/G1c/G-single/G2 in near-linear time using a list-append workload, vastly outscaling Knossos. [source](https://people.ucsc.edu/~palvaro/elle_vldb21.pdf) Relevance: The checker IronCache's clustering tests should use; defines the precise anomaly taxonomy any consistency claim must rule out.
- **Jepsen: Redis-Raft 1b3fbf6** (Jepsen analysis 2020). Empirical analysis finding 21 issues (split-brain, lost updates, stale/aborted reads, total data loss on failover) in a Raft-based Redis, and clarifying that WAIT does not give strong consistency. [source](https://jepsen.io/analyses/redis-raft-1b3fbf6) Relevance: Direct evidence that Redis-lineage clustering is hard to get right; sets the bar IronCache's clustering must clear before any consistency claim.
- **FoundationDB: A Distributed Unbundled Transactional Key Value Store** (SIGMOD 2021). Describes Flow, the deterministic single-threaded actor runtime enabling whole-system simulation testing where any failure is reproducible from one seed across simulated network/disk/clock faults. [source](https://www.foundationdb.org/files/fdb-paper.pdf) Relevance: The foundational case for Deterministic Simulation Testing and why determinism must be an architectural decision, not a bolt-on, for IronCache.
- **How NOT to Measure Latency (Coordinated Omission)** (Strange Loop 2015 (Gil Tene)). Closed-loop load generators systematically omit the latency of requests delayed by server stalls; constant-rate open-loop generation plus HdrHistogram expected-interval correction restores honest tail percentiles. [source](https://github.com/giltene/wrk2) Relevance: The single most important methodology fix for IronCache to report credible p99/p999/p9999 instead of flattering closed-loop numbers.

## Open questions

- Single-threaded-per-shard vs an abstracted async runtime: which concurrency model best supports both max throughput-per-core AND Deterministic Simulation Testing without forcing a rewrite later?
- What exactly is IronCache's compatibility contract — bit-identical RESP replies and error strings, or behavioral equivalence only? This decides whether differential testing must match error message text verbatim.
- Should IronCache expose Redis-style OBJECT ENCODING values (listpack/quicklist/etc.) for conformance compatibility even if its internal representations differ, or report its own encodings and shim the test suite?
- Which Redis/Valkey versions to pin as the conformance/differential oracle, and how to track behavioral drift as upstream evolves (Redis vs Valkey divergence post-fork)?
- What is the canonical published hardware/config matrix for headline benchmarks (instance types, core counts, pinning, kernel/io_uring settings) so results stay comparable over time?
- For memory-at-fixed-hit-ratio, what reference working-set generator and zipf exponent define the 'standard' IronCache memory benchmark, and how is hit ratio held constant across implementations?
- Do we build our own deterministic simulator (Flow/VOPR-style) or pay for Antithesis, and at what milestone does the cost/benefit cross over?
- Which fuzzing harness (cargo-fuzz/libFuzzer vs AFL++ vs bolero) and what RESP3 push/out-of-band coverage is needed beyond the request parser (e.g. inline commands, RESP3 attributes, pub/sub frames)?
