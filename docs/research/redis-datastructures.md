## Summary

Redis's memory model is built around a small polymorphic object header (robj/redisObject) that points to one of several internal encodings selected dynamically per value. In the classic layout (verified at 7.4.0, src/server.h) the header is a 4-bit type, 4-bit encoding, 24-bit lru packed into one 32-bit word, a 4-byte int refcount, and an 8-byte void* ptr = 16 bytes on 64-bit. A landmark change shipped in Redis 8.x: the "kvobj" redesign (verified at 8.8.0, src/object.h) repacks the header to type:4, encoding:4, refcount:23, iskvobj:1, metabits:8, lru:24, ptr — and, critically, embeds the key SDS and optional metadata (expire) directly after the object via a single allocation, plus dict.c bucket pointer-tagging to skip dictEntry allocation entirely when a bucket holds a single key. This is precisely the per-key-overhead reduction IronCache cares about: it collapses what used to be (dictEntry + key robj + key SDS + value robj + value SDS) into far fewer allocations.

Strings use SDS with five header variants (sdshdr5/8/16/32/64) chosen by length; sdshdr8 is a 3-byte header (1B len, 1B alloc, 1B flags). Small strings (<=44 bytes, OBJ_ENCODING_EMBSTR_SIZE_LIMIT in object.c) are stored embedded in the same allocation as the robj (createEmbeddedStringObject), always SDS_TYPE_8; longer strings are OBJ_ENCODING_RAW (separate allocation). Integer-parseable strings up to 20 digits become OBJ_ENCODING_INT and share a pool of 10000 pre-allocated shared integer objects (OBJ_SHARED_INTEGERS), which is disabled under maxmemory + LRU/LFU eviction because shared objects can't carry per-key LRU.

The compact collection encodings all converge on listpack (src/listpack.c), a single contiguous, malloc-friendly, cache-friendly blob with a 6-byte header (LP_HDR_SIZE: 32-bit total-bytes + 16-bit element count) and per-entry back-length for reverse traversal; it replaced ziplist (the ZIPLIST encoding constant is retained only as "no longer used"). Lists are a quicklist (src/quicklist.h): a doubly linked list of 32-byte quicklistNodes each wrapping a listpack, with optional LZF compression of interior nodes (list-compress-depth) and a fill factor (list-max-listpack-size default -2 = 8KB per node). Hashes and sorted sets start as listpack and convert to hashtable (dict) / skiplist+hashtable respectively once thresholds are crossed; sets use intset (sorted int array, src/intset.h: 4-byte encoding + 4-byte length + contents[]) for all-integer small sets, listpack for small mixed sets, and hashtable otherwise. Streams use a radix tree (rax, src/rax.h: raxNode with iskey/isnull/iscompr/size:29 bitfields, path compression) keyed by 128-bit stream IDs whose leaves are listpacks of entries. HyperLogLog (src/hyperloglog.c) uses P=14 -> 16384 6-bit registers; dense form is HLL_HDR_SIZE(16) + 12288 = 12304 bytes, sparse form is a run-length-encoded opcode stream capped by hll-sparse-max-bytes (default 3000) before promotion to dense. Bitmaps/GETBIT/SETBIT are just operations over a raw SDS string.

The encoding thresholds (verified in src/config.c at 8.8.0) are: hash-max-listpack-entries 512 / hash-max-listpack-value 64; list-max-listpack-size -2 (8KB); set-max-intset-entries 512; set-max-listpack-entries 128 / set-max-listpack-value 64; zset-max-listpack-entries 128 / zset-max-listpack-value 64. Conversions are one-way (compact -> general) and triggered on insert. The general hash table (dict) keeps two tables (ht_table[2], ht_used[2], ht_size_exp[2], rehashidx) and rehashes incrementally one bucket-step at a time, expanding when load factor crosses dict_force_resize_ratio (4) and shrinking below 1/(HASHTABLE_MIN_FILL * dict_force_resize_ratio) = 1/32.

For IronCache the strategic lessons are: (1) the single-allocation embedded-key/embedded-value pattern (Redis 8 kvobj + Valkey's earlier embedded-key work) is the biggest per-key memory win and should be a first-class design goal; (2) the listpack-as-universal-compact-container idea is excellent and worth borrowing wholesale, but the fixed scalar thresholds (128/512/64) are crude and a candidate for adaptive/learned tuning; (3) incremental rehashing is non-negotiable for tail-latency on a single-threaded-per-core event loop and must be designed in from the start; (4) the 16-byte (or 16+ byte) per-object header dominates small-value workloads, so a header-free / inline path for tiny values is where the "minimal memory" claim is won or lost.

## Mechanisms: borrow, adapt, or reject

| Mechanism | System | Stance | What it does | Rationale for IronCache |
| --- | --- | --- | --- | --- |
| Embedded-key / embedded-value single allocation (kvobj + embstr) | Redis 8.x (src/object.h, object.c) / Valkey embedded-key work | **borrow** | Collapses key SDS, value object header, and small value buffer into a single malloc by embedding the key into the value object (kvobj) and embedding strings <=44B into the robj (embstr), and uses dict bucket pointer-tagging to skip dictEntry allocation for single-key buckets. | This is the single largest per-key memory win available and directly serves IronCache's 'minimal memory' goal. Fewer allocations also means fewer cache misses and less allocator metadata overhead. IronCache should make a one-allocation-per-key layout the default from day one rather than retrofitting it as Redis had to. |
| SDS with multiple header sizes (sdshdr5/8/16/32/64) | Redis (src/sds.h) | **adapt** | Length-prefixed byte strings whose header (len+alloc fields) scales from 1 byte (sdshdr5) to 17 bytes (sdshdr64) based on string length, keeping small-string overhead minimal while supporting large strings and O(1) length. | The size-class header idea is excellent and worth keeping, but in Rust we'd implement it as a custom unsized type / NonNull<u8> with a variable-width prefix rather than C packed structs. Drop the legacy SDS_TYPE_5 path complexity; keep the embedded-vs-raw split tied to value-inlining. |
| listpack universal compact container | Redis (src/listpack.c, replaced ziplist) | **borrow** | A single contiguous, allocation-friendly blob with a 6-byte header and self-describing variable-length entries (with back-length for reverse iteration) used as the small-collection encoding for hashes, sets, sorted sets, quicklist nodes, and stream entry groups. | One cache-friendly contiguous container shared across all small collection types is exactly the right primitive for max-throughput-per-core and low memory. IronCache should adopt a listpack-equivalent (fixing ziplist's O(n) cascade-update flaw, which listpack already largely addressed) as its universal compact encoding. |
| Fixed scalar encoding-conversion thresholds | Redis (src/config.c: *-max-listpack-entries/value, set-max-intset-entries) | **adapt** | Hard-coded per-type integer limits (128/512 entries, 64-byte values) that trigger a one-way promotion from compact encoding to the general hashtable/skiplist representation. | The mechanism (promote-on-threshold) is sound and must stay, but the magic constants are crude and workload-blind, and conversions are one-way (no demotion when a collection shrinks). IronCache should keep promotion but consider self-tuning thresholds (size/access-pattern aware) and optional demotion, aligning with the 'AI-driven innovation' goal. |
| quicklist (linked list of LZF-compressed listpacks) | Redis (src/quicklist.h/.c) | **adapt** | Represents lists as a doubly linked list of 32-byte nodes each holding a listpack (default 8KB), with interior nodes optionally LZF-compressed (list-compress-depth) for cold middle data. | Good for huge lists and a nice compression hook, but the 32-byte node + pointer chasing hurts cache locality for medium lists, and LZF is dated. IronCache should keep the chunked-listpack idea but evaluate a more cache-friendly chunk index (e.g. a small B-tree/rope of chunks) and a modern codec (LZ4/zstd) for cold chunks, tying into the 'smart compression' goal. |
| intset (sorted packed integer array) | Redis (src/intset.h) | **borrow** | Stores all-integer sets as a sorted contiguous int16/int32/int64 array (encoding upgrades in place) with binary-search membership; extremely compact, no per-element pointers. | Near-optimal memory and cache behavior for integer sets, a very common workload (IDs). O(n) insert is acceptable up to the 512-entry cap. IronCache should keep this verbatim and possibly extend the cap or add a delta/bitset variant for dense integer ranges. |
| incremental two-table rehashing | Redis (src/dict.c) | **borrow** | Maintains two hash tables during growth/shrink and migrates a bounded number of buckets per operation (amortized), reading from both tables until migration completes, avoiding a stop-the-world rehash. | Essential for bounded tail latency in a single-threaded-per-core event loop; a monolithic rehash would cause multi-ms stalls on large keyspaces. IronCache must design its hashtable around incremental migration (or a concurrent/lock-free scheme) from the outset. |
| shared integer object pool (10000 objects) | Redis (src/server.h OBJ_SHARED_INTEGERS, object.c) | **adapt** | Pre-allocates and reuses immutable robj instances for integers 0-9999 to avoid per-value allocation for common small integers; disabled under LRU/LFU maxmemory eviction. | The interning idea saves allocations for hot small integers, but Redis's 16-byte shared robj and the eviction incompatibility are baggage. In IronCache, if values are already inlined into the key object (no separate value allocation), interning yields little; better to inline small integers directly in the value slot and skip a shared pool entirely. |
| rax (radix tree) stream index | Redis (src/rax.c/.h) | **adapt** | Path-compressed radix tree keyed on 128-bit stream IDs whose leaves point to listpacks of entries, giving ordered range scans and compact storage of sorted IDs. | A solid structure for ordered 128-bit-key indexes, but streams are a lower priority for an efficiency-first cache. IronCache should treat stream support as optional/later (non-goal for M0-M2) and, if built, reuse a generic ordered-map (e.g. a B-tree or ART) rather than a bespoke rax. |
| HyperLogLog dense/sparse dual encoding | Redis (src/hyperloglog.c) | **borrow** | Probabilistic cardinality with P=14 (16384 6-bit registers, ~0.81% error); starts in a run-length sparse encoding and promotes to a 12304-byte dense array past hll-sparse-max-bytes (3000). | Memory-efficient cardinality estimation with a clean sparse->dense ladder is exactly the kind of compact probabilistic structure that fits IronCache's efficiency thesis; the dual-encoding pattern is reusable. Keep P=14 default for wire compatibility but allow configurable precision. |

## Implications for IronCache

- Make single-allocation-per-key the default object layout: embed the key bytes, the small value, and per-key metadata (TTL, eviction bits) into one allocation, matching/exceeding Redis 8's kvobj + embstr + dict pointer-tagging. This is where the 'minimal memory' claim is won; target <16 bytes of fixed overhead per small key.
- Adopt a listpack-equivalent as the single universal compact container for hashes/sets/zsets and list chunks, but fix ziplist's historical cascade-update pathology up front and design entry encoding for SIMD-friendly scans to maximize throughput-per-core.
- Design the main hashtable around incremental rehashing (or a concurrent/lock-free resize) from the start; never allow a stop-the-world rehash. Specify the per-op migration budget as a tunable and verify p99.9 latency under continuous resize in benchmarks.
- Replace Redis's fixed scalar conversion thresholds (128/512/64) with self-tuning, workload-aware promotion (and optional demotion) policy, surfaced as the 'AI-driven innovation' angle; ship Redis-compatible defaults but allow an adaptive mode.
- Keep intset and a HyperLogLog sparse/dense ladder verbatim for wire compatibility and because they are already near-optimal; consider a dense-integer-range bitset variant and configurable HLL precision as efficiency extensions.
- Inline small integers directly into the value slot rather than maintaining a 10000-entry shared-integer pool; with inlined values the pool saves little and adds eviction-policy complexity.
- Modernize the cold-data compression hook: where Redis quicklist uses LZF on interior nodes, evaluate LZ4/zstd dictionary compression on cold chunks/values to serve the 'smart compression' goal, with a per-chunk compressed/uncompressed flag like quicklistNode.recompress.
- Treat Streams/rax as an optional, post-prototype feature (non-goal for M0-M2); if implemented later, reuse a generic ordered map (B-tree or ART) instead of a bespoke radix tree to reduce code surface in a single static binary.
- Document an explicit per-key and per-encoding memory model (header bytes + container bytes + allocator rounding) in the README/vision, with measured numbers, since per-key overhead is the headline metric IronCache competes on against Redis/Valkey/Dragonfly.

## Research papers and primary sources

- **HyperLogLog: the analysis of a near-optimal cardinality estimation algorithm** (AofA 2007 (Flajolet, Fusy, Gandouet, Meunier)). Cardinality estimation using m registers each tracking the max leading-zero count of hashed values, with a harmonic-mean estimator giving ~1.04/sqrt(m) relative error. [source](https://algo.inria.fr/flajolet/Publications/FlFuGaMe07.pdf) Relevance: Directly underpins Redis's HLL (P=14 -> m=16384 -> ~0.81% error). Defines the math IronCache must match for OSS-compatible PFADD/PFCOUNT semantics.
- **HyperLogLog in Practice: Algorithmic Engineering of a State of The Art Cardinality Estimation Algorithm** (EDBT 2013 (Heule, Nunkesser, Hall, Google)). Sparse representation, bias correction for small cardinalities, and 64-bit hashing to remove large-range correction; the basis for HLL++. [source](https://research.google/pubs/pub40671/) Relevance: Justifies Redis's sparse encoding and informs whether IronCache should adopt HLL++ improvements (bias correction, sparse->dense ladder) for better accuracy at low cardinality.
- **Adaptive Radix Tree: ARTful Indexing for Main-Memory Databases** (ICDE 2013 (Leis, Kemper, Neumann)). A radix tree with adaptive node sizes (Node4/16/48/256) and path compression that is both space-efficient and cache-friendly for in-memory ordered indexes. [source](https://db.in.tum.de/~leis/papers/ART.pdf) Relevance: A concrete, more cache-efficient alternative to Redis's rax for any ordered-key index (streams, sorted structures) IronCache might build.
- **Cuckoo Filter: Practically Better Than Bloom** (CoNEXT 2014 (Fan, Andersen, Kaminsky, Mitzenmacher)). A compact approximate-membership structure supporting deletion with better space efficiency than Bloom filters at low false-positive rates. [source](https://www.cs.cmu.edu/~dga/papers/cuckoo-conext2014.pdf) Relevance: Informs compact probabilistic set membership beyond Redis's intset/HLL; a candidate efficiency extension for IronCache's probabilistic data types.
- **Scaling Memcache at Facebook** (NSDI 2013 (Nishtala et al.)). Production lessons on slab allocation, memory fragmentation, and per-item overhead at cache scale; the cost of per-item metadata dominates small-value workloads. [source](https://www.usenix.org/system/files/conference/nsdi13/nsdi13-final170_update.pdf) Relevance: Reinforces that per-key/per-item overhead and allocator behavior, not algorithmic cleverness, decide real-world cache memory efficiency — IronCache's core thesis.

## Open questions

- What is IronCache's exact target per-key fixed overhead (bytes) for a small string value, and how does it compare to measured Redis 8.x kvobj (~ header + embedded key SDS + embedded value) and Valkey?
- Should the universal compact container be a listpack clone (byte-packed, reverse back-length) or a redesigned layout optimized for SIMD scanning and Rust's ownership model?
- Will IronCache use jemalloc/mimalloc-style slab classes (like Redis's libc/jemalloc reliance) and how does allocator rounding interact with the single-allocation key layout to avoid hidden waste?
- Should encoding conversions be one-way (Redis behavior, simpler) or support demotion when collections shrink, and what hysteresis avoids thrash near thresholds?
- For incremental rehashing under a thread-per-core architecture, is a two-table amortized migration (Redis dict) sufficient, or should IronCache use a concurrent/lock-free hash table (e.g. open-addressing with epoch-based resize)?
- Which probabilistic structures are in scope for M2 (HLL only, or also Cuckoo/Bloom/Top-K/t-digest) given the single-static-binary size budget?
- How should IronCache report OBJECT ENCODING for Redis-wire compatibility when its internal encodings differ (e.g. expose synthetic 'listpack'/'hashtable' names)?
- Is Streams (and thus a rax/ART index) in scope at all for the first releases, or an explicit non-goal?

## Proposed issues (seeds for the tracker)

- **[decision, M1]** Decide IronCache's per-key object layout (embedded key + inlined small value, single allocation). Specify a one-allocation-per-key object layout that embeds the key bytes, small values, and per-key metadata (TTL, eviction bits), matching or beating Redis 8.x kvobj + embstr and dict bucket pointer-tagging. Set an explicit byte-overhead target for small keys.
- **[design, M1]** Design the universal compact collection container (listpack-equivalent). Define a single contiguous, cache/SIMD-friendly compact container used by hashes, sets, zsets, and list chunks, with a small header and self-describing entries, avoiding ziplist's cascade-update pathology.
- **[design, M1]** Specify the main hashtable with incremental (non-stop-the-world) rehashing. Adopt incremental two-table or concurrent resize so growth/shrink never stalls the event loop; define per-operation migration budget and load-factor expand/shrink thresholds (Redis baseline: expand at 4, shrink below 1/32).
- **[research, M0]** Research: adaptive vs fixed encoding-conversion thresholds. Evaluate replacing Redis's fixed thresholds (hash/zset/set-max-listpack-entries 128/512, value 64; set-max-intset-entries 512; list-max-listpack-size -2) with workload-aware self-tuning promotion/demotion, including hysteresis to avoid thrash.
- **[decision, M1]** Decide string size-class header scheme (SDS-style variable headers in Rust). Choose a variable-width length-prefixed string representation (analogous to sdshdr5/8/16/32/64) and the embstr<=44B inline-vs-raw split, implemented idiomatically in Rust without the legacy SDS_TYPE_5 complexity.
- **[task, M2]** Adopt intset and HyperLogLog sparse/dense encodings for wire compatibility. Implement an intset-equivalent for all-integer sets (sorted packed array, in-place width upgrade, 512 cap) and an HLL with P=14/16384 6-bit registers plus a sparse->dense ladder (promote past ~3000 bytes) for PFADD/PFCOUNT compatibility.
- **[research, M0]** Define cold-data compression strategy (modern codec vs Redis LZF). Compare LZ4/zstd (with optional shared dictionaries) against Redis quicklist's LZF for compressing cold chunks/values, including a per-chunk compressed/recompress flag, to serve the 'smart compression' goal without hurting hot-path throughput.
- **[non-goal, M0]** Non-goal: Streams / radix-tree index in initial releases. Declare Redis Streams (and a bespoke rax index) out of scope for M0-M2; if pursued later, reuse a generic ordered map (B-tree or Adaptive Radix Tree) rather than a custom radix tree to limit binary size and code surface.
- **[design, M2]** Define OBJECT ENCODING compatibility mapping. Specify how IronCache reports OBJECT ENCODING (and DEBUG OBJECT) for Redis-wire compatibility when internal encodings differ, exposing synthetic names like 'embstr', 'int', 'listpack', 'intset', 'hashtable', 'skiplist', 'quicklist'.
- **[meta, M0]** Establish a measured per-encoding memory model and benchmark harness. Build a harness that measures real per-key and per-encoding memory (header + container + allocator rounding) versus Redis 8.x/Valkey/Dragonfly, so the 'minimal memory' claim is backed by reproducible numbers in the README and vision EPIC.
