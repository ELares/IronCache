## Summary

Valkey is the BSD-3-Clause, Linux Foundation-governed fork of Redis, created in March 2024 after Redis Ltd. relicensed Redis 7.4 to dual SSPLv1/RSALv2. Its license (verified directly in the repo COPYING file at tag 8.1.8: SPDX BSD-3-Clause) and neutral governance are the strategic reason it matters for IronCache: it is the permissively-licensed, AWS/Google/Oracle-backed continuation of the Redis wire protocol and RESP semantics that IronCache aims to be compatible with. We should treat Valkey (not relicensed Redis) as the canonical RESP-compatibility reference and conformance oracle. Note that as of 2026-06-13 the line has advanced well past the prompt's 8.0/8.1 framing: 9.0.0 GA shipped 2025-10-21, 9.1.0 on 2025-05-19 (sic — actually 2026; latest tags pulled live: 9.1.0, 8.1.8, 9.0.4), so 8.0/8.1 are now the historical baseline of the threading/memory work, with 9.x extending it (zero-copy responses, MPTCP, AVX-512, atomic slot migration, cluster multi-DB).

Threading model: Valkey 8.0's "asynchronous I/O threading" (AWS-contributed) keeps command EXECUTION single-threaded on the main thread but offloads the full connection lifecycle — socket read, RESP protocol PARSING, reply formatting, socket write, epoll polling, and even memory deallocation — to I/O threads via per-thread single-producer/single-consumer atomic ring-buffer job queues (verified in src/io_threads.c at 8.1.8: cache-line-aligned _Atomic head/tail IOJobQueue). This is strictly more aggressive than Redis's io-threads, which historically threaded only the read/write syscalls. The headline number is ~1.2M QPS on AWS r7g vs ~380K previously (~3.16x), with TLS workloads benefiting most. Crucially, io-threads defaults to OFF (the `io-threads 4` line is commented in valkey.conf; setting 1 = main-thread-only), and the old `io-threads-do-reads` knob is deprecated/no-op. The execution path stays single-threaded, so this scales I/O, not command throughput — the well-understood ceiling that Dragonfly critiques and that IronCache must decide whether to accept or break.

Memory efficiency is Valkey's most directly borrowable work. 8.0 delivered ~20% RSS reduction vs 7.2 (benchmark: 693.64MB→550.56MB at ~6.3M keys) via two mechanisms: (1) per-slot dictionaries / the kvstore refactor — splitting the single global hashtable into 16,384 per-slot dictionaries eliminated the two cluster slot-tracking linked-list pointers, saving 16 bytes/entry; and (2) embedding the key's SDS directly into the dictionary entry instead of a separate pointer+allocation, saving ~8 bytes/entry and improving cache locality. 8.1 went further by replacing the chaining `dict` entirely with a new open-addressing `hashtable` (designed by Viktor Söderqvist, Swiss-tables-inspired, bucket-chaining idea from Madelyn Olson), giving a ~20 byte/entry reduction for keys without TTL and ~30 bytes with TTL, plus ~47% lower fork copy-on-write. Verified in src/hashtable.c@8.1.8: 64-bit layout uses cache-line-sized buckets of 7 entries (ENTRIES_PER_BUCKET 7) carrying a presence bitmask + 7 one-byte stored hashes + 7 entry pointers, with soft max fill ~91.43%. This is essentially the design IronCache should start from.

Replication and transport: dual-channel replication (8.0, #60) streams the RDB and the replica backlog on two parallel connections during full sync, cutting sync time up to ~50% under heavy reads and moving Client-Output-Buffer tracking to the replica to relieve primary memory pressure — but it defaults OFF (`dual-channel-replication-enabled no` at 8.1.8). Valkey Over RDMA (8.0, #477) is an experimental, Linux-only connection module claiming up to +275% throughput; it remains explicitly "may be removed in any minor/major version" and does not support replication (TCP/TLS still required). For IronCache's single-binary, single-node-first ethos these are M2+/optional concerns, but dual-channel's COB-on-replica design and RDMA's zero-copy framing are worth studying.

Bottom line for IronCache: borrow Valkey's memory layout aggressively (open-addressing cache-line buckets, embedded keys, slot-partitioned tables, AVX SIMD for bitops/HLL), study its I/O-thread job-queue design as a proven RESP offload pattern, but treat its single-threaded execution core as the bar to BEAT — IronCache's "max throughput per core" thesis should evaluate sharded-by-key multi-threaded execution (Dragonfly/shared-nothing style) rather than inheriting Valkey's main-thread bottleneck. Valkey should also be wired in as a differential-fuzzing/conformance oracle for RESP2/RESP3 and as the head-to-head benchmark baseline.

## Mechanisms: borrow, adapt, or reject

| Mechanism | System | Stance | What it does | Rationale for IronCache |
| --- | --- | --- | --- | --- |
| Asynchronous I/O threading (full connection-lifecycle offload) | Valkey 8.0 | **adapt** | Offloads socket read, RESP parse, reply format, socket write, epoll poll, and memory free to I/O threads via per-thread SPSC atomic ring-buffer job queues, while command execution stays on the single main thread. ~1.2M QPS on r7g. | Borrow the proven job-queue/offload pattern and the batched-prefetch idea, but do NOT inherit the single-threaded execution core. IronCache's 'max throughput per core' goal argues for sharded-by-key parallel execution (shared-nothing) so adding cores scales command throughput, not just I/O. Use Valkey's model as the conservative baseline and the thing to beat. |
| Open-addressing cache-line hashtable (Swiss-tables-inspired) | Valkey 8.1 | **borrow** | Replaces chaining dict with 64-byte buckets of 7 entries: 1-byte presence bitmask, 7 one-byte secondary hashes for SIMD/branch-light probing, 7 entry pointers; bucket-chained on overflow; incremental rehash via two tables; soft max fill ~91.43%. Saves ~20-30 bytes/entry. | This is close to ideal for a memory-efficient cache: high cache locality, low per-entry overhead, SIMD-friendly probing. IronCache should start from this design (or a Rust hashbrown/SwissTable variant) rather than a chaining map. The stored-hash-byte trick and presence bitmask are directly applicable. |
| Embedded key in dictionary entry | Valkey 8.0 | **borrow** | Stores the key's SDS bytes inline in the hash-table entry instead of a separate pointer + heap allocation, removing one pointer and one malloc header per key (~8 bytes) and improving locality. | Eliminating per-key indirection and allocator overhead is exactly the kind of memory win IronCache wants. In Rust this maps to a single allocation holding key bytes + value handle inline (inline small keys, e.g. <=23 bytes, like SSO). |
| Per-slot dictionaries / kvstore partitioning | Valkey 8.0 | **adapt** | Splits the single global dict into 16,384 per-cluster-slot dictionaries, eliminating two slot-tracking linked-list pointers per entry (-16 bytes) and enabling per-slot stats and finer rehashing. | The memory win (no slot pointers) is real, but 16,384 fixed slots is a Redis-cluster artifact. IronCache should partition the keyspace for its OWN purposes — per-execution-shard tables (to enable parallel command execution) — and let those shards double as slot/migration units, rather than hardcoding 16,384. |
| Batched dictionary prefetch for pipelined commands | Valkey 8.0/8.1 | **borrow** | When I/O threads have parsed N commands ahead, the engine software-prefetches the hash-table entries for the next batch (prefetch-batch-max-size default 16) before executing, hiding memory-access latency; reported ~3.5x faster key ops / ~10% pipeline throughput. | Memory-latency hiding via lookahead prefetch is high-value for a cache and orthogonal to the threading decision. IronCache should design its command pipeline to expose a lookahead window and issue __builtin_prefetch-equivalents on the bucket addresses. |
| Dual-channel replication | Valkey 8.0 | **adapt** | During full sync, streams the RDB snapshot and the live replica backlog over two parallel connections, with the replica owning Client-Output-Buffer accounting; cuts sync time up to ~50% and reduces primary memory pressure. Default off. | Replication is M2+ for a single-node-first project, but the COB-on-replica idea (don't buffer the backlog on the busy primary) is a clean design and worth adopting when IronCache grows to multi-node. The two-connection complexity should be opt-in, mirroring Valkey defaulting it off. |
| Valkey Over RDMA transport (experimental) | Valkey 8.0 | **reject** | A Linux-only connection-module transport over RDMA verbs (librdmacm/libibverbs) claiming up to +275% throughput; incompatible with replication; explicitly may be removed in any release. | RDMA conflicts with IronCache's single-static-binary, easy-setup, broad-portability ethos (requires RDMA-capable NICs + kernel libs, Linux-only, still experimental upstream). Note it as a far-future optional feature gate, not a core direction. |
| AVX2/AVX-512 SIMD for bitops and HyperLogLog | Valkey 8.1/9.0 | **borrow** | Vectorizes PFCOUNT/PFMERGE (reported ~12x via AVX2) and BITCOUNT (up to +514%), with AVX-512 added in 9.0. | SIMD acceleration of bitmap/HLL/aggregation paths is a pure efficiency win aligned with IronCache's throughput-per-core goal; Rust gives us std::arch intrinsics and portable SIMD. Worth a dedicated design issue for the data-structure command paths. |
| Differential testing vs Valkey as RESP oracle | Valkey (all versions) | **borrow** | Valkey is the permissively-licensed reference implementation of the RESP2/RESP3 protocol and Redis command semantics. | Because it is BSD-3 (not relicensed Redis), IronCache can legally run Valkey as a side-by-side conformance/differential-fuzzing oracle and as the head-to-head benchmark baseline without SSPL contamination. This should be a standing part of the test harness. |

## Implications for IronCache

- Adopt an open-addressing, cache-line-bucketed hash table as IronCache's core index from day one (Rust hashbrown/SwissTable-style): 64-byte buckets, stored secondary-hash bytes for SIMD probing, presence bitmask, incremental two-table rehash. This directly captures Valkey 8.1's ~20-30 bytes/entry savings without inheriting C's dict baggage.
- Store small keys inline in the entry (SSO-style, e.g. keys <= ~23 bytes inline) to replicate Valkey's embedded-key win, eliminating a pointer indirection and an allocator header per key; benchmark the inline threshold against real key-size distributions.
- Make the threading decision explicit and contrarian: Valkey proves I/O offload alone hits a single-threaded execution ceiling. IronCache should prototype shared-nothing, key-sharded parallel command execution (one shard = one thread = one partition of the index) so 'throughput per core' scales with cores for the common single-key GET/SET path, falling back to Valkey-style I/O-only threading only if sharding proves too costly for multi-key/transaction semantics.
- Reuse Valkey's I/O-thread job-queue pattern (per-thread SPSC atomic ring buffer, main-thread fallback when full) as the connection-handling layer regardless of the execution model; it is a clean, lock-light design that maps well to Rust (crossbeam ArrayQueue / a hand-rolled SPSC).
- Build a lookahead command pipeline that prefetches index buckets for the next batch (Valkey default 16) using prefetch intrinsics — a memory-latency win orthogonal to threading and high-impact for pipelined workloads.
- Keep advanced replication/transport OFF by default and out of the core binary's hot path, exactly as Valkey defaults dual-channel-replication and io-threads off; favor a simple single-node story first (matches IronCache's easy-setup goal).
- Reject RDMA as a core transport (Linux-only, special NICs, still experimental upstream) — it violates the single-binary/portable/easy-setup thesis. Track it only as a far-future optional cargo feature flag.
- Vectorize bitops/HLL/aggregation command paths with portable SIMD (std::arch AVX2/AVX-512 with runtime feature detection) — Valkey's 12x PFCOUNT and +514% BITCOUNT show the upside and these are self-contained.
- Wire Valkey (BSD-3, license-safe) into CI as a RESP2/RESP3 differential-testing oracle and as the head-to-head benchmark baseline (target: beat Valkey 9.x QPS-per-core and bytes-per-key on identical hardware). Pin to specific Valkey tags for reproducibility.
- Track the live Valkey roadmap (9.0 atomic slot migration, cluster multi-DB, zero-copy responses, MPTCP) as the moving compatibility/perf target — IronCache's single-to-multi-node story should study atomic slot migration as the modern alternative to Redis's key-by-key cluster resharding.

## Research papers and primary sources

- **Swiss Tables / flat_hash_map design (Abseil)** (Google open-source / CppCon 2017). Open-addressing hash table with a control byte per slot (7-bit hash + presence) probed with SSE/SIMD to test a group of slots per instruction, yielding high load factors and cache locality. [source](https://abseil.io/about/design/swisstables) Relevance: Valkey 8.1's new hashtable explicitly credits Swiss-tables for its cache-line bucket design; this is the canonical reference IronCache should base its core index on (via Rust's hashbrown, which is a SwissTable port).
- **Scaling Memcache at Facebook** (NSDI 2013). Engineering a high-throughput distributed cache: lease mechanisms, request coalescing, regional pools, and handling thundering herds at scale. [source](https://www.usenix.org/conference/nsdi13/technical-sessions/presentation/nishtala) Relevance: Provides the systems context for what a cache must do beyond single-node speed; informs IronCache's eventual single-to-multi-node design and contrasts with Valkey/Redis's cluster model.
- **MICA: A Holistic Approach to Fast In-Memory Key-Value Storage** (NSDI 2014). Shared-nothing partitioning of the keyspace across cores with each core owning a partition, plus lossy/lossless index structures, to achieve near-linear multicore scaling for KV operations. [source](https://www.usenix.org/conference/nsdi14/technical-sessions/presentation/lim) Relevance: Directly supports IronCache's contrarian thesis: partition-per-core execution beats Valkey's single-threaded-execution + I/O-threads model for throughput-per-core.
- **Memshare / log-structured memory & memory-efficient KV (and RAMCloud log-structured memory)** (USENIX ATC 2017 / SOSP 2011). Log-structured memory management and adaptive memory partitioning to minimize fragmentation and maximize hit rate per byte in in-memory caches. [source](https://www.usenix.org/conference/atc17/technical-sessions/presentation/cidon) Relevance: Valkey leans on jemalloc + active defrag; IronCache's 'minimal memory' goal should evaluate log-structured/slab approaches as an alternative to allocator-dependent layouts.
- **Dragonfly: a modern replacement for Redis and Memcached (VLL / shared-nothing threading)** (Dragonfly engineering (2022-)). A shared-nothing, thread-per-core architecture (built on a userspace fiber/io_uring runtime) that shards the keyspace so command execution itself scales across cores, unlike Redis/Valkey's single execution thread. [source](https://www.dragonflydb.io/blog/why-threading-models-matter-dragonfly-vs-valkey) Relevance: The concrete counter-design to Valkey's threading model and the architecture IronCache should benchmark its own multicore-execution prototype against.

## Open questions

- Can IronCache's RESP/transaction/MULTI-EXEC and Lua/scripting semantics be preserved under a shared-nothing key-sharded execution model, or do multi-key/cross-slot commands force a Valkey-style single-execution-thread fallback for those paths?
- What is the real bytes-per-entry IronCache can achieve in Rust vs Valkey 8.1's hashtable once you account for the value handle, TTL, LRU/LFU metadata, and Rust's allocator — is the ~20-30 byte/entry parity beatable?
- Does io_uring (Linux) give IronCache a materially better I/O path than Valkey's epoll + I/O-thread job queues, and can it be made portable/optional without breaking the single-binary story on macOS/BSD?
- Should IronCache target Valkey 9.x's atomic slot migration semantics for its eventual cluster mode, or design a different resharding primitive better suited to a from-scratch engine?
- How much of Valkey's claimed throughput is TLS-dominated (the offload helps TLS most) — what is the apples-to-apples plaintext QPS-per-core baseline IronCache must beat?
- What is the right keyspace-partition count for IronCache (Valkey's 16,384 is a cluster artifact) given that the same partitions must serve as both execution shards and migration units?

## Proposed issues (seeds for the tracker)

- **[decision, M1]** Decide IronCache threading model: shared-nothing key-sharded execution vs Valkey-style I/O-only offload. Choose whether command execution is parallelized by sharding the keyspace per core (Dragonfly/MICA style) or kept single-threaded with I/O offload (Valkey style). This is the load-bearing decision for the 'max throughput per core' goal; document the multi-key/transaction tradeoffs.
- **[design, M2]** Design IronCache core index as an open-addressing cache-line hashtable (SwissTable/hashbrown). Spec a 64-byte-bucket open-addressing table with stored secondary-hash bytes, presence bitmask, SIMD group probing, and incremental two-table rehash, mirroring Valkey 8.1's hashtable to capture its ~20-30 bytes/entry savings.
- **[design, M2]** Embed small keys inline in index entries (SSO-style) to eliminate per-key indirection. Replicate Valkey 8.0's embedded-key win: store keys below an inline threshold directly in the entry to remove a pointer and allocator header per key; determine the threshold from key-size distributions.
- **[task, M0]** Benchmark harness: Valkey 9.x as RESP differential-test oracle and head-to-head baseline. Stand up a CI harness that runs IronCache against pinned Valkey tags for RESP2/RESP3 conformance (differential fuzzing) and side-by-side QPS-per-core and bytes-per-key benchmarks on identical hardware.
- **[research, M1]** Research: lookahead command pipeline with bucket prefetching. Evaluate Valkey's batched index prefetch (default lookahead 16) in a Rust context: expose a pipeline lookahead window and prefetch index buckets for the next batch to hide memory latency; quantify the win.
- **[non-goal, M0]** Non-goal: RDMA as a core transport. Declare RDMA out of scope for the core single binary (Linux-only, special NICs, experimental upstream), preserving the single-static-binary/easy-setup thesis; revisit only as an optional far-future feature flag.
- **[design, M2]** Design: connection I/O layer using per-core SPSC job queues (Valkey io_threads pattern) and/or io_uring. Spec the socket-read/parse/write offload layer: adopt Valkey's lock-light per-thread SPSC ring-buffer job queue with main-thread fallback, and evaluate io_uring as an optional faster path without breaking portability.
- **[research, M1]** Research: SIMD acceleration for bitops, HyperLogLog, and aggregation command paths. Scope portable-SIMD (std::arch AVX2/AVX-512 + runtime feature detection) implementations of BITCOUNT/PFCOUNT/PFMERGE-equivalents, targeting Valkey's reported 12x HLL and +514% BITCOUNT gains.
- **[decision, M1]** Decide keyspace partition count and dual-purpose shard/migration unit. Pick IronCache's partition count (Valkey's 16,384 is a Redis-cluster artifact) so partitions serve as both per-core execution shards and future migration units; study Valkey 9.0 atomic slot migration as the resharding model.
- **[meta, M0]** Meta: pin Valkey as the canonical RESP-compatibility reference (not relicensed Redis). Record the decision to treat BSD-3 Valkey as IronCache's license-safe compatibility reference and oracle, avoiding any SSPL/RSAL-licensed Redis code or test artifacts; pin specific Valkey versions in docs.
