## Summary

The Redis-lineage ecosystem ships multiple separate binaries (`redis-server`, `redis-cli`, `redis-benchmark`, `redis-check-rdb`, `redis-check-aof`, `redis-sentinel`), where the mode-switching ones (`redis-sentinel`, `redis-check-rdb`, `redis-check-aof`) are symlinks to `redis-server` that branch on argv[0]. KeyDB mirrors this exactly (keydb-server/keydb-cli, plus redis-cli/redis.conf symlinks for drop-in compat). Valkey is the same fork shape but cleaner to install (`brew install valkey`, a plain formula, vs Redis now requiring a tap + cask on macOS, and Redis being a conflicting formula). Dragonfly is the modern outlier and the right model to emulate: a single self-contained binary you download, rename to `dragonfly`, and run with gflags-style `--flag=value` options; on Mac/Windows it expects Docker. Config across all of them is a flat keyword-arg file (redis.conf style) with command-line overrides; Redis runs with no config file at all by default (daemonize no, dev-friendly defaults). This gives IronCache a clear easiest-setup target: one static binary that is BOTH server and CLI via clap subcommands (server | cli | bench | check | upgrade), runnable with zero config and sane defaults.

Live config in Redis is mature and is the parity bar: `CONFIG SET` (multi-param since 7.0), `CONFIG GET *` to enumerate, `CONFIG REWRITE` to persist back to the file. Notably Redis does NOT reload config on SIGHUP — runtime changes flow only through CONFIG SET/REWRITE. IronCache should ship CONFIG GET/SET/REWRITE parity for wire compatibility but ADD an explicit reload path (SIGHUP and/or a `CONFIG RELOAD`) so a TOML edit can be applied without a client, which Redis users repeatedly want and don't have.

Observability is where IronCache can leapfrog. Redis/Valkey expose only the RESP `INFO` command (sections: server, clients, memory, persistence, threads, stats, replication, cpu, commandstats, latencystats, cluster, keyspace, keysizes, errorstats, etc.; `default` is the implicit arg), SLOWLOG (default slowlog-log-slower-than=10000us/10ms, slowlog-max-len=128), and LATENCY monitoring (latency-monitor-threshold default 0 = OFF; LATENCY LATEST/HISTORY/RESET/DOCTOR). Crucially, NEITHER Redis nor Valkey has a built-in Prometheus endpoint — operators must run the third-party oliver006/redis_exporter on :9121 that scrapes INFO. Dragonfly fixed this: it serves Prometheus-format metrics natively at `:6379/metrics` (same port as RESP, HTTP/RESP multiplexed) and a K8s admin port 9999. IronCache should follow Dragonfly: ship INFO + SLOWLOG + LATENCY for wire-parity AND a native `/metrics` endpoint so there is no sidecar, with latency monitoring on-by-default (Redis ships it off, so users fly blind).

For build/distribution, Rust gives IronCache an inherent advantage. The `x86_64-unknown-linux-musl` target sets `crt-static` by default and produces a fully static binary with no glibc/runtime dependency, runnable on any Linux of the same arch (this is the single-binary thesis). The practical cross-build toolchain is cargo-zigbuild (v0.22.3, 2026-04-26) which uses Zig as the linker/C-toolchain: it can target musl for x86_64+aarch64, pin a minimum glibc per target (e.g. `--target aarch64-unknown-linux-gnu.2.17`) for gnu builds that still run on old distros, and build macOS `universal2`. `cross` is the alternative (Docker-image-per-target). For packaging, distroless `static-debian13` (~2 MiB, includes CA certs + nonroot user) or `scratch` is the right container base for a static binary — sub-2 MiB images vs ~124 MiB debian.

Self-update is a solved problem in Rust: the `self_update` crate (v0.44.0) pulls release artifacts from GitHub/GitLab/S3/GCS, compares semver, and replaces the running executable via the `self-replace` crate, which writes the new file next to the current one and does an atomic rename (POSIX rename(2); Windows rename-then-can't-unlink dance). Neither provides automatic ROLLBACK — that is the gap IronCache must design (keep the previous binary as `ironcache.old`, verify the new binary's `--version`/healthcheck post-swap, roll back on failure). For supply chain, `cargo-auditable` (v0.7.5, 2026-05-21) embeds a timestamp-free, sorted dependency manifest into the binary (reproducible-build safe), and `cargo-cyclonedx` / the auditable2cdx bridge emit CycloneDX SBOMs; Cargo.lock pins the reproducible-build base. cargo-dist / `dist` (axodotdev, ~v0.30–0.31, actively maintained) automates the whole release: cross-built archives + checksums + a one-line `curl … | sh` installer that manages PATH and embeds checksums. The easiest-setup target for IronCache: a single reproducible static musl binary per arch, a `curl … | sh` installer plus `brew install ironcache` (plain formula, no tap, no cask, no conflict — beating Redis), a distroless image, a systemd unit, and `ironcache upgrade` with atomic swap + verified rollback.

## Mechanisms: borrow, adapt, or reject

| Mechanism | System | Stance | What it does | Rationale for IronCache |
| --- | --- | --- | --- | --- |
| Single static musl binary (crt-static default) | Rust toolchain + Dragonfly model | **borrow** | x86_64-unknown-linux-musl (and aarch64) produce a fully static binary with zero glibc/runtime dependency; Dragonfly proves the download-rename-run UX in this space. | This is IronCache's core thesis (single static binary, easy setup). Rust gives it for free on musl targets; nothing simpler exists for the operator. |
| Binary-is-both-server-and-CLI via argv[0] symlinks | Redis / KeyDB | **adapt** | One compiled binary behaves as server, sentinel, or checker depending on the name it's invoked as (ln -sf to redis-server). | Keep the single-binary economy, but use explicit clap SUBCOMMANDS (ironcache server \| cli \| bench \| check \| upgrade \| config) instead of argv[0] magic — clearer UX, --help discoverability, and avoids fragile symlink installs. Still ship `ironcache-cli`-style symlink/alias for redis-cli muscle memory. |
| redis-cli RESP client interop | Redis / Valkey / KeyDB | **borrow** | A small RESP client (redis-cli) and redis.conf symlinks ship alongside so existing tooling works unchanged. | Wire compatibility is a hard requirement; existing redis-cli and clients must Just Work against IronCache. Provide our own `ironcache cli` but never break stock redis-cli. |
| CONFIG SET / GET / REWRITE live config | Redis | **adapt** | Mutate config at runtime (multi-param since 7.0), enumerate with GET *, persist back to file with REWRITE; no SIGHUP reload. | Implement CONFIG GET/SET/REWRITE for wire parity, but ADD a real reload path (SIGHUP + CONFIG RELOAD) so a TOML file edit can be applied without a client — closing a long-standing Redis gap. |
| INFO sectioned introspection | Redis / Valkey | **borrow** | RESP INFO returns server/clients/memory/persistence/stats/cpu/commandstats/latencystats/keyspace etc.; 'default' subset when no arg. | INFO is the universal Redis observability contract every dashboard/exporter parses; IronCache must emit byte-compatible sections to drop into existing redis_exporter + Grafana setups. |
| Native Prometheus /metrics on the data port | Dragonfly | **borrow** | Serves Prometheus text-format metrics at :6379/metrics (HTTP multiplexed on RESP) with no external exporter; admin port 9999 in K8s. | Eliminates the redis_exporter sidecar that Redis/Valkey force on operators. A self-contained binary with self-contained metrics matches the easiest-setup goal exactly. |
| SLOWLOG + LATENCY monitoring | Redis | **adapt** | SLOWLOG (default 10ms/128 entries) records slow commands; LATENCY LATEST/HISTORY/RESET/DOCTOR tracks latency spikes (threshold default 0 = off). | Ship SLOWLOG + LATENCY for parity, but enable latency monitoring with a sane non-zero default and surface percentiles in /metrics — Redis ships it OFF, leaving users blind by default. |
| cargo-zigbuild cross-compilation | rust-cross/cargo-zigbuild | **borrow** | Uses Zig as linker to cross-build musl + gnu for x86_64/aarch64 from one host, pin minimum glibc per target, and build macOS universal2. | Simplest reliable matrix build for x86_64 + aarch64 static binaries from CI without a Docker-per-target setup; glibc-pinning gives a portable gnu fallback if musl ever bites. |
| self_update + self-replace (atomic swap) | self_update / self-replace crates | **adapt** | Pull a release artifact (GitHub/S3/GCS), semver-compare, and atomically rename the new binary over the running one. | Adopt for `ironcache upgrade`, but the crates have NO rollback — IronCache must add: keep previous binary as ironcache.old, post-swap health/version check, automatic rollback on failure, and signature verification. |
| cargo-auditable + cargo-cyclonedx (SBOM/supply-chain) | rust-secure-code / CycloneDX | **borrow** | Embeds a timestamp-free, sorted dependency manifest in the binary (reproducible-build safe) and emits CycloneDX SBOMs. | Reproducible builds + embedded SBOM are table stakes for a serious infra binary in 2026 and cost almost nothing to wire into CI; Cargo.lock already pins inputs. |
| cargo-dist curl\|sh installer | axodotdev cargo-dist / dist | **adapt** | Generates cross-built archives + checksums and a one-line curl\|sh installer that validates checksums and manages PATH. | Use the pattern (one-line installer + GitHub Releases) but treat the curl\|sh script as one of several install paths; pin/own the installer rather than depend on a third party for a security-sensitive bootstrap. |
| Distroless/scratch container image | GoogleContainerTools/distroless | **borrow** | Minimal base (static-debian13 ~2 MiB with CA certs + nonroot) for shipping a static binary as a tiny, low-CVE-surface image. | A static Rust binary on distroless static (or scratch) yields a sub-5 MiB image with almost no OS attack surface — directly serves minimal-footprint and easy-deploy goals. |
| gflags --flagfile style config | Dragonfly | **reject** | All config is --flag=value, optionally loaded from a --flagfile with one flag per line. | gflags is an Abseil-ism with no typing, no comments structure, and no nesting. IronCache should use TOML (idiomatic Rust, typed, commentable, nestable) as the file format, with flags overriding TOML overriding defaults. |
| Redis runs with no config file by default | Redis | **borrow** | Starts with built-in dev defaults and daemonize no, no config file needed. | Zero-config first-run is the easiest-setup ideal; IronCache should boot on `ironcache server` with safe defaults and require config only to deviate. |

## Implications for IronCache

- Ship ONE static musl binary per arch (x86_64 + aarch64) that is server + CLI + bench + check + upgrade via clap subcommands; default `ironcache server` boots with zero config and safe defaults (bind localhost, protected mode, sane maxmemory policy).
- Provide redis-cli wire interop: stock redis-cli and existing clients must connect unmodified; ship `ironcache cli` as the native client and optionally a redis-cli-named alias for muscle memory.
- Config precedence: built-in defaults < TOML file < CLI flags < runtime CONFIG SET. Implement CONFIG GET/SET/REWRITE for Redis parity but ADD SIGHUP + `CONFIG RELOAD` to apply TOML edits without a client (the gap Redis never closed).
- Native Prometheus /metrics served on the data port (like Dragonfly's :6379/metrics) plus an optional separate admin port, so there is NO redis_exporter sidecar; also emit Redis-compatible INFO sections so existing redis_exporter + Grafana dashboards still work as a fallback.
- Observability defaults flipped to ON: ship SLOWLOG (10ms/128 parity defaults) and LATENCY monitoring enabled with a sensible non-zero threshold, and surface command latency percentiles directly in /metrics — don't make users discover they were flying blind.
- Build pipeline: cargo-zigbuild in CI for the x86_64/aarch64 musl matrix (with a glibc-pinned gnu fallback target), reproducible builds via pinned Cargo.lock, cargo-auditable embedding the SBOM, and cargo-cyclonedx emitting a CycloneDX artifact attached to each release.
- `ironcache upgrade`: use self_update/self-replace for the atomic rename swap but BUILD the missing rollback — verify minisign/cosign signature of the artifact, keep the prior binary as ironcache.old, run a post-swap `--version`+health probe, auto-rollback on failure, and never auto-update a running server without an explicit flag.
- Distribution surface aiming to beat Redis on ease: (1) one-line `curl … | sh` installer (own/pin it), (2) `brew install ironcache` as a PLAIN formula — no tap, no cask, no conflicting-formula, which is strictly easier than today's Redis on macOS, (3) distroless-static / scratch Docker image under 5 MiB, (4) a hardened systemd unit (NoNewPrivileges, ProtectSystem, dynamic user).
- Adopt TOML (not gflags --flagfile) as the config file format for typed, commentable, nestable config that round-trips cleanly with CONFIG GET/SET/REWRITE.

## Research papers and primary sources

- **Reproducible Builds: Increasing the Integrity of Software Supply Chains** (IEEE Software, 2021). Bit-for-bit reproducible builds let independent parties verify that a binary corresponds to its source, defending against build-time tampering. [source](https://arxiv.org/abs/2104.06020) Relevance: Directly motivates IronCache's reproducible static-binary + embedded-SBOM strategy (Cargo.lock + cargo-auditable) so the single binary users download is verifiable.
- **in-toto: Providing farm-to-table guarantees for bits and bytes** (USENIX Security 2019). A framework that cryptographically attests every step of a software supply chain from source to packaged artifact. [source](https://www.usenix.org/conference/usenixsecurity19/presentation/torres-arias) Relevance: Informs how IronCache should sign and attest its release artifacts and the curl|sh installer, beyond just a checksum.
- **The Update Framework (TUF): Survivable Key Compromise in Software Update Systems** (ACM CCS 2010). A signing/role/threshold model for software update systems that remains secure even when individual keys are compromised. [source](https://theupdateframework.io/papers/survivable-key-compromise-ccs2010.pdf) Relevance: Design reference for a hardened `ironcache upgrade` self-update channel (signature verification, key rotation, rollback protection) since self_update/self-replace provide none of this.

## Open questions

- Can a single binary HTTP-multiplex Prometheus /metrics on the RESP port (Dragonfly does) without hurting hot-path throughput, or should /metrics live on a dedicated admin port by default?
- TOML vs redis.conf-flat-format for the on-disk config: how do we round-trip CONFIG REWRITE into TOML while preserving comments and ordering?
- What is the rollback contract for `ironcache upgrade` — keep N previous binaries? where (alongside binary vs a state dir)? what health probe constitutes 'new binary is good'?
- musl allocator penalty: musl's malloc is notably slower under multithread contention; do we statically link a better allocator (mimalloc/jemalloc) into the musl build, or ship a glibc-pinned gnu build as the performance default and musl as the portable default?
- Signing: minisign vs sigstore/cosign for release artifacts and the self-update channel — which gives the best verify-on-a-fresh-box UX with no extra tooling?
- How do we keep INFO byte-compatible enough for stock redis_exporter while also exposing IronCache-specific fields without breaking parsers?
- Windows story: Redis/Dragonfly punt to WSL/Docker on Windows — does IronCache ship a native Windows binary (self-replace supports it) or also punt to containers for M2?

## Proposed issues (seeds for the tracker)

- **[epic, M0]** EPIC: Single static binary that is both server and CLI. Track the operability thesis: one reproducible static musl binary per arch exposing server/cli/bench/check/upgrade subcommands, zero-config first run, and easiest-in-class install. Umbrella for the issues below.
- **[decision, M1]** DECISION: clap subcommands vs argv[0] symlink mode-switching. Decide between Redis-style argv[0] symlink dispatch and explicit clap subcommands (ironcache server|cli|bench|check|upgrade|config). Recommendation: clap subcommands + an optional redis-cli-named alias for interop.
- **[design, M1]** DESIGN: Native Prometheus /metrics endpoint (no exporter sidecar). Specify a built-in Prometheus text-format endpoint à la Dragonfly (:port/metrics on the data port and/or an admin port), the metric set, and how it coexists with Redis-compatible INFO so existing redis_exporter dashboards still work.
- **[design, M2]** DESIGN: INFO / SLOWLOG / LATENCY wire-parity with sane defaults. Define byte-compatible INFO sections, SLOWLOG (10ms/128 parity), and LATENCY commands, but flip latency monitoring ON by default (Redis defaults it to 0/off) and feed percentiles into /metrics.
- **[design, M2]** DESIGN: TOML config with CONFIG GET/SET/REWRITE parity + live reload. Specify defaults<TOML<flags<CONFIG SET precedence, CONFIG GET/SET/REWRITE wire parity, REWRITE round-tripping into TOML, and a SIGHUP/CONFIG RELOAD path that Redis lacks. Reject gflags --flagfile.
- **[design, M2]** DESIGN: ironcache upgrade — atomic swap with verified rollback. Build on self_update/self-replace (atomic rename) but add the missing pieces: signature verification, keep ironcache.old, post-swap health/version probe, and automatic rollback on failure; never auto-update a live server implicitly.
- **[task, M2]** TASK: CI cross-build matrix via cargo-zigbuild (musl x86_64+aarch64, glibc-pinned gnu fallback). Stand up cargo-zigbuild (v0.22.x) in CI to produce static musl binaries for x86_64 and aarch64 plus a glibc-version-pinned gnu fallback, with SHA256 checksums attached to releases.
- **[task, M2]** TASK: Reproducible builds + embedded SBOM (cargo-auditable, cargo-cyclonedx). Pin Cargo.lock, build reproducibly, embed a timestamp-free SBOM with cargo-auditable (v0.7.x), and publish a CycloneDX SBOM artifact per release.
- **[research, M1]** RESEARCH: musl allocator penalty vs glibc — which to ship as default. Benchmark musl's malloc vs statically-linked mimalloc/jemalloc under multithread contention to decide whether musl is the performance default or a portable fallback to a glibc-pinned build.
- **[decision, M1]** DECISION: Artifact + self-update signing scheme (minisign vs cosign/sigstore). Choose the signing/verification mechanism for release artifacts and the upgrade channel that gives the best verify-on-fresh-box UX with minimal extra tooling; reference TUF/in-toto for key compromise resilience.
- **[task, M2]** TASK: Packaging — curl|sh installer, plain Homebrew formula, distroless image, systemd unit. Ship a one-line curl|sh installer (checksum-validated, PATH-managing), a PLAIN brew formula (no tap/cask/conflict — beating Redis on macOS), a sub-5MiB distroless-static/scratch image, and a hardened systemd unit.
- **[non-goal, M0]** NON-GOAL: native Windows server binary for M2. Like Redis/Dragonfly, defer native Windows server support to Docker/WSL for the prototype; ship Linux (musl) + macOS first. Revisit post-M2.
