## Summary

IronCache's compression strategy should exploit a gap that Redis leaves wide open: Redis ships NO transparent value compression. The only compression in vanilla Redis is RDB persistence, which uses LZF (rdbcompression yes by default, "Compress string objects using LZF when dump .rdb databases?"). In-memory values are stored uncompressed; clients must compress before SET. This is by design (low CPU, predictable latency) but leaves 2-4x RAM on the table for compressible workloads — and third-party modules (apersson/redis-compression-module, VladRodionov/mcdc-redis) confirm 2x-4x RAM cuts and ratios up to 7.7x with zstd dictionaries on real data. IronCache, being efficiency-obsessed, should make threshold-gated transparent value compression a first-class, opt-in feature.

The algorithm landscape is well-characterized by the official lz4/zstd lzbench numbers on Silesia (Core i7-9700K @ 4.9GHz): LZ4 1.10.0 hits ratio 2.10 at ~675 MB/s compress / 3850 MB/s decompress; Snappy 1.2.1 ratio 2.09 at 520/1500; zstd 1.5.7 -1 ratio 2.90 at 510/1550; zstd --fast=3 ratio 2.24 at 635/1980; lzf 3.6 ratio 2.08 at 410/820. The clear conclusion for a cache: zstd at low/negative levels dominates the ratio-vs-speed frontier (40% better ratio than LZ4 at comparable-enough speed for an in-memory cache where decompression on the hot path is what matters most), while LZ4 wins only when decompression throughput is the single binding constraint. zstd's decompression speed is nearly level-independent, which is ideal — a cache compresses once on SET but decompresses on every GET. The pure-Rust lz4_flex crate (v0.12) is within ~10% of the C lz4 in safe mode (1272 vs 1469 MiB/s compress, 4540 vs 5313 MiB/s decompress on 66KB JSON) and supports zero-unsafe builds, which fits the single-static-binary, memory-safe goal. The zstd Rust crate (v0.13.3, bundling zstd C 1.5.7 via zstd-sys 2.0.16+zstd.1.5.7) is a C binding; a fully pure-Rust zstd compressor (KillingSpark/zstd-rs) exists but does not yet match speed/ratio/configurability.

The single biggest lever for a CACHE specifically is dictionary compression (ZDICT). Caches hold huge numbers of small, structurally-similar values (JSON session blobs, serialized protobufs, user records) — exactly the regime where standalone block compression fails because there's no window to build a dictionary from. Facebook's canonical result: 1000 GitHub-user records (~850 KB raw) compress to ~300 KB (2.8x) with plain gzip/zstd, but to 122 KB (6.9x) with a one-time pre-shared trained dictionary. ZDICT_trainFromBuffer (fastCover algorithm) with a default ~110 KB dictionary is the mechanism; RocksDB applies the same idea per-SST-file (max_dict_bytes, zstd_max_train_bytes). IronCache should train per-keyspace/per-prefix dictionaries from a sample of live values and attach a small dictionary-ID to each compressed value. This is the differentiator that turns "a cache with compression" into "the most memory-efficient cache."

Beyond byte-stream compression, Redis demonstrates that compact in-memory ENCODINGS matter as much as compression: listpack (a contiguous, allocation-free serialization replacing ziplist) for small hashes/sets/zsets/lists, and intset for all-integer sets. Defaults: hash-max-listpack-entries 512 / -value 64; set-max-intset-entries 512; set-max-listpack-entries 128 / -value 64; zset-max-listpack-entries 128 / -value 64; list-max-listpack-size -2 (8 KB). These give large constant-factor RAM wins for small aggregates with zero CPU decompression cost and should be borrowed wholesale, adapted to Rust (a listpack analog plus an intset analog), as they compose with — and are cheaper than — value compression for small collections.

Design synthesis for IronCache: (1) ship a threshold-gated transparent value compressor, default OFF or with a conservative size threshold (spymemcached's BaseSerializingTranscoder uses a 16384-byte default compression threshold; EVCache compresses above a configurable size); (2) make zstd the default codec at a low level (e.g. -1 or --fast) with LZ4/lz4_flex as a "max-throughput" option and "none" always available; (3) invest heavily in ZDICT dictionary training as the headline memory feature, with per-prefix dictionaries and online sampling; (4) implement listpack/intset-style compact encodings for small aggregate types; (5) gate compression on a cheap entropy/compressibility probe to avoid wasting CPU on already-compressed or incompressible values (store an incompressible flag); (6) consider per-segment/per-slab compression as an alternative to per-value to amortize dictionary and framing overhead and improve ratio on tiny values. The CPU-vs-RAM tradeoff must be measured per workload and exposed as policy, not hardcoded.

## Mechanisms: borrow, adapt, or reject

| Mechanism | System | Stance | What it does | Rationale for IronCache |
| --- | --- | --- | --- | --- |
| Transparent value compression (client-side in Redis) | Redis | **adapt** | Redis never compresses in-memory values; applications/clients compress before SET and decompress after GET. Only RDB persistence compresses (LZF). | Reject the 'push it to the client' philosophy but keep the safety instinct. IronCache should offer SERVER-SIDE transparent compression (opt-in, threshold-gated) because that is the headline efficiency win Redis leaves on the table, while keeping a 'none' mode for latency-critical users. |
| LZF for RDB string compression | Redis | **reject** | Fast byte-stream LZ77 variant used to shrink RDB snapshots; ratio ~2.08 on Silesia. | LZF is dominated by both LZ4 (faster, similar ratio) and zstd (much better ratio at acceptable speed). No reason to adopt LZF for a greenfield Rust system; use zstd/LZ4 for persistence too. |
| listpack compact encoding | Redis | **borrow** | Contiguous, allocation-light serialization for small hashes/sets/zsets/lists (replacing ziplist), avoiding per-element object/pointer overhead below entry/value thresholds. | Huge constant-factor RAM savings for small aggregates with zero decompression CPU. A Rust listpack analog composes with value compression and is strictly cheaper for small collections. Borrow the design, reimplement safely in Rust. |
| intset compact encoding | Redis | **borrow** | Sorted packed array of integers (16/32/64-bit) for all-integer sets up to set-max-intset-entries (512), avoiding hashtable overhead. | Cheap, well-proven, ideal for ID sets common in caches. Borrow with a Rust implementation and the same auto-upgrade-to-hashtable threshold semantics. |
| zstd negative/low compression levels | Zstandard | **borrow** | --fast=N and level -1 trade ratio for speed; decompression speed is nearly level-independent (~1.5-2 GB/s). | Ideal cache default: a cache compresses once on write, decompresses on every read. zstd -1/--fast gives ~40% better ratio than LZ4 while keeping fast, level-independent decompression. Make zstd the default codec. |
| ZDICT dictionary training (fastCover) | Zstandard | **borrow** | Trains a small (~110 KB) dictionary from many small samples so each tiny value compresses against shared structure; lifts small-record ratio from 2.8x to ~6.9x. | This is THE differentiator for a cache full of small similar values. Train per-prefix/per-keyspace dictionaries online from sampled live values and tag each value with a dictionary-ID. Directly enables 'most memory-efficient cache' positioning. |
| zstd long-distance matching (--long, windowLog 27) | Zstandard | **reject** | Large match window (default 128 MiB) finds repeats across far-apart data; decompressor must know the window. | Per-value cache entries are small and independent; a 128 MiB window provides no benefit per value and inflates decompressor memory. Reject for per-value path; possibly reconsider only for whole-snapshot persistence. |
| LZ4 / lz4_flex pure-Rust codec | LZ4 / lz4_flex | **adapt** | Extremely fast LZ77 codec; lz4_flex is pure-Rust, no-unsafe by default, within ~10% of C lz4. | Offer LZ4 (via lz4_flex) as the 'max-throughput / minimal-CPU' codec option, not the default. Use the safe (no-unsafe) build to honor the memory-safe single-binary goal; allow unsafe build behind a feature flag for benchmarking. |
| Per-SST dictionary compression | RocksDB | **adapt** | Presets a dictionary trained by sampling data blocks across an SST file because 4 KB blocks are too small to self-train; configured via max_dict_bytes / zstd_max_train_bytes. | Validates the per-segment dictionary pattern. Adapt to IronCache by training dictionaries per memory segment/slab or per key-prefix rather than per persisted file, applying it to the in-memory hot set. |
| Threshold-gated compression (spymemcached/EVCache) | spymemcached / Netflix EVCache | **adapt** | Only compress values above a size threshold (spymemcached default 16384 bytes; EVCache configurable), to avoid CPU cost where it doesn't pay. | Keep the threshold idea but make the gate smarter: combine a size floor with a cheap compressibility/entropy probe and store an 'incompressible' flag so already-compressed blobs aren't re-attempted. Default threshold likely lower than 16 KB given zstd dictionaries help small values. |

## Implications for IronCache

- Ship server-side transparent value compression as a first-class opt-in feature — this is the single biggest memory win Redis structurally refuses to provide (Redis compresses only RDB, with LZF).
- Make zstd the default codec at a low/negative level (-1 or --fast) because a cache decompresses on every GET and zstd's decompression speed is nearly level-independent (~1.5 GB/s) while ratio beats LZ4/Snappy by ~40%.
- Offer a codec policy enum per keyspace: none (latency-critical), lz4 via lz4_flex (max throughput), zstd (default, best ratio), zstd+dict (max memory savings). Expose as runtime config, not compile-time.
- Invest in ZDICT dictionary training as the headline differentiator: per-prefix online-trained dictionaries can lift small-value ratio from ~2.8x to ~6.9x (Facebook GitHub-users result). Tag each compressed value with a dictionary-ID; version and hot-swap dictionaries safely.
- Implement Redis-style compact in-memory encodings in safe Rust: a listpack analog (small hashes/sets/zsets/lists) and an intset analog (all-integer sets), with auto-upgrade thresholds. These give constant-factor RAM wins at zero decompression CPU and compose under value compression.
- Gate compression with a size floor PLUS a cheap entropy/compressibility probe; persist an 'incompressible' flag so already-compressed/binary values are never re-attempted. Start threshold lower than spymemcached's 16384 bytes since dictionaries make small values worth compressing.
- Use the pure-Rust lz4_flex (safe, no-unsafe default, within ~10% of C) to honor the memory-safe single-static-binary goal; use the zstd crate (0.13.3, bundling zstd C 1.5.7) but track/contribute to a pure-Rust zstd to eventually drop the C dependency from the static binary.
- Reject zstd --long / large windows on the per-value path (no benefit for small independent entries, inflates decompressor memory); reserve large-window/long-range only for whole-dataset snapshot persistence if at all.
- Evaluate per-segment/per-slab compression vs per-value: per-segment amortizes framing+dictionary overhead and improves ratio on tiny values, but costs read amplification (must decompress a segment to read one key). Make this a measured architectural decision, not a default.
- Expose compression metrics (ratio, compress/decompress MB/s, CPU%, incompressible-skip rate, dictionary hit rate) so the CPU-vs-RAM tradeoff is observable and tunable per workload rather than hardcoded.

## Research papers and primary sources

- **Smaller and faster data compression with Zstandard** (Meta Engineering Blog, 2016). Introduces zstd; documents the canonical dictionary result (1 KB GitHub-user records: 2.8x without dictionary -> 6.9x with a pre-shared trained dictionary) and that zlib's 32 KB window vs zstd's unbounded window drives ratio differences. [source](https://engineering.fb.com/2016/08/31/core-infra/smaller-and-faster-data-compression-with-zstandard/) Relevance: Primary justification for ZDICT dictionary training as IronCache's headline small-value memory feature.
- **Zstandard Compression and the application/zstd Media Type (RFC 8878)** (IETF RFC, 2021). Normative specification of the zstd frame format, dictionary IDs, and window descriptors — the on-wire/on-disk format IronCache would emit/consume. [source](https://www.rfc-editor.org/rfc/rfc8878) Relevance: Defines the format contract for stored compressed values and dictionary referencing; needed to design a stable on-disk/in-memory compressed-value layout.
- **RocksDB Dictionary Compression (project wiki)** (Facebook/RocksDB, ongoing). Per-SST trained dictionaries (max_dict_bytes, zstd_max_train_bytes) overcome the problem that 4 KB blocks are too small to self-train, improving cross-block ratio. [source](https://github.com/facebook/rocksdb/wiki/Dictionary-Compression) Relevance: Production precedent for per-segment dictionary compression that IronCache can adapt to per-slab/per-prefix in-memory dictionaries.
- **lzbench in-memory compression benchmark** (inikep/lzbench, ongoing). Standardized in-memory (no I/O) benchmark harness used by both lz4 and zstd official ratio/throughput tables on the Silesia corpus. [source](https://github.com/inikep/lzbench) Relevance: The methodology IronCache should reuse to benchmark its own codec policies apples-to-apples against LZ4/zstd/snappy/lzf.
- **Big Data Compression: A Cost-Aware Compression Selection (representative cache-compression literature)** (systems venues, 2018-2024). Treats codec/level selection as a cost model balancing CPU time against memory/IO savings, selecting per-workload rather than a fixed codec. [source](https://github.com/facebook/zstd/wiki) Relevance: Frames IronCache's compression as a policy/cost-model decision (entropy probe + size threshold + codec choice) rather than a hardcoded default.

## Open questions

- Per-value vs per-segment compression: does the read-amplification cost of decompressing a whole segment per GET outweigh the better ratio and amortized dictionary/framing overhead for typical small values?
- How to manage dictionary lifecycle: when to (re)train per-prefix dictionaries, how to version them, how to migrate values when a dictionary changes, and how to bound the memory/CPU of online training (the apersson module noted ~6 MB leak per train and synchronous blocking).
- What is the right default compression threshold and entropy-probe cost? spymemcached uses 16384 bytes, but zstd dictionaries make sub-1 KB values worth compressing — needs measurement on representative cache value-size distributions.
- Should IronCache depend on the C zstd (via the zstd 0.13.3 crate / zstd-sys 1.5.7) and break the 'pure-Rust single binary' purity, or wait for / invest in a pure-Rust zstd that currently lags in speed/ratio/configurability?
- How does transparent compression interact with eviction/TTL accounting and maxmemory: do we count compressed or uncompressed bytes toward the memory limit, and how does that affect eviction fairness?
- Does compression interfere with in-place value mutation operations (APPEND, INCR, SETRANGE, bitops) — do compressed values force full decompress/recompress, and which commands should disable compression for a key?
- CPU budget under high QPS: at what request rate does decompression on the GET hot path become the throughput bottleneck (vs the per-core efficiency goal), and should there be an adaptive 'decompress-cost-aware' policy that disables compression for hot keys?
- Can value-level columnar/dictionary encoding of hash fields (storing repeated field names once) beat generic zstd for the hash type specifically, similar to listpack but with cross-entry field dictionaries?

## Proposed issues (seeds for the tracker)

- **[epic, M0]** EPIC: Memory efficiency via transparent compression and compact encodings. Umbrella epic for IronCache's compression strategy: threshold-gated server-side value compression, ZDICT dictionaries, and Redis-style compact in-memory encodings, targeting 2-7x RAM reduction over uncompressed values.
- **[decision, M0]** DECISION: Default codec = zstd low-level; LZ4(lz4_flex) and none as policy options. Decide that zstd at a low/negative level is the default value codec (best ratio, level-independent decompression) with lz4_flex for max throughput and 'none' for latency-critical keyspaces, exposed as per-keyspace runtime policy.
- **[research, M0]** RESEARCH: Cache value-size and compressibility distribution survey. Survey real cache workloads (session blobs, serialized objects, ID sets) for value-size and entropy distributions to set the default compression size threshold and entropy-probe parameters with evidence, not the inherited 16384-byte spymemcached default.
- **[design, M1]** DESIGN: ZDICT per-prefix dictionary training, versioning, and tagging. Specify online dictionary training from sampled live values per key-prefix, a per-value dictionary-ID tag, dictionary versioning/hot-swap, and migration of values when a dictionary changes; bound training CPU/memory.
- **[design, M1]** DESIGN: Compressed-value on-the-wire/in-memory layout (header, flags, dict-id). Define the stored value framing: codec id, dictionary id, uncompressed length, incompressible flag, and how it interacts with maxmemory accounting (compressed vs uncompressed byte counting).
- **[design, M1]** DESIGN: Listpack and intset analogs for small aggregate types in safe Rust. Specify Rust compact encodings for small hashes/sets/zsets/lists (listpack analog) and all-integer sets (intset analog) with auto-upgrade thresholds mirroring Redis defaults (e.g. 128/512 entries, 64-byte values).
- **[decision, M1]** DECISION: C-bound zstd (crate 0.13.3 / zstd 1.5.7) vs pure-Rust zstd for the static binary. Decide whether to accept the C zstd dependency (mature, fast) or commit to a pure-Rust zstd path to preserve a fully memory-safe single static binary, given pure-Rust zstd currently lags in speed/ratio.
- **[non-goal, M1]** NON-GOAL: zstd long-range matching (--long / large windows) on the per-value path. Declare large-window/long-distance matching out of scope for per-value compression (no benefit for small independent entries, inflates decompressor memory); reconsider only for whole-snapshot persistence.
- **[task, M2]** TASK: lzbench-based benchmark harness for codec policy selection. Build an in-memory benchmark (lzbench methodology) measuring ratio and compress/decompress MB/s for none/lz4_flex/zstd/zstd+dict across representative value corpora to drive the default-policy decision.
- **[design, M2]** DESIGN: Compression interaction with mutating commands and hot-key cost. Specify behavior of APPEND/INCR/SETRANGE/bitops on compressed values (decompress-mutate-recompress vs disable-compression-for-key) and an adaptive policy that skips compression for very hot keys to protect throughput.
- **[research, M2]** RESEARCH: Per-value vs per-segment compression tradeoff for small values. Prototype and measure per-segment/per-slab compression (amortizes dictionary/framing, better ratio on tiny values) against per-value (no read amplification) to choose the architecture for the small-value regime.
