## Summary

The dominant production design for a Redis-wire-compatible cache cluster is Redis Cluster's: a fixed key space of 16384 hash slots, with slot = CRC16(key, XMODEM variant) mod 16384, slots statically assigned to master shards, and a "smart client" routing model where clients cache a slot->node map and follow MOVED/ASK redirections rather than going through a proxy. Membership and config propagation ride a full-mesh "cluster bus" (base port + 10000) using a gossip heartbeat, with failure detection via PFAIL->FAIL escalation gated on cluster-node-timeout (default 15000 ms) and replica-promotion elections versioned by a configEpoch logical clock. Replication is asynchronous leader-follower by default (WAIT gives bounded sync). This design is operationally proven but has two well-known pain points: 16384 slots was deliberately chosen so the per-node slot bitmap is 2 KB (vs 8 KB at 65536) and caps practical clusters at ~1000 masters (antirez, redis#2576); and legacy slot migration (CLUSTER SETSLOT MIGRATING/IMPORTING + per-key MIGRATE) is non-atomic, slow (~hundreds of thousands of round-trips for large slots), and leaves slots in an inconsistent state on failure — which Valkey 9.0's fork-based atomic slot migration (snapshot via child process, stream incremental mutations, atomic ownership handoff) fixes and is the better model to follow.

On partitioning algorithms, the academic state of the art has moved well past Karger ring/ketama. Rendezvous/HRW hashing (Thaler & Ravishankar, 1996) is simpler and more general than consistent hashing, gives minimal-disruption reassignment (only the failed node's keys move) and is O(n) per lookup (O(log n) with a skeleton tree). Jump consistent hash (Lamping & Veach, 2014) is ~5 lines, needs zero memory beyond registers, runs in O(ln n), and distributes near-perfectly — but its fatal limitation is that buckets must be 0..N-1 sequential, so you can only add/remove the last bucket, making it unfit for clusters with arbitrary node removal. Newer algorithms (AnchorHash, MementoHash 2023) restore arbitrary node removal at minimal memory; MementoHash also removes AnchorHash's requirement to fix a maximum capacity up front. For IronCache, the right separation is: use a slot abstraction (like Redis, for clean ownership transfer and multi-key hash-tag locality) but make the slot count and slot->node assignment a first-class, consensus-managed map rather than gossip-propagated bitmaps.

On consistency, the spectrum runs from Redis Cluster's async leader-follower (fast, can lose acknowledged writes on the minority side of a partition) through Dynamo-style leaderless quorums (Amazon Dynamo 2007: N/R/W with R+W>N, sloppy quorum + hinted handoff for availability under partition, vector clocks for causality with conflict resolution pushed to the app — which the industry found painful) to CRDT-based active-active. Redis Enterprise CRDB and KeyDB active-replica implement active-active: CRDB maps strings to LWW registers, counters to op-based PN-counters (captures total increments since last sync, so concurrent INCRs both survive), and sets to Add-Wins Observed-Remove sets; KeyDB uses pure timestamp LWW on all types, which silently loses concurrent counter increments. The research lineage here is Shapiro et al. CRDTs and Berkeley's Anna KVS (coordination-free lattice/CRDT actors, up to 700x Masstree).

For membership/failure detection, Redis Cluster's full-mesh gossip works to ~1000 nodes but scales O(N^2) in connections and bitmap-heavy heartbeats. SWIM (Das, Gupta, Motivala, 2002) decouples failure detection (randomized direct ping + k indirect pings) from dissemination (infection-style gossip) and proves expected detection time, false-positive rate, and per-member message load are all independent of group size. HashiCorp memberlist/Serf is the battle-tested Rust/Go-portable embodiment, with the Lifeguard extensions (2017) cutting false positives ~50x under CPU/network stress. SWIM is the clear borrow for IronCache's membership layer.

For IronCache specifically (Rust, single static binary, most-efficient-cache-per-core, single-node-first then multi-node), the recommended path is: ship a rock-solid single node first; design the on-wire and storage layers so a slot abstraction can be bolted on later without a rewrite; for multi-node v1 adopt Redis-Cluster-compatible client semantics (16384 slots, CRC16, hash tags, MOVED/ASK) so existing cluster clients "just work," but internally manage the authoritative slot map and membership with a small Raft group (or a Raft-managed config plane + SWIM data-plane membership) rather than gossip bitmaps, and implement atomic, fork/copy-on-write-streamed slot migration from day one. Defer active-active/CRDT mode to a later milestone as an opt-in mode (LWW strings, PN-counters, OR-sets) clearly documented as a different consistency contract.

## Mechanisms: borrow, adapt, or reject

| Mechanism | System | Stance | What it does | Rationale for IronCache |
| --- | --- | --- | --- | --- |
| Fixed 16384 CRC16 hash slots | Redis Cluster | **adapt** | Maps every key to one of 16384 slots via CRC16(key) mod 16384; slots are the unit of ownership and migration, decoupling keys from physical nodes. | Borrow the slot indirection (clean ownership transfer, hash-tag multi-key locality, MOVED/ASK compatibility), and keep 16384/CRC16 for client compatibility so existing Redis Cluster clients work unchanged. Adapt by making the slot count a build-time constant abstraction and the slot->node map a consensus-managed object instead of a gossiped bitmap, so we are not bound to the ~1000-node bitmap ceiling internally. |
| Smart-client MOVED/ASK routing | Redis Cluster | **borrow** | Clients cache a slot->node map, send directly to the owning node, and self-correct on -MOVED (permanent) and -ASK (transient, with ASKING) redirections, avoiding a proxy hop. | Zero-extra-hop routing is exactly aligned with max-throughput-per-core; it removes a proxy from the data path and is what every mature Redis client already speaks. IronCache must emit MOVED/ASK and CLUSTER SLOTS/SHARDS so the existing client ecosystem just works. |
| Proxy-based routing (twemproxy / Envoy redis_proxy / dynomite) | twemproxy, Envoy, Netflix Dynomite | **reject** | A separate proxy process shards keys (ketama / cluster-aware) and routes; Envoy adds ReadPolicy for replica reads, Dynomite adds Dynamo-style multi-DC replication around single-node stores. | A mandatory proxy adds a network hop, a CPU tax, and an extra failure/operational surface that contradicts the single-binary, max-efficiency goal. Smart-client routing dominates for the core path. (An optional embedded proxy/router mode can be revisited later for dumb clients, but not as the default.) |
| Jump consistent hash | Lamping & Veach 2014 | **reject** | Stateless ~5-line function mapping a 64-bit key to a bucket in O(ln n) with near-perfect balance and minimal key movement when bucket count grows. | Its hard constraint that buckets be 0..N-1 sequential (only the last bucket may be added/removed) is incompatible with clusters where any node can fail or be decommissioned. Great for fixed-N sharding, wrong for an elastic cache. Slots + a consensus-managed assignment give us elasticity jump hash cannot. |
| Rendezvous / HRW hashing | Thaler & Ravishankar 1996 | **adapt** | Assigns each key to the node with the highest hash(key,node) weight; minimal disruption on node change, no ring/state, O(n) (O(log n) with skeleton tree). | Use HRW not for the primary key->slot mapping (slots win for migration/compat) but for secondary placement decisions: assigning replica sets to slots, weighting heterogeneous nodes, and rebalancing — it needs no stored ring and naturally moves only the affected node's share. |
| AnchorHash / MementoHash | AnchorHash 2020 / MementoHash 2023 | **adapt** | Minimal-memory consistent hashes that, unlike jump hash, support arbitrary bucket removal; MementoHash additionally avoids fixing a maximum capacity up front. | Candidate engines for internal rebalancing/placement when a non-slot, low-memory mapping with arbitrary removal is wanted. Track as research; not the primary client-facing partitioning scheme (which stays slot-based for compatibility). |
| Async leader-follower replication | Redis Cluster | **borrow** | Master replicates to replicas asynchronously; replicas promote via configEpoch-versioned elections on failure; WAIT offers bounded synchronous acks. | Async primary/replica is the right default for a cache (latency over durability) and matches Redis semantics clients expect. Borrow it, expose WAIT for callers who need bounded sync, and document the partition-loss window honestly. |
| Dynamo leaderless quorums (sloppy quorum, hinted handoff, vector clocks) | Amazon Dynamo | **reject** | N/R/W quorums with R+W>N, sloppy quorum over first-N-healthy nodes, hinted handoff during outages, vector clocks with app-resolved conflicts. | For a Redis-compatible cache the read/write amplification (R+W replica contacts per op) and app-level conflict resolution via vector clocks are too costly and non-idiomatic; they trade exactly the per-core efficiency we are optimizing for. Hinted handoff is a useful idea to selectively adapt for replica catch-up, but the full quorum model is rejected as the default. |
| Active-active CRDTs (CRDB data-type mapping) | Redis Enterprise CRDB | **adapt** | Maps strings to LWW registers, counters to op-based PN-counters (merging concurrent increments), sets/zsets to Add-Wins Observed-Remove sets for conflict-free geo-replication. | The correct model for multi-region active-active, and strictly better than KeyDB's blanket LWW (which loses concurrent counter increments). Adapt as an opt-in, clearly separate consistency mode in a later milestone — not the default, because per-key CRDT metadata costs memory, violating the minimal-memory goal for the common case. |
| KeyDB blanket timestamp LWW active-replica | KeyDB | **reject** | Multi-master async replication where every write is timestamped and newest wins across all data types. | Simplicity is appealing but applying LWW to counters/sets silently drops concurrent updates (a counter losing increments is a correctness bug for many users). If we do active-active, do it with proper per-type CRDTs (CRDB-style), not blanket LWW. |
| Raft for slot-map / membership / config | Ongaro & Ousterhout | **borrow** | Understandable consensus (leader election + replicated log + safety) for a strongly-consistent metadata/config object. | A small Raft group owning the authoritative slot->node map, epoch, and membership gives atomic, linearizable topology changes and a clean source of truth for resharding — avoiding Redis's gossip-propagated bitmap convergence races. Keep the Raft group small (3-5 config nodes) off the data hot path. |
| SWIM membership + infection-style dissemination | SWIM (Das/Gupta/Motivala) | **borrow** | Randomized direct+indirect pinging for failure detection with gossip dissemination; per-member load and detection time independent of cluster size. | Replace Redis's O(N^2) full-mesh cluster bus with SWIM for data-plane liveness so membership scales past ~1000 nodes with bounded per-node overhead — directly serving the efficiency goal. |
| Lifeguard SWIM extensions | HashiCorp memberlist | **borrow** | Local-health-aware adjustments to SWIM probe timing/suspicion that cut false positives ~50x under CPU starvation and network loss. | A cache node under GC/CPU pressure must not be wrongly evicted, triggering needless failover. Lifeguard is proven in production (Consul/Serf) and should be adopted with the SWIM layer; a mature Rust SWIM crate likely already implements it. |
| Atomic (fork-streamed) slot migration | Valkey 9.0 | **borrow** | Child-process snapshot of migrating slots + incremental mutation streaming + atomic ownership handoff, replacing per-key MIGRATE round-trips. | This is the modern fix for resharding pain (no inconsistent half-migrated slots, no large-key OOM, far fewer round-trips). Build IronCache resharding this way from the start; in Rust, leverage copy-on-write fork or an in-process snapshot iterator over the slot's data structure. |
| Hash tags for multi-key locality | Redis Cluster / twemproxy | **borrow** | {tag} substring forces co-location of related keys in one slot so MGET/MSET/transactions/Lua over them stay single-slot. | Essential for any multi-key operation in a sharded cache and already a client convention. Implement identical {first-non-empty-brace} semantics for drop-in compatibility, and enforce CROSSSLOT errors otherwise. |

## Implications for IronCache

- Adopt Redis-Cluster-compatible client semantics for multi-node v1: 16384 slots, CRC16/XMODEM, {hash-tag} co-location, CROSSSLOT errors, and MOVED/ASK redirection plus CLUSTER SLOTS/SHARDS, so the entire existing Redis cluster client ecosystem works against IronCache with zero client changes.
- Separate the control plane from the data plane: own the authoritative slot->node map, config epoch, and membership in a small (3-5 node) Raft group rather than gossiping slot bitmaps; this gives atomic, linearizable resharding and removes Redis's ~1000-node bitmap ceiling as an internal constraint while still emitting a Redis-shaped slot map to clients.
- Use SWIM (+Lifeguard) for data-plane liveness/failure detection so per-node overhead and detection time stay constant as the cluster grows, instead of a full-mesh cluster bus that is O(N^2) in connections.
- Implement atomic slot migration (Valkey-9.0 style: fork/COW or in-process snapshot iterator + incremental mutation stream + atomic ownership handoff) from day one; never ship the legacy per-key MIGRATE model, which is slow, non-atomic, and OOM-prone on large keys.
- Keep async primary/replica replication as the default consistency contract (latency-first, cache-appropriate), expose WAIT for bounded sync, and document the partition write-loss window explicitly in the README.
- Design the single-node storage engine with a slot-indexed structure (e.g., per-slot shards/dictionaries like Valkey 8.0's per-slot dict) even before clustering exists, so multi-node is a bolt-on, not a rewrite — this is the concrete realization of 'single-node-first path to multi-node'.
- Treat active-active/geo-replication as an opt-in mode in a later milestone using proper per-type CRDTs (LWW-register strings, op-based PN-counters, Add-Wins OR-Sets a la Redis CRDB), and reject KeyDB-style blanket LWW because it silently loses concurrent counter increments; gate the per-key CRDT metadata cost behind this mode to protect the minimal-memory default.
- Reject a mandatory proxy in the data path (twemproxy/Envoy/dynomite) to preserve max-throughput-per-core; if a 'dumb client' path is ever needed, offer it as an optional embedded router mode, not the default topology.
- Consider HRW/rendezvous (stateless, minimal-disruption) for internal placement decisions — assigning replica sets to slots and weighting heterogeneous-capacity nodes — rather than for the client-facing key mapping.
- Tune failure-detection defaults for cache workloads: pick a node-timeout default (Redis uses 15000 ms) with profiles for LAN vs cloud/WAN, and lean on Lifeguard-style local-health awareness so GC/CPU-starved nodes are not falsely evicted and made to fail over.

## Research papers and primary sources

- **A Fast, Minimal Memory, Consistent Hash Algorithm (Jump Consistent Hash)** (arXiv 1406.2294, 2014). A ~5-line, zero-memory, O(ln n) consistent hash that distributes keys near-perfectly and minimizes movement when bucket count grows, at the cost of requiring sequential bucket numbering. [source](https://arxiv.org/abs/1406.2294) Relevance: Defines the trade-off space for partitioning; its sequential-bucket limitation is exactly why IronCache should use a slot+consensus map instead of jump hash for an elastic cluster.
- **SWIM: Scalable Weakly-consistent Infection-style Process Group Membership Protocol** (IEEE DSN 2002). Decouples randomized ping-based failure detection from infection-style membership dissemination so per-member load and detection time are independent of group size. [source](https://www.cs.cornell.edu/projects/Quicksilver/public_pdfs/SWIM.pdf) Relevance: The recommended membership/failure-detection layer for IronCache, replacing Redis's O(N^2) full-mesh cluster bus.
- **Lifeguard: Local Health Awareness for More Accurate Failure Detection** (HashiCorp / arXiv 1707.00788, 2017). Adds local-health awareness to SWIM probe timing and suspicion, cutting false positives ~50x under CPU starvation and network loss. [source](https://arxiv.org/pdf/1707.00788) Relevance: Hardens IronCache's SWIM layer against falsely evicting GC/CPU-stressed cache nodes and triggering needless failovers.
- **In Search of an Understandable Consensus Algorithm (Raft)** (USENIX ATC 2014). A Paxos-equivalent consensus protocol decomposed into leader election, log replication, and safety, optimized for understandability and practical implementation. [source](https://raft.github.io/raft.pdf) Relevance: Basis for IronCache's strongly-consistent slot-map/membership/config control plane.
- **Dynamo: Amazon's Highly Available Key-value Store** (SOSP 2007). Leaderless availability via N/R/W quorums (R+W>N), sloppy quorum + hinted handoff under partition, and vector-clock causality with app-level conflict resolution. [source](https://www.cs.cornell.edu/courses/cs5414/2017fa/papers/dynamo.pdf) Relevance: Defines the high-availability/eventual-consistency end of the spectrum; IronCache rejects the full quorum model for the default path but may selectively adapt hinted handoff.
- **Anna: A KVS for Any Scale** (IEEE ICDE 2018). Coordination-free actors merging lattice/CRDT-based state, achieving up to 700x Masstree while offering flexible (coordination-free) consistency across scales. [source](https://dsf.berkeley.edu/jmh/papers/anna_ieee18.pdf) Relevance: Reference design for IronCache's optional active-active/CRDT mode and for how far coordination-free per-core throughput can be pushed.
- **MementoHash: A Stateful, Minimal Memory, Best Performing Consistent Hash Algorithm** (IEEE / arXiv 2306.09783, 2023). A minimal-memory consistent hash that supports arbitrary node removal (unlike JumpHash) and does not require fixing maximum capacity up front (unlike AnchorHash/DxHash). [source](https://arxiv.org/pdf/2306.09783) Relevance: Candidate internal placement/rebalancing engine; informs the survey of post-ketama consistent-hashing options for IronCache.
- **AnchorHash: A Scalable Consistent Hash** (IEEE/ACM Transactions on Networking, 2021). Consistent hashing with very low memory and fast lookups supporting arbitrary bucket removal, given a fixed maximum capacity. [source](https://www.researchgate.net/publication/347760049_AnchorHash_A_Scalable_Consistent_Hash) Relevance: Comparison point for IronCache's partitioning research; trade-off is the fixed-capacity requirement MementoHash later removes.

## Open questions

- Should IronCache keep exactly 16384 slots for wire compatibility, or expose a larger/configurable slot count internally while presenting 16384 to clients via a mapping layer?
- Can a fork/COW snapshot for atomic slot migration be done efficiently in a Rust single binary, or do we need an in-process MVCC/snapshot-iterator over per-slot structures to avoid doubling RSS?
- Is a single Raft group for the slot map sufficient at target scale, or do we need a sharded/hierarchical config plane (multi-Raft) for very large clusters?
- Does SWIM membership compose cleanly with Redis-Cluster-shaped CLUSTER SLOTS/SHARDS output that clients poll, or do we need a translation/cache layer to present a Redis-style view over SWIM state?
- What is the right default failure-detection timeout and suspicion configuration for cache workloads across LAN vs cloud/WAN, balancing fast failover against false positives under GC pauses?
- For the optional active-active mode, what is the per-key memory overhead of PN-counter/OR-Set metadata, and can it be made opt-in per-key/per-keyspace so the minimal-memory default is unaffected?
- How should multi-key/transaction/Lua semantics behave during atomic slot migration to avoid the -TRYAGAIN/CROSSSLOT corner cases that plague Redis Cluster?
- Should replica reads be supported (Envoy-style ReadPolicy) given async replication staleness, and if so how is staleness bounded and surfaced to clients?

## Proposed issues (seeds for the tracker)

- **[decision, M0]** Decision: single-node-first roadmap with slot-ready storage layout. Commit to shipping a single-node binary first while structuring the storage engine around per-slot shards so multi-node clustering is a bolt-on rather than a rewrite. Capture the staged path: single node -> async replica -> sharded cluster -> optional active-active.
- **[design, M1]** Design: Redis-Cluster-compatible client contract (16384 slots, CRC16, hash tags, MOVED/ASK). Specify the client-facing clustering wire contract — CRC16/XMODEM slotting into 16384 slots, {hash-tag} co-location, CROSSSLOT errors, MOVED/ASK redirection, and CLUSTER SLOTS/SHARDS — so existing Redis cluster clients work unchanged.
- **[design, M1]** Design: Raft-managed authoritative slot map and membership control plane. Define a small (3-5 node) Raft group that owns the linearizable slot->node map, config epoch, and membership, replacing gossip-propagated slot bitmaps and removing the ~1000-node bitmap ceiling as an internal constraint.
- **[design, M1]** Design: SWIM + Lifeguard data-plane membership and failure detection. Adopt a SWIM-based membership layer with Lifeguard local-health awareness for constant per-node overhead and ~50x fewer false positives, and decide how to present a Redis-shaped cluster view over SWIM state.
- **[design, M2]** Design: atomic, fork/snapshot-streamed slot migration. Specify Valkey-9.0-style resharding — snapshot a migrating slot, stream incremental mutations, then atomically hand off ownership — so migration is non-blocking, crash-safe, and free of inconsistent half-migrated slots and large-key OOM.
- **[research, M0]** Research: post-ketama consistent hashing (jump, HRW, AnchorHash, MementoHash) for internal placement. Benchmark jump hash, rendezvous/HRW, AnchorHash, and MementoHash for replica-set placement, heterogeneous-node weighting, and rebalancing, documenting why slot-based partitioning stays the client-facing scheme.
- **[decision, M1]** Decision: replication and consistency default (async primary/replica + WAIT). Adopt asynchronous primary/replica replication as the default with WAIT for bounded synchronous acks, and document the partition write-loss window; reject Dynamo-style quorums as the default path.
- **[research, M2]** Research: opt-in active-active CRDT mode (LWW strings, PN-counters, OR-Sets). Evaluate a CRDB-style per-type CRDT mode for geo active-active, quantifying per-key metadata overhead, and explicitly reject KeyDB-style blanket LWW that loses concurrent counter increments.
- **[non-goal, M0]** Non-goal: mandatory proxy in the data path. Declare that IronCache will not require a proxy (twemproxy/Envoy/dynomite) on the hot path; routing is smart-client via MOVED/ASK to preserve max throughput per core. An optional embedded router may be revisited later.
- **[epic, M0]** Epic: IronCache distributed clustering (partitioning, routing, replication, membership). Umbrella epic tracking the multi-node design: slot-based partitioning with Redis-compatible client semantics, Raft control plane, SWIM data-plane membership, atomic slot migration, async replication, and an optional CRDT active-active mode.
