## Summary

Redis OSS core (pinned to tag 8.8.0, commit 5a693aa, released 2026-05-25) is built around a single-threaded command-execution model on top of an event loop (`ae.c`: `aeMain` -> `aeProcessEvents` -> `aeApiPoll`), with a compile-time-selected multiplexing backend (evport > epoll > kqueue > select). The crucial invariant: all command execution, all keyspace mutation, expiration, and eviction run on ONE main thread; there is no per-key locking. Since 6.0 Redis offloaded socket I/O to `io-threads`, and Redis 8.0 shipped a rewritten threaded-I/O model where the main thread assigns clients to I/O threads which do socket reads + RESP parsing and socket writes, while the main thread still executes every command serially. The documented gain is "up to 112% throughput" at `io-threads 8` on a multi-core Intel CPU, but the default is `io-threads 1` (single-threaded out of the box); `io-threads-do-reads` is now a deprecated no-op (reads are always threaded in the new model). RESP2/RESP3 are handled at the reply-encoding layer per-client (HELLO negotiates protocol), not as separate execution paths.

Expiration is two-pronged. Lazy expiration checks TTL on access (`expireIfNeeded`); active expiration runs `activeExpireCycle` in two modes — a SLOW cycle from `databasesCron`/`serverCron` at `hz` (default 10, max 500) bounded to ~25% CPU, and a FAST cycle in `beforeSleep` on every event-loop iteration bounded to 1000us. Defaults (`expire.c`): `ACTIVE_EXPIRE_CYCLE_KEYS_PER_LOOP 20`, `ACTIVE_EXPIRE_CYCLE_FAST_DURATION 1000us`, `ACTIVE_EXPIRE_CYCLE_SLOW_TIME_PERC 25`, `ACTIVE_EXPIRE_CYCLE_ACCEPTABLE_STALE 10%`, all scaled by `active-expire-effort` (default 1, range 1-10). Active expiry runs only on masters; replicas wait for DEL/UNLINK propagation. This sampling design tolerates up to ~10% stale keys before escalating — a deliberate "good-enough" probabilistic approach rather than an exact timer wheel.

Eviction is sampled and approximate. `maxmemory` defaults to 0 (no limit) and `maxmemory-policy` defaults to `noeviction` (`MAXMEMORY_NO_EVICTION`), meaning out-of-the-box Redis is NOT a cache that evicts — it returns OOM errors on writes. The 8 classic policies plus two new in 8.6 (`volatile-lrm`/`allkeys-lrm` = Least Recently Modified, evicting by last-write rather than last-access) are dispatched via flag bits. Approximated LRU stores a 24-bit clock (`LRU_BITS 24`, `LRU_CLOCK_MAX = 2^24-1`, `LRU_CLOCK_RESOLUTION 1000ms`) in the robj's `lru:24` bitfield; eviction samples `maxmemory-samples` keys (default 5; ~10 approaches true LRU per Redis's own simulation graphs) into a 16-entry pool (`EVPOOL_SIZE 16`). LFU repurposes the same 24-bit field as 16-bit last-decay-time (minutes) + 8-bit Morris probabilistic counter; `LFULogIncr` increments with probability 1/((counter-5)*lfu-log-factor+1) (defaults `lfu-log-factor 10`, `LFU_INIT_VAL 5`, saturating ~1M accesses), and `LFUDecrAndReturn` decays one unit per `lfu-decay-time` minutes (default 1).

Memory accounting uses `zmalloc_used_memory()` (allocator logical bytes, NOT OS RSS), subtracting replica output buffers and the AOF buffer, so fragmentation can push RSS well above `maxmemory`. `performEvictions` runs before command execution and is time-bounded by `maxmemory-eviction-tenacity` (default 10 -> 500us; geometric to ~2min at 99); if it can't keep up it starts a timer-proc to keep evicting and may return EVICT_RUNNING/EVICT_FAIL. Freeing big objects is offloaded to background BIO threads via lazyfree/UNLINK when `lazyfreeGetFreeEffort > LAZYFREE_THRESHOLD (64)` and refcount==1; `lazyfree-lazy-eviction`/`-expire`/`-server-del`/`-user-del` all default to 0 (off). active-defrag (defaults `active-defrag-cycle-min 1`, `-cycle-max 25`, `-threshold-lower 10`, `-ignore-bytes 100MB`) only works when compiled with jemalloc (`HAVE_DEFRAG`).

For IronCache, the load-bearing lessons: (1) the single-threaded keyspace is what makes Redis simple and predictable but also what caps it at one core for command execution — IronCache's "max throughput per core" goal should consciously decide between Redis's shared-nothing-per-core sharding (like KeyDB/Dragonfly) vs. fine-grained concurrent structures; (2) Redis's approximate-everything philosophy (sampled LRU/LFU, probabilistic counters, sampled expiry) trades exactness for O(1) memory and bounded latency — a strong default to borrow; (3) the 24-bit-in-the-object-header metadata budget is a model worth beating with better admission/eviction (TinyLFU/W-TinyLFU); (4) `noeviction` + `maxmemory 0` defaults make stock Redis a surprising footgun as a cache — IronCache should ship cache-sane defaults; (5) accounting on allocator bytes not RSS, plus jemalloc-only defrag, is a portability and accuracy wart IronCache can address from day one in Rust.

## Mechanisms: borrow, adapt, or reject

| Mechanism | System | Stance | What it does | Rationale for IronCache |
| --- | --- | --- | --- | --- |
| Single-threaded event loop (ae) with single-threaded command execution | Redis OSS 8.8.0 | **adapt** | aeMain drives aeProcessEvents -> aeApiPoll (epoll/kqueue); all command execution and keyspace mutation happen on one thread with no per-key locks, giving lock-free determinism and trivial atomicity per command. | The lock-free simplicity is the single best thing about Redis and worth preserving for correctness, but a strictly single command thread caps throughput at one core, contradicting IronCache's 'max throughput per core' AND multi-core scaling goals. IronCache should adapt toward a shared-nothing per-core sharded model (thread-per-core, like Dragonfly/Seastar) or sharded locking, keeping each shard single-threaded internally so per-key atomicity stays lock-free while total throughput scales across cores. |
| Redis 8.0 threaded I/O (offload read/parse/write to io-threads) | Redis OSS 8.0-8.8 | **reject** | Main thread assigns clients to io-threads; io-threads do socket recv + RESP parse and socket send; main thread still executes every command. Default io-threads=1; up to ~112% throughput at 8. | This is a workaround for the single command thread: it parallelizes only I/O while command execution remains a serial bottleneck, and it adds cross-thread handoff overhead and complexity. IronCache should instead make the whole request path (I/O + parse + execute) parallel via per-core sharding, avoiding the awkward main-thread fan-in entirely. |
| Approximated LRU via random sampling + eviction pool | Redis OSS 8.8.0 | **adapt** | Stores a 24-bit second-resolution clock per object; on eviction samples maxmemory-samples (default 5) keys into a 16-entry pool sorted by idle time and evicts the best candidate. ~10 samples approaches true LRU. | O(1)-memory approximate eviction is the right philosophy, but sampled LRU is provably worse than modern admission policies (it has no frequency signal and scans random keys). IronCache should adapt to a W-TinyLFU/SLRU design (count-min sketch admission + segmented LRU) which beats LRU/LFU hit-rate at comparable metadata cost, while keeping the sample-don't-sort, bounded-work spirit. |
| LFU with 8-bit Morris probabilistic counter + decay | Redis OSS 8.8.0 | **adapt** | Reuses the 24-bit field as 16-bit last-decay-minute + 8-bit counter; LFULogIncr increments with prob 1/((counter-5)*lfu_log_factor+1) (saturates ~1M hits at factor 10), LFUDecrAndReturn decays one per lfu-decay-time minutes (default 1). | The probabilistic-counter idea (track frequency in tiny space, log-scale to avoid saturation, time-decay to age out) is excellent and directly relevant. Adapt by folding it into a TinyLFU count-min sketch with aging (the academic refinement of exactly this idea), giving frequency estimation decoupled from per-object storage so cold keys cost zero counter bytes. |
| Dual-mode active expiration (slow cron cycle + fast beforeSleep cycle) | Redis OSS 8.8.0 | **adapt** | Lazy expiry on access plus a SLOW activeExpireCycle at hz (default 10) capped at 25% CPU and a FAST cycle each event-loop iteration capped at 1000us; samples 20 keys/db, escalates if >10% sampled are stale; masters only. | Probabilistic sampled expiry with a stale-ratio feedback loop is a sound, low-overhead default to borrow. Adapt the scheduling: a per-shard timer/hierarchical timing wheel can give tighter bounds on memory held by expired-but-not-reclaimed keys than blind sampling, and IronCache can expire proactively per shard without the master-only constraint. |
| maxmemory accounting on allocator logical bytes (zmalloc), not RSS | Redis OSS 8.8.0 | **adapt** | getMaxmemoryState compares zmalloc_used_memory() (minus replica output buffers + AOF buffer) to maxmemory; OS RSS can exceed maxmemory due to fragmentation, requiring separate active-defrag. | Accounting against a true budget is essential for a 'minimal memory' cache, but counting only allocator-requested bytes lets fragmentation blow past the limit and forces a jemalloc-only defrag band-aid. IronCache should account for real allocator overhead (slab/size-class internal fragmentation) up front and choose an allocator with predictable fragmentation, so 'maxmemory' is an honest bound. |
| Lazyfree / UNLINK background reclamation | Redis OSS 8.8.0 | **borrow** | Large objects (free_effort > LAZYFREE_THRESHOLD 64, refcount==1) are freed on background BIO threads to keep the main thread responsive; gated by lazyfree-lazy-* configs (all default off). | Moving expensive frees off the hot path is straightforwardly good and maps cleanly onto a Rust background-drop thread or per-shore reclamation queue. IronCache should borrow this and, unlike Redis, make async reclamation the default rather than opt-in. |
| noeviction + maxmemory=0 default posture | Redis OSS 8.8.0 | **reject** | Out of the box Redis has no memory limit and never evicts; writes fail with OOM once the OS limit is hit unless an admin configures maxmemory and a policy. | For a product whose identity is 'a cache', defaulting to a non-evicting unbounded store is a footgun that pushes correctness onto operators. IronCache should ship a sane cache default (a memory ceiling derived from the host plus an LRU/W-TinyLFU eviction policy enabled) while still allowing a strict no-evict mode for datastore use. |
| active-defrag (jemalloc-dependent compaction) | Redis OSS 8.8.0 | **reject** | Incrementally relocates allocations to reduce external fragmentation, CPU-bounded 1-25%, only when fragmentation > threshold and only if built with jemalloc (HAVE_DEFRAG). | Active-defrag exists only because malloc-based heap layout fragments badly; it is jemalloc-specific, CPU-costly, and a symptom rather than a cure. IronCache should design storage (slab/arena/region allocation with compaction-friendly layout) so external fragmentation is structurally bounded, removing the need for an online defragmenter. |

## Implications for IronCache

- Decide the concurrency model explicitly: Redis caps command execution at one core; IronCache's 'max throughput per core AND scale to multi-node' goal points to thread-per-core shared-nothing sharding (each shard single-threaded internally for lock-free per-key atomicity), which sidesteps Redis's awkward io-threads fan-in. Document the chosen model in the vision EPIC.
- Ship cache-sane defaults: a memory ceiling auto-derived from host RAM plus an eviction policy ON by default (unlike Redis noeviction + maxmemory=0). Offer a strict 'no-evict / datastore' mode as opt-in. This single decision differentiates IronCache's out-of-box UX.
- Adopt W-TinyLFU (admission via count-min sketch + segmented LRU) rather than Redis's sampled LRU/LFU; it has higher hit ratios at comparable or lower metadata cost and is the state of the art (Caffeine, Ristretto). Keep Redis's bounded-work, sample-don't-globally-sort discipline.
- Budget per-object metadata deliberately: Redis spends 24 bits in the object header for LRU/LFU. IronCache should aim to keep frequency/recency metadata out of per-object headers (e.g. a shared sketch) so cold keys cost near-zero, directly serving the 'minimal memory' goal.
- Make maxmemory an honest bound: account for allocator internal fragmentation and size-class overhead, not just requested bytes, and pick an allocator with predictable fragmentation so IronCache does not need a Redis-style online defragmenter.
- Make background reclamation (async drop of large values, lazy-free) the default, using a Rust background thread or per-shard reclamation queue, instead of Redis's opt-in lazyfree-lazy-* flags.
- Use a per-shard hierarchical timing wheel for TTLs to bound the memory held by expired-but-unreclaimed keys, improving on Redis's probabilistic sampled active-expire while keeping lazy-on-access expiry as a backstop.
- Maintain Redis RESP2/RESP3 wire compatibility but keep the parser and reply encoder zero-copy and per-shard; benchmark against redis-benchmark and memtier so the 'most efficient cache' claim is grounded in head-to-head numbers vs Redis 8.8 io-threads=N.
- Consider a Least-Recently-Modified-style signal (Redis 8.6 LRM) for read-heavy workloads where write-recency predicts liveness better than access-recency; evaluate whether W-TinyLFU already subsumes this need.

## Research papers and primary sources

- **TinyLFU: A Highly Efficient Cache Admission Policy** (ACM TOS / EUROMICRO-PDP 2014 (journal 2017)). Use an aging count-min sketch as a frequency-based admission filter in front of an eviction policy, deciding whether a newly-arrived item should displace the eviction victim. Tiny metadata, near-optimal hit ratio. [source](https://arxiv.org/abs/1512.00727) Relevance: Directly the principled successor to Redis's sampled LFU/Morris counter; the basis for IronCache's recommended W-TinyLFU eviction with shared (not per-object) frequency state.
- **Adaptive Software Cache Management (W-TinyLFU / Caffeine)** (ACM Middleware 2018). Window-TinyLFU adds an admission window + segmented main region (SLRU) and adaptively resizes the window, beating LRU/LFU/ARC across diverse traces; implemented in Caffeine. [source](https://dl.acm.org/doi/10.1145/3274808.3274816) Relevance: Concrete, production-proven eviction design IronCache should adopt; quantifies hit-ratio wins over the LRU/LFU family Redis uses.
- **Memshare: a Dynamic Multi-tenant Key-value Cache** (USENIX ATC 2017). Log-structured, slab-free memory management with sharing across tenants and cleaning to control fragmentation, raising effective hit rate by reclaiming wasted memory. [source](https://www.usenix.org/conference/atc17/technical-sessions/presentation/cidon) Relevance: Informs IronCache's allocator/storage layout decision so it avoids Redis's jemalloc-dependent active-defrag and bounds fragmentation structurally.
- **The CacheLib Caching Engine: Design and Experiences at Scale** (USENIX OSDI 2020). A general-purpose embeddable cache library with hybrid DRAM/flash, slab-based allocation, and pluggable eviction; documents real-world tradeoffs of approximate LRU, TTL reaping, and memory accounting at Facebook scale. [source](https://www.usenix.org/conference/osdi20/presentation/berg) Relevance: Engineering reference for IronCache's memory accounting, eviction abstraction, and the value of bounded-work background maintenance over Redis-style sampling.
- **Scaling Memcache at Facebook** (USENIX NSDI 2013). Operational lessons on running a large key-value cache fleet: lease/stale handling, the lazy vs proactive expiration tradeoff, and the importance of predictable latency over peak throughput. [source](https://www.usenix.org/conference/nsdi13/technical-sessions/presentation/nishtala) Relevance: Grounds IronCache's expiration and single-node-to-multi-node design choices in real distributed-cache operating experience.
- **Counting Large Numbers of Events in Small Registers (Morris counter)** (Communications of the ACM 1978). Probabilistic approximate counting that tracks counts up to large magnitudes in very few bits by incrementing with decreasing probability. [source](https://dl.acm.org/doi/10.1145/359619.359627) Relevance: The exact algorithm behind Redis's 8-bit LFU counter; documents the accuracy/space tradeoff IronCache must reason about if it keeps any per-object counter.

## Open questions

- Thread-per-core shared-nothing (Dragonfly/Seastar style) vs sharded-lock vs Redis-style single command thread + io-threads: which maximizes throughput-per-core for IronCache while keeping per-key operations atomic and RESP-compatible MULTI/EXEC and Lua semantics intact?
- Can W-TinyLFU's frequency sketch + SLRU be made cross-shard or must each shard keep its own admission state, and what is the hit-ratio cost of per-shard partitioning of the frequency signal?
- What is the right default memory ceiling heuristic (e.g. fraction of host RAM) and how should IronCache expose 'cache mode' vs 'datastore (no-evict) mode' without surprising operators migrating from Redis defaults?
- How should maxmemory account for allocator internal fragmentation in Rust (which allocator: jemalloc/mimalloc/custom slab) so the limit is honest without needing an online defragmenter?
- Does IronCache need an LRM-style last-modified signal, or does a frequency+recency policy (W-TinyLFU) already capture the read-heavy workloads LRM targets?
- What is the latency/throughput cost of RESP3 push/attribute support and client-side caching invalidation tracking, and should tracking state be per-shard?
- What is the empirical head-to-head throughput-per-core and hit-ratio of an IronCache prototype vs Redis 8.8 (io-threads tuned) on standard memtier/zipfian traces, and which workloads expose Redis's single-command-thread ceiling most?

- **[research, M0]** RESEARCH: Per-object metadata budget for recency/frequency vs shared sketch.
