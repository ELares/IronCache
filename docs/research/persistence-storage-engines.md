## Summary

The persistence design space for a cache splits into four points, in increasing cost: ephemeral-only (no disk, fastest, lose everything on restart — Memcached's historical default), snapshot (periodic point-in-time dump — Redis RDB), append-log (every write logged — Redis AOF), and tiered RAM->SSD (larger-than-RAM, hot in memory, cold on flash — FASTER/F2, KeyDB FLASH, Memcached extstore). IronCache's stated goal (max throughput per core, minimal memory, single binary) argues for a default of ephemeral-or-snapshot with an optional tiered backend, and a hard rejection of fork()-based copy-on-write snapshotting because it is the single biggest memory-efficiency footgun in the incumbent (Redis) design.

The most important mechanism to borrow is Dragonfly's fork-less, versioned point-in-time snapshot. Instead of fork()+COW (which can double RSS under write-heavy load, the exact behavior Redis exhibits and KeyDB budgets ~30% headroom for), Dragonfly maintains a monotonic version counter; each entry records its last-update version; snapshot start captures an epoch cut; a traversal fiber serializes all entries with version <= cut, and a mutation hook (the "conservative" variation, used for file backups) pushes the pre-image of any entry to the serialization sink before an in-flight write modifies it. Memory overhead is constant regardless of dataset size, and the push-on-write creates natural back-pressure. This is the single best-fit idea for a most-efficient-cache: it gives a consistent snapshot with no fork, no RSS spike, and integrates with io_uring async writes. The dependency is a sharded, single-writer-per-shard architecture (shared-nothing) so each shard serializes only its own data through a blocking channel — an architectural commitment IronCache should make regardless.

For larger-than-RAM, the two viable engine families are the FASTER/F2 hybrid log and an LSM (RocksDB). FASTER's HybridLog is a single logical address space split into a mutable region (in-place updates, no allocation), a read-only region (immutable, RCU to the tail on update), and a stable/disk region; epoch protection provides latch-free memory reclamation and lazy cross-thread synchronization. F2 evolves this into an explicit hot-log (memory) + cold-log (disk) two-tier design with a read cache, reporting up to 3.2x higher throughput than RocksDB on skewed workloads (original FASTER hit ~160M ops/sec in-memory on YCSB). The hybrid-log/epoch approach is a strong borrow because it is latch-free and Rust-implementable, and it avoids the LSM write-amplification tax. RocksDB-as-cold-backend (KeyDB FLASH, Memcached extstore both effectively do this) is rejected as a core engine: leveled compaction write amplification is "often larger than 10" (Facebook prod uses a size multiplier of 10), which is poison for SSD lifetime on a high-churn cache workload, and it imports a large C++ dependency that breaks the single-static-binary promise.

Memcached extstore is the cleanest minimalist tiering design and worth borrowing selectively: keep keys + a 12-byte [page, offset, version] pointer in RAM, write values to flash, with tunables ext_item_size (default 512B), ext_page_size (default 64M), ext_wbuf_size (default 8M, 32M reserved total), and compaction governed by ext_max_frag / ext_compact_under (default 1/4 of pages) / ext_drop_unread. The 12-bytes-per-resident-value model is the key efficiency insight: you can address far more flash than RAM with tiny per-item overhead. Notably extstore is incompatible with warm restart in Memcached, a constraint IronCache should design around rather than inherit.

Warm restart (Memcached 1.5.18+) is a high-value, low-cost feature to adapt: put item data in an mmap file on a tmpfs/DAX mount via -e, send SIGUSR1 for graceful shutdown, write a .meta state file, and on restart fix up internal pointers and regenerate the hash table — preserving cache contents across binary upgrades with zero disk-format serialization. For a single-binary cache that will be redeployed often, mmap warm-restart is more valuable than full durability and far cheaper than snapshotting. The constraints (clock must not jump; writes during the restart window are lost; incompatible with the external-storage path) are acceptable for a cache.

io_uring is the clear I/O substrate for any snapshot or tiering writes: registered/fixed buffers plus SQPOLL can remove the submit syscall from the hot path entirely, with reported throughput gains (e.g., 238k -> 546k tx/s with SQPOLL in one benchmark) and 20-40% p99 improvements over epoll for batched I/O. In Rust this is reachable via tokio-uring or glommio's thread-per-core model, which aligns with a shared-nothing shard design. The synthesis for IronCache: default ephemeral; opt-in fork-less versioned snapshot (Dragonfly-style) written via io_uring; opt-in mmap warm-restart for fast upgrades; and a longer-term FASTER-style hybrid-log tiered backend rather than an LSM, to honor the throughput-per-core and SSD-friendliness goals. Append-log (AOF-style) durability is a possible later add for users who need bounded data loss (everysec = <=1s loss window), but it is not the efficiency-optimal default.

## Mechanisms: borrow, adapt, or reject

| Mechanism | System | Stance | What it does | Rationale for IronCache |
| --- | --- | --- | --- | --- |
| Fork-less versioned point-in-time snapshot | Dragonfly | **borrow** | Monotonic version counter + per-entry last-update version + epoch cut; a traversal fiber serializes all entries <= cut, and an OnWrite hook pushes the pre-image of an entry to the serialization sink before a concurrent write mutates it. Memory overhead is constant, no fork, no COW RSS spike, with natural push-on-write back-pressure. | This is the single best mechanism for a most-efficient cache: a consistent snapshot with zero fork and no memory doubling. Directly serves IronCache's minimal-memory and throughput-per-core goals and pairs with io_uring writes. Requires committing to a shared-nothing per-shard single-writer architecture (which IronCache should do anyway). |
| fork()+copy-on-write snapshot (BGSAVE) | Redis | **reject** | Forks a child that writes the dataset to disk while the parent serves traffic; modified pages are COW-duplicated. | COW can roughly double RSS under write-heavy load (the exact reason KeyDB budgets +30% headroom). This is the biggest memory-efficiency footgun in Redis and is antithetical to IronCache's minimal-memory goal. Replace with Dragonfly-style versioned snapshot. |
| HybridLog (mutable / read-only / stable regions) | FASTER | **borrow** | Single logical address space partitioned into a mutable region (in-place updates, no allocation), a read-only region (immutable, RCU appends to the tail on update), and a stable on-disk region; recency determines region. | In-place updates in the mutable region give allocation-free hot writes (great for throughput-per-core), and the unified log naturally spills cold data to disk for larger-than-RAM. Latch-free and Rust-implementable. Foundation for IronCache's optional tiered backend. |
| Epoch protection framework | FASTER | **borrow** | Global epoch counter; threads register the current epoch; trigger actions fire when all threads pass an epoch (drain list), enabling latch-free, lazy cross-thread synchronization and safe memory reclamation. | Latch-free reclamation is exactly what a thread-per-core/shared-nothing Rust cache needs to avoid per-op atomics/latches on the hot path. Maps cleanly onto crossbeam-epoch-style reclamation in Rust. |
| Hot-log + cold-log two-tier with read cache | F2 (FASTER follow-on) | **adapt** | Explicit in-memory hot-log and on-disk cold-log; records migrate hot->cold as they age; a read cache absorbs cold-log re-reads. Reports 2-11.9x throughput over KV stores and up to ~3.2x over RocksDB on skewed workloads. | The two-tier separation and read cache are the right tiering model for skewed cache traffic; adapt by simplifying F2's two-level hash index and aligning the migration policy with cache eviction (LRU/LFU) rather than database-style aging. |
| extstore: keys in RAM, values on flash with 12B pointer | Memcached | **adapt** | Hash table + keys + a 12-byte [page,offset,version] pointer stay in RAM; values flushed to flash above ext_item_size (512B) into 64M pages via 8M write buffers; compaction governed by ext_max_frag/ext_compact_under (1/4 pages)/ext_drop_unread. | The 12-bytes-per-resident-value model is the cleanest minimal-overhead tiering design and lets you address far more flash than RAM. Adapt the file/page/compaction layer to use io_uring and a Rust-native log allocator; keep the in-RAM-pointer principle and the drop-unread compaction idea. |
| RocksDB / LSM as cold backend | KeyDB FLASH / Memcached (alt) | **reject** | Use an embedded LSM (leveled compaction) as the durable cold tier for larger-than-RAM data. | Leveled compaction write amplification is often >10x, which destroys SSD endurance on high-churn cache workloads, and embedding RocksDB (large C++ dep) breaks the single-static-binary, max-efficiency promise. Prefer a hybrid-log (FASTER-style) cold tier whose write amplification is bounded by log append + occasional compaction. |
| mmap warm restart (-e memory_file, SIGUSR1, .meta) | Memcached | **borrow** | Item data lives in an mmap file on tmpfs/DAX; SIGUSR1 triggers graceful shutdown writing a .meta state file; on restart pointers are fixed up and the hash table regenerated, preserving cache contents across binary upgrades without on-disk serialization. | For a frequently-redeployed single binary, warm restart preserves the working set across upgrades at near-zero cost and no disk format. Higher value-per-effort than full durability for a cache. Inherit the SIGUSR1/state-file pattern; design to avoid Memcached's warm-restart-XOR-extstore limitation. |
| io_uring with SQPOLL + registered buffers | Linux kernel | **borrow** | Async submission/completion ring; SQPOLL kernel thread removes the submit syscall from the hot path; registered/fixed buffers and files remove per-op validation/copy for zero-copy I/O. | The right substrate for snapshot and tiering writes; keeps the I/O path off the request-serving cores. In Rust via tokio-uring or glommio (thread-per-core), aligning with the shared-nothing shard model. Must keep a non-uring fallback for older kernels/macOS dev. |
| AOF append-log durability (appendfsync everysec) | Redis | **adapt** | Logs every write command to a file, fsynced every second on a background thread (<=1s loss window); periodic rewrite compacts the log, optionally with an RDB preamble base. | Append-log is the right tool only for users who need bounded data loss; it is not the efficiency-optimal default. Offer it as an opt-in tier (write-ahead binary op-log via io_uring) rather than as the primary persistence path, since continuous fsync taxes throughput-per-core. |

## Implications for IronCache

- Default to ephemeral; make durability strictly opt-in. The most-efficient-cache default is no persistence overhead at all; durability is a feature flag, not a tax everyone pays.
- Adopt a shared-nothing, single-writer-per-shard architecture from day one. It is the prerequisite for Dragonfly-style fork-less snapshots, FASTER-style epoch protection, and glommio/tokio-uring thread-per-core I/O.
- Implement snapshotting as Dragonfly-style versioned point-in-time, never fork()+COW. Per-entry version + epoch cut + push-pre-image-on-write gives a consistent snapshot with constant memory overhead, the opposite of Redis's up-to-2x RSS spike.
- Write snapshots through io_uring (SQPOLL + registered buffers) on the I/O cores so the snapshot stream never blocks request-serving shards; keep a portable fallback path for older kernels and macOS development.
- Ship mmap warm-restart early (SIGUSR1 graceful shutdown + .meta state file + pointer fixup) so binary upgrades preserve the working set with no on-disk format — high value for a frequently-redeployed single binary.
- For larger-than-RAM, build a FASTER/F2-style hybrid log (mutable in-place / read-only RCU / cold on SSD) rather than embedding RocksDB; this avoids the >10x leveled-compaction write amplification that shortens SSD life on high-churn cache workloads and keeps the single static binary.
- Borrow extstore's principle of keeping keys + a tiny (~12-byte) location pointer in RAM while values live on flash; this is what lets you address far more data than RAM with minimal per-item overhead. Tie cold-eviction to LRU/LFU rather than database-style aging.
- Treat append-log (AOF-style) durability as an opt-in tier for users needing a bounded loss window (<=1s with everysec), implemented as a binary op-log via io_uring, not as the default — continuous fsync is a throughput-per-core cost.
- Design the tiered backend's compaction/GC up front to bound write amplification (log-structured append + drop-unread-style compaction), and expose SSD-write metrics so users can reason about endurance — a differentiator vs RocksDB-backed competitors.
- Resolve the warm-restart-vs-tiering conflict that Memcached punted on: IronCache's mmap warm-restart and SSD tiering should be designed to coexist (e.g., warm-restart only re-maps the RAM tier + reloads cold-tier metadata).

## Research papers and primary sources

- **FASTER: A Concurrent Key-Value Store with In-Place Updates** (SIGMOD 2018). HybridLog (mutable in-place / read-only RCU / stable-on-disk regions) over one logical address space, with an epoch-protection framework for latch-free synchronization and memory reclamation; ~160M ops/sec on YCSB. [source](https://www.microsoft.com/en-us/research/wp-content/uploads/2018/03/faster-sigmod18.pdf) Relevance: The canonical larger-than-RAM, latch-free hot-path design; the hybrid log + epoch protection are the core borrows for IronCache's tiered backend and shard concurrency model.
- **From FASTER to F2: Evolving Concurrent Key-Value Store Designs for Large Skewed Workloads** (VLDB 2024/2025 (arXiv 2305.01516)). Two-tier hot-log (memory) + cold-log (disk) record-oriented store with a two-level hash index and read cache; 2-11.9x throughput over KV stores and up to ~3.2x over RocksDB on skewed workloads. [source](https://arxiv.org/abs/2305.01516) Relevance: Directly addresses the cache-relevant skewed-workload tiering case and shows the hybrid-log approach beats an LSM (RocksDB) on throughput, supporting the reject-RocksDB stance.
- **Achieving High Throughput and Elasticity in a Larger-than-Memory Store (Shadowfax)** (VLDB 2021 (arXiv 2006.03206)). Scales the FASTER hybrid-log design across cores and machines with asynchronous, latch-free, zero-copy migration and epoch-based coordination for elasticity. [source](https://arxiv.org/pdf/2006.03206) Relevance: Informs IronCache's single-node-to-multi-node path: how to grow a hybrid-log cache horizontally without latches, relevant to the eventual cluster milestone.
- **Optimizing Space Amplification in RocksDB** (CIDR 2017). Quantifies the write/read/space-amplification trade-off triangle in leveled LSM compaction and the levers (size multiplier, dynamic level sizing) to tune it. [source](https://www.cidrdb.org/cidr2017/papers/p82-dong-cidr17.pdf) Relevance: Primary evidence for the LSM write-amplification cost that motivates rejecting RocksDB as IronCache's core cold engine in favor of a hybrid log.
- **Dragonfly Point-in-Time Snapshotting Design (rdbsave.md / Balanced vs Unbalanced)** (Engineering design doc + blog, 2022). Replaces fork()+COW with a versioned, asynchronous, fiber-based serialization that captures an epoch cut and pushes entry pre-images on write, yielding constant snapshot memory overhead and natural back-pressure. [source](https://www.romange.com/2022/10/30/balanced-vs-unbalanced/) Relevance: The exact mechanism IronCache should implement for snapshots; the most direct prior art for fork-less, memory-stable point-in-time persistence in a cache.

## Open questions

- Can a Dragonfly-style versioned snapshot coexist cleanly with a FASTER-style hybrid-log cold tier, given that cold-tier records already have an on-disk representation? Does snapshot need to serialize only the mutable+read-only regions plus cold-tier metadata?
- What is the right Rust io_uring substrate — glommio (thread-per-core, opinionated) vs tokio-uring (integrates with the tokio ecosystem) — and what is the fallback for kernels <5.x and macOS dev environments?
- How should cold-tier migration policy be driven: pure recency (FASTER aging) or cache eviction signals (LRU/LFU/W-TinyLFU)? Does coupling tiering to the eviction policy simplify or complicate the design?
- What write-amplification budget is acceptable on SSD for a high-churn cache, and what compaction/GC scheme (log append + drop-unread) keeps it well below RocksDB's >10x while bounding space amplification?
- Should IronCache offer an AOF-style binary op-log at all, or is fork-less snapshot + warm-restart sufficient for the target users, leaving strict durability out of scope as a non-goal?
- For warm restart, is mmap-on-tmpfs portable enough, or should IronCache target DAX/persistent-memory explicitly — and how does this interact with the single-static-binary, run-anywhere promise?
- What exactly is F2's measured write amplification and its read-cache hit-rate behavior on skewed workloads (needs verification against the paper's results tables, since the fetched ~3.2x-vs-RocksDB figure was model-extracted)?
- Does maintaining a per-entry version counter for snapshotting impose unacceptable per-item memory overhead at IronCache's minimal-footprint target, and can it be packed into existing metadata words?
