## Summary

DragonflyDB (current release v1.39.0, 2026-06-09) is the strongest modern "efficiency per core" claim in the cache space and the most direct architectural competitor for IronCache. Its thesis: a single multi-threaded process can replace a Redis Cluster by scaling vertically with a shared-nothing, thread-per-core design over the helio framework on io_uring (Linux 5.11+, epoll fallback). The keyspace is sharded into N parts (N <= thread count); each shard is owned and accessed by exactly one thread, and threads communicate only by message passing — never mutexes/spinlocks on the hot path. Async concurrency is built on userspace fibers (Boost.Fibers + helio fb2), and the connection fiber doubles as the transaction coordinator. Headline marketing claims: "25X more throughput" vs Redis and "up to 80% less resources"; the README's own AWS numbers show ~3.8M QPS on c6gn.16xlarge (crossing 3.8M / "25X" vs single Redis), 10M QPS SET and 15M QPS GET at --pipeline=30, and a separate blog shows 6.43M ops/sec on a 64-vCPU c7gn.16xlarge (P50 0.3ms, P99 1.1ms). Notably, on a single-core m5.large, Dragonfly is only ~comparable to Redis (173K vs 159K SET QPS) — the win is purely vertical scaling, which is the single most important framing point for IronCache.

The two load-bearing data-structure innovations are Dashtable and forkless snapshotting. Dashtable is an implementation of the Dash paper (arXiv 2003.07302), itself an evolution of 1979 extendible hashing: a directory of pointers to fixed-size segments, each segment a flat open-addressed mini-hashtable. In Dragonfly's compile-time config a segment has 60 buckets (56 regular + 4 stash), 14 slots/bucket, 840 records/segment. Growth is incremental per-segment split (constant allocation), avoiding Redis dict's 2N bucket-array reallocation and its ~48N-byte peak during incremental rehash. The doc's own arithmetic: Redis dict overhead is 16-32 bytes/item (dictEntry is 24 bytes + 8-byte bucket pointer); Dashtable is ~19N bytes at 100% util and 22-32 bytes/item in practice, i.e. 6-16 bytes overhead, with directory cost effectively vanishing (~9.6KB for 1M items, fits in L1). On debug-populate of 20M small items, Dragonfly used 1GB vs Redis 6's 1.73GB (1.0GB of which was metadata). This is the clearest "metadata tax" argument in the literature and directly relevant to IronCache's "minimal memory" goal.

Forkless snapshotting (rdbsave.md) is Dragonfly's standout availability feature and the reason the dashtable metadata investment pays off twice. Instead of Redis's fork()+copy-on-write BGSAVE (which can spike memory toward 2-3x under write load), each shard runs a serialization fiber over its own Dashtable while a per-shard monotonic epoch/version counter establishes a virtual "cut." The serialization loop sends any entry with version <= cut.epoch and bumps its version; an OnWriteHook intercepts concurrent mutations and, for conservative (point-in-time/backup) snapshots, pushes the OLD value to the sink before overwriting — guaranteeing snapshot isolation with zero fork and constant memory overhead regardless of dataset size. A "relaxed" variant (for replication) sends new values / incremental diffs instead, avoiding a separate changelog. README benchmark: on ~5GB, Dragonfly was 30% more memory-efficient idle and showed no visible spike during bgsave, while Redis peaked at almost 3X Dragonfly's memory. Snapshot isolation is enforced at Dash *bucket* granularity (blobs never partially cover a bucket).

Atomic multi-key/multi-shard transactions use VLL (Very Lightweight Locking, Ren/Thomson/Abadi, VLDBJ). VLL replaces a heavyweight lock manager with two per-record integers and a global transaction queue, using logical ordering rather than lock objects. Dragonfly adapts this: a single global atomic counter assigns monotonically increasing txids; each shard keeps an ordered transaction queue; multi-shard commands are split into per-shard subcommands that "hop" to the owning threads, execute, and aggregate. Because shards are single-threaded, the common single-shard case needs no lock at all (the thread simply owns the data); multi-shard coordination is deadlock-free by txid ordering. The allocator is mimalloc v2.2.4 (vendored with 5 Dragonfly patches adding allocation stats). Protocol surface: RESP (Redis ~185 commands, ~Redis 5.0 API) plus full Memcached text protocol except cas; HTTP and Prometheus /metrics share the main TCP port via protocol auto-detection.

For IronCache the takeaways are sharp. Borrow wholesale: shared-nothing thread-per-core, io_uring (Rust has glommio/monoio/tokio-uring), single-thread-owns-shard to eliminate hot-path locks, and the forkless versioned-snapshot design (a fork-based COW approach is a non-starter for a Rust single-binary that wants bounded memory). Adapt: Dashtable's segmented extendible hashing is the right model, but IronCache can revisit segment geometry, SIMD fingerprint probing, and integrate compression-awareness into segment metadata; VLL-style txn coordination should be kept but IronCache must decide early whether to expose MULTI/EXEC and Lua at all. Reject for now: Dragonfly's emulated-only cluster mode and its C++/Boost.Fibers stack (Rust async + a uring runtime is the idiomatic equivalent). The single most important strategic lesson: Dragonfly's efficiency story is vertical scaling + metadata reduction + forkless persistence, NOT raw single-core speed — IronCache's "max throughput per core" goal is a genuinely different and more aggressive target than Dragonfly's, and must be benchmarked single-core, not just at 64 vCPUs.

## Mechanisms: borrow, adapt, or reject

| Mechanism | System | Stance | What it does | Rationale for IronCache |
| --- | --- | --- | --- | --- |
| Shared-nothing thread-per-core sharding | DragonflyDB / helio | **borrow** | Partitions the keyspace into N shards (N <= thread count); each shard is owned and accessed by exactly one thread. Threads never share a data structure; they communicate only via message passing. Key routing is k = HASH(KEY) % N. | This is the canonical way to hit 'max throughput per core' with no hot-path locks and is exactly IronCache's target architecture. Rust enforces this even better than C++ via ownership (a shard struct moved into a per-core task is statically un-shareable). |
| io_uring async I/O with epoll fallback | helio | **borrow** | All network and disk I/O is submitted through io_uring (Linux 5.11+), including registered buffers and multishot receives; falls back to epoll on older kernels. | Batched syscall submission/completion is the lowest-overhead I/O path on Linux. Rust has mature equivalents (monoio, glommio, tokio-uring). Must keep an epoll/kqueue fallback so the single binary still runs on macOS dev boxes and pre-5.11 kernels. |
| Userspace fibers (Boost.Fibers + helio fb2) | DragonflyDB | **adapt** | Concurrency is expressed as cheap stackful fibers scheduled per thread; the connection fiber also acts as the transaction coordinator. std::thread/std::mutex are banned to avoid deadlocks and contention. | The model (cooperative per-core scheduling) is right, but IronCache should use Rust async tasks (stackless, no per-fiber stack memory) on a thread-per-core runtime rather than stackful fibers — better memory footprint, which matters for the 'minimal memory' goal. |
| Dashtable (segmented extendible hashing) | DragonflyDB | **borrow** | Directory of pointers to fixed-size segments; each segment is a flat open-addressed mini-hashtable (56 regular + 4 stash buckets, 14 slots each, 840 records). Growth is per-segment split (constant allocation), not full 2N rehash. Open addressing + small fingerprints avoid per-entry dictEntry, cutting overhead to 6-16 bytes/item vs Redis's 16-32. | Directly serves IronCache's minimal-memory + low-tail-latency goals: no rehash latency spikes, L1-resident directory, stateless scan, and metadata small enough to hang extra bits (versioning, cache rank) off each entry. This is the single most copyable idea. |
| Stash buckets + neighbor probing | DragonflyDB Dashtable | **adapt** | Each segment reserves 4 stash buckets to absorb spillover from the 56 regular buckets; an item probes its home bucket, the right neighbor, then stashes, before declaring the segment full and splitting. | Raises load factor before splitting (less memory waste), but the exact 56/4/14 geometry should be re-tuned for IronCache's cache lines and SIMD width; consider AVX/NEON fingerprint scan over a bucket's 14 slots in one vector op. |
| VLL transaction coordination | DragonflyDB | **adapt** | Global atomic counter assigns monotonic txids; each shard has an ordered transaction queue; multi-shard commands split into per-shard subcommands that hop to owning threads, run, and aggregate. Single-shard ops need no lock (thread owns data); multi-shard is deadlock-free by txid order. | Keep the lock-free single-shard fast path and txid-ordered multi-shard hop. But IronCache should decide whether to support MULTI/EXEC/WATCH and Lua at all in M1 — full transactional generality is a large cost the 'most efficient cache' may not need. |
| Forkless versioned snapshot | DragonflyDB | **borrow** | Per-shard monotonic epoch sets a cut; a serialization fiber emits entries with version <= cut and bumps their version; an OnWriteHook pushes the OLD value to the sink before any concurrent mutation (conservative) or the new value/diff (relaxed). No fork(), constant memory overhead, bucket-granularity isolation. | Redis fork()+COW can spike memory ~3X under write load — fatal for a Rust single binary targeting bounded memory. The versioned-cut design gives point-in-time RDB-compatible snapshots with O(1) extra memory and is enabled precisely because Dashtable has spare metadata bits. |
| mimalloc allocator | DragonflyDB | **adapt** | Uses Microsoft mimalloc v2.2.4 (vendored, 5 patches for allocation stats / heap-collect fixes) as the process allocator for low fragmentation and fast thread-local alloc. | A sharded, low-fragmentation allocator is essential. In Rust, evaluate mimalloc-rust vs jemallocator vs a custom slab; the choice interacts with per-shard arenas and with returning freed pages to the OS (RSS accuracy matters for the memory claim). |
| RESP + Memcached + HTTP on one port | DragonflyDB | **adapt** | Auto-detects protocol on the main TCP port (6379): RESP (~185 Redis cmds, ~Redis 5 API), Memcached text protocol (all but cas), and an HTTP console + Prometheus /metrics. | Redis-wire compat is a hard IronCache requirement; sharing the port with HTTP/metrics is ergonomic for a single-binary story. IronCache should target a newer Redis API baseline (RESP3, Redis 7 commands) and decide whether Memcached compat is in scope (likely a non-goal initially). |
| Unified adaptive cache eviction (cache_mode) | DragonflyDB | **adapt** | A single adaptive eviction policy (claimed higher hit rate than LRU/LFU with zero memory overhead) that only evicts near maxmemory, with eviction metadata folded into Dashtable entries. | 'Zero memory overhead' eviction by piggybacking on hashtable metadata is exactly the kind of efficiency IronCache wants, but the claim is unquantified — IronCache should design its own policy (e.g., TinyLFU/W-TinyLFU/S3-FIFO) and benchmark hit-rate honestly rather than copy an opaque heuristic. |
| Emulated-only cluster mode | DragonflyDB | **reject** | Per the README, cluster_mode supports only 'emulated' — the cluster command API on a single process — rather than true multi-node sharding (which has since evolved in newer releases). | IronCache's stated path is single-node to multi-node as a real distributed system; an emulated single-process cluster is a stopgap, not a model to copy. Design real multi-node from the architecture spec instead. |

## Implications for IronCache

- Adopt shared-nothing thread-per-core as the core architecture: one Rust task/runtime per physical core, each owning a disjoint shard. Use Rust ownership to make cross-shard data access a compile error, going beyond Dragonfly's convention-only enforcement.
- Benchmark single-core first. Dragonfly's own README shows near-parity with Redis at 1 vCPU (173K vs 159K SET QPS); its 25X is purely vertical scaling. IronCache's 'max throughput per core' goal demands a single-core target (e.g. beat Redis 7 and Dragonfly per-core), not just a 64-vCPU headline.
- Implement a Dashtable-equivalent: directory + fixed-size open-addressed segments with stash buckets, incremental per-segment split, and SIMD fingerprint probing across a bucket's slots. Target <=16 bytes/item overhead and reserve spare metadata bits per entry for versioning + eviction rank from day one.
- Make snapshotting forkless and versioned, never fork()+COW. Use a per-shard epoch cut + write-hook that emits the pre-image (or diff) of concurrently-mutated entries, giving point-in-time snapshots with O(1) extra memory — this is what lets IronCache claim bounded memory under write load.
- Use io_uring with an epoll/kqueue fallback so the single static binary runs on Linux 5.11+ at peak speed but still starts on older kernels and macOS dev machines. Pick the runtime (monoio/glommio vs custom) as an explicit M1 decision.
- Prefer Rust stackless async tasks over stackful fibers to cut per-connection/per-coroutine memory — directly serving the minimal-memory goal where Dragonfly's Boost.Fibers stacks cost more.
- Decide the transaction surface early. Keep VLL's lock-free single-shard fast path and txid-ordered multi-shard hop, but treat full MULTI/EXEC/WATCH/Lua as a scoped decision — a pure cache may ship a smaller transactional surface to stay lean.
- Treat Memcached protocol and emulated cluster as likely non-goals; commit instead to RESP3 / modern Redis API compatibility and a real (not emulated) single-node-to-multi-node story.
- Choose the allocator deliberately (mimalloc-rust vs jemallocator vs per-shard slab arenas) and measure RSS return-to-OS, since the 30%+ idle memory-efficiency claim lives or dies on fragmentation and page reclaim, not just per-entry overhead.
- Design an honest, quantified eviction policy (S3-FIFO / W-TinyLFU) with metadata folded into hashtable entries ('zero overhead'), and publish hit-rate numbers — avoid Dragonfly's unquantified 'better than LRU/LFU' claim.

## Research papers and primary sources

- **Dash: Scalable Hashing on Persistent Memory** (VLDB 2020 (arXiv 2003.07302)). Extendible-hashing variant with fixed-size segments, fingerprints, stash/bucket balancing, and optimistic concurrency designed for persistent memory; achieves high load factor with cheap incremental resize and stateless scan. [source](https://arxiv.org/abs/2003.07302) Relevance: Direct basis for Dragonfly's Dashtable and the strongest template for IronCache's in-memory hash index; the fingerprint + segment-split mechanics translate directly to a SIMD-probed Rust implementation.
- **VLL: a lock manager redesign for main memory database systems** (VLDB Journal (Ren, Thomson, Abadi)). Replaces a heavyweight lock manager with two per-record integers (CX/CS) plus a global transaction queue, using logical ordering rather than lock objects; separate contention-free and contention execution paths. [source](https://www.cs.umd.edu/~abadi/papers/vldbj-vll.pdf) Relevance: The transactional foundation Dragonfly cites; informs how IronCache can offer atomic multi-key ops on a shared-nothing layout without mutexes, and where to draw the line on transactional generality.
- **Extendible Hashing - A Fast Access Method for Dynamic Files** (ACM TODS 1979 (Fagin, Nievergelt, Pippenger, Strong)). Original directory-of-buckets scheme that grows incrementally by splitting buckets and doubling the directory, avoiding full-table rehash. [source](https://dl.acm.org/doi/10.1145/320083.320092) Relevance: The 47-year-old root idea behind Dashtable; understanding it clarifies which incremental-growth and directory-doubling tradeoffs IronCache inherits and can improve.
- **S3-FIFO: FIFO queues are all you need for cache eviction** (SOSP 2023 (Yang, Qiu, et al.)). A simple three-FIFO eviction scheme that matches or beats LRU/LFU hit rates with O(1) operations and tiny metadata, scaling well under concurrency. [source](https://dl.acm.org/doi/10.1145/3600006.3613147) Relevance: A concrete, published, quantified alternative to Dragonfly's opaque adaptive cache_mode; a strong candidate for IronCache's 'zero-overhead' eviction policy that can be benchmarked honestly.

## Open questions

- What is Dragonfly's actual single-core throughput vs Redis 7 / Valkey on identical hardware in 2026? The README's 1-vCPU parity numbers are from m5.large and Redis 6 — IronCache needs fresh per-core data to know the real bar to beat.
- How much per-entry metadata does Dashtable actually reserve today (versioning + eviction + expiry bits), and does it still hit the doc's ~19N bytes once all Dragonfly features are enabled? The 6-16 byte overhead may be optimistic in the shipped binary.
- Does the relaxed (replication) snapshot's incremental-diff path require an extended RDB format, and how does that interact with point-in-time correctness under heavy multi-key writes? IronCache must decide if it needs both conservative and relaxed variants.
- What concrete eviction algorithm does cache_mode implement, and is the 'higher than LRU/LFU with zero memory overhead' claim reproducible? No paper or pseudocode is published; IronCache should not copy it blind.
- Stackful fibers vs stackless Rust async: what is the real per-connection memory cost difference at, say, 1M idle connections, and does Rust's stackless model meaningfully improve the memory claim?
- How does mimalloc's page reclaim behave under churn for Dragonfly's RSS, and which Rust allocator best preserves the 'memory returns to OS' property that underpins the idle-efficiency claim?
- Is there an up-to-date third-party (non-vendor) benchmark of Dragonfly vs KeyDB vs Valkey on per-core efficiency, since most numbers (incl. KeyDB winning on SET/mixed) come from blogs of varying rigor?

## Proposed issues (seeds for the tracker)

- **[epic, M1]** EPIC: Shared-nothing thread-per-core architecture for IronCache. Define IronCache's core as one Rust runtime/task per physical core, each owning a disjoint shard with no shared mutable state, communicating via message passing. Establish this as the foundational architecture decision against which all other designs are measured.
- **[design, M2]** DESIGN: Segmented extendible-hashing index (Dashtable/Dash-style) with SIMD fingerprint probing. Specify a directory + fixed-size open-addressed segment hash index with stash buckets and incremental per-segment split, targeting <=16 bytes/item overhead and reserving per-entry metadata bits for version, expiry, and eviction rank. Define segment geometry and SIMD slot scan.
- **[research, M0]** RESEARCH: Benchmark single-core throughput vs Redis 7 / Valkey / Dragonfly. Establish the real per-core performance bar. Dragonfly is only ~at parity with Redis at 1 vCPU; IronCache's 'max throughput per core' goal needs fresh, identical-hardware single-core numbers across the field before architecture is frozen.
- **[design, M2]** DESIGN: Forkless versioned point-in-time snapshotting. Adopt Dragonfly's per-shard epoch-cut + write-hook design (emit pre-image on concurrent mutation) to produce RDB-compatible snapshots with O(1) extra memory and no fork(). Specify conservative (backup) vs relaxed (replication) variants and bucket-granularity isolation.
- **[decision, M1]** DECISION: Transaction surface — VLL single-shard fast path, and whether to support MULTI/EXEC/Lua. Keep VLL's lock-free single-shard fast path and txid-ordered multi-shard hop, but decide explicitly how much Redis transactional/scripting surface (MULTI/EXEC/WATCH, Lua) IronCache commits to, since full generality is a large cost for a cache.
- **[decision, M1]** DECISION: Async runtime and io_uring strategy with epoll/kqueue fallback. Choose between monoio/glommio/tokio-uring/custom for the thread-per-core runtime, mandate io_uring on Linux 5.11+ with an epoll/kqueue fallback so the single static binary runs everywhere, and prefer stackless async tasks over stackful fibers for memory.
- **[research, M0]** RESEARCH: Eviction policy selection (S3-FIFO / W-TinyLFU) with zero-overhead metadata. Evaluate published, quantified eviction algorithms whose metadata folds into the hash index entries, instead of copying Dragonfly's opaque adaptive cache_mode. Produce honest hit-rate benchmarks vs LRU/LFU.
- **[decision, M1]** DECISION: Allocator choice and RSS return-to-OS behavior. Decide between mimalloc-rust, jemallocator, and per-shard slab arenas, measuring fragmentation and page reclaim, since IronCache's idle/under-load memory-efficiency claims depend on returning freed memory to the OS, not just per-entry overhead.
- **[non-goal, M0]** NON-GOAL: Memcached protocol and emulated single-process cluster mode. Scope out Memcached text-protocol compatibility and Dragonfly-style emulated cluster mode for the initial milestones; commit instead to RESP3 / modern Redis API and a real single-node-to-multi-node distribution design.
- **[research, M0]** RESEARCH: Audit Dragonfly headline claims (25X, 80% less resources, 6.43M RPS) for benchmark conditions. Document the exact conditions behind Dragonfly's marketing numbers (instance, pipelining, value size, version) and the single-core counter-evidence, so IronCache's own claims and comparisons are framed honestly in the README and vision EPIC.
