## Summary

KeyDB is a multithreaded fork of Redis, originally by EQ-Alpha (John Sully), acquired by Snap Inc. in May 2022 and re-homed to the Snapchat/KeyDB GitHub repo with a unified BSD-3 license. Its headline differentiator is multithreaded command execution on a SHARED keyspace (in contrast to Redis 6+/Valkey, which only multithread network I/O while keeping a single command-execution thread). KeyDB runs the normal Redis event loop on N server-threads; each accepted connection is pinned to a thread, and access to the shared dict is mediated by a custom fast ticket spinlock (src/fastlock.cpp/.asm) rather than a coarse global mutex. The lock spins with x86 PAUSE / ARM YIELD for a bounded loop count (0x100000 normally, 0x10000 under CPU pressure) before falling back to a Linux futex (FUTEX_WAIT_BITSET_PRIVATE). Marketed numbers claim ~5x ops/sec over single-threaded Redis 5 and up to ~7x with TLS, but these are vendor benchmarks on m5.4xlarge-class hardware and should be treated as directional.

For datasets larger than RAM, KeyDB FLASH uses RocksDB as a pluggable storage provider (src/storage/rocksdb.cpp, two column families: one for data, one for expires). Enabled via `--storage-provider flash <path>` with `maxmemory` + an `allkeys-lru`/`allkeys-lfu` eviction policy, it keeps hot values in RAM and evicts cold values to RocksDB while (by default) keeping every key — but not its value — in a RAM key cache; v6.3.4 added `flash-disable-key-cache` and moved the cluster-slot-to-key map and expires into the storage provider so cold data can also be expired/evicted. Vendor testing showed 200M keys * 1KB = 190GB on an m5d.2xlarge NVMe instance achieving ~85% of RAM throughput with ~0.5ms extra latency. FLASH is explicitly labelled Beta/experimental throughout the docs.

KeyDB's distribution story is Active Replication / multi-master active-active: `active-replica yes` makes a replica bidirectional, and `multi-master yes` plus multiple `replicaof` lines forms ring/full-mesh topologies. Each instance computes an ephemeral per-process UUID (not persisted) used to break replication loops. Conflict resolution is described two ways in the docs — the active-rep page claims timestamped "newest write wins," while the multi-master page admits it "defaults to last operation wins" and that with identical keys on two masters "it is undefined which key will be taken," with no automatic propagation of REPLICAOF REMOVE and a known tombstone gap on deletes. This is weaker than a true CRDT and is best characterized as approximate LWW, not a rigorous causal/MVCC merge. The MVCC claim is real but scoped: it provides non-blocking SCAN/KEYS and non-blocking BGSAVE via point-in-time snapshots (RocksDB Snapshot for FLASH; an in-memory snapshot for the dict), NOT a general multi-version concurrent-write store. Subkey expiry (EXPIREMEMBER / EXPIREMEMBERAT, with TTL/PTTL extended to members) is a genuine differentiator Redis lacks, though v6.3.4 "removed expireset and restored redis expire behaviour" for the internal data structure.

Maintenance status is the most load-bearing finding for IronCache strategy: the repo is NOT archived, but the last tagged release is v6.3.4 (2023-10-30) and the last commit to main was 2024-03-24. There has been no release in ~20 months and no commit in ~26 months as of 2026-06-13. KeyDB is based on Redis 6.x and never tracked Redis 7's functions, listpacks, RESP3 refinements, or sharded pub/sub; it has been effectively superseded by Valkey (the Linux Foundation Redis fork) as the community's multithreaded/active-active alternative. The README states Snap maintains it as internal caching infrastructure with no commercial product and no paid support. For IronCache, KeyDB is a rich source of mechanisms to study but a cautionary tale on governance: the shared-keyspace-with-fine-grained-locking model and pluggable RocksDB tiering are worth borrowing in spirit, while the bolted-on approximate-LWW replication and the single-vendor dormancy are anti-patterns to avoid.

## Mechanisms: borrow, adapt, or reject

| Mechanism | System | Stance | What it does | Rationale for IronCache |
| --- | --- | --- | --- | --- |
| Multithreaded command execution on a shared keyspace | KeyDB | **adapt** | Runs the Redis event loop on N server-threads (default 2, recommended 4); each connection is pinned to a thread at accept(), and all threads operate on one shared dict guarded by fine-grained locking. This contrasts with Redis 6+/Valkey which thread only network I/O. | IronCache's max-throughput-per-core goal demands real multicore command execution, so the shared-keyspace + per-thread event-loop direction is right. But adapt: KeyDB pins connections to threads and serializes on a per-keyspace spinlock, which caps scaling. IronCache should evaluate sharded/partitioned keyspaces (per-core hash partitions a la Seastar/memcached extstore-free sharding) so most ops are lock-free, reserving cross-shard coordination only for multi-key commands. Borrow the goal, reject the single-shared-dict-with-one-lock topology. |
| Fast ticket spinlock with futex fallback (fastlock) | KeyDB | **borrow** | A custom ticket spinlock (m_ticket.m_active/m_avail via __atomic_fetch_add) that spins with x86 PAUSE / ARM YIELD up to a loop limit (0x100000, or 0x10000 under CPU pressure) then blocks on a Linux futex (FUTEX_WAIT/WAKE_BITSET_PRIVATE). Recursive (m_depth) and tracks owner pid. | This is a well-engineered, FIFO-fair, low-latency lock primitive: ticket ordering avoids starvation, adaptive spin-then-futex avoids burning cores under contention, and PAUSE/YIELD respect SMT siblings. IronCache (Rust) can borrow the pattern directly using crossbeam/parking_lot-style adaptive spinning or a custom ticket lock over futex/Windows WaitOnAddress. The exact tuning constants are a useful starting point to benchmark against. |
| Connection-to-thread balancing (min-clients-per-thread, SO_REUSEPORT, replica-weighting-factor) | KeyDB | **adapt** | Distributes accepted connections across server-threads using min-clients-per-thread (default 50), active-client-balancing, SO_REUSEPORT kernel distribution, and replica-weighting-factor (default 2) to under-load threads hosting expensive replica clients. | Load-balancing connections across worker threads is essential and the SO_REUSEPORT + best-effort rebalancing approach is sound. IronCache should adopt SO_REUSEPORT and per-core accept queues, but in a sharded design the relevant balancing is request-to-shard (by key hash) not connection-to-thread; connection affinity matters less when work is routed by key. Adapt the mechanism to a shard-routing model. |
| RocksDB-backed FLASH tiering (storage provider) | KeyDB FLASH | **adapt** | Pluggable RocksDBStorageProvider with separate column families for data and expires; hot values stay in RAM, cold values evicted to RocksDB on disk under maxmemory + allkeys-lru/lfu; an in-RAM key cache holds every key (toggleable via flash-disable-key-cache). | A RAM-over-flash tier is highly relevant to IronCache's memory-efficiency goal, but RocksDB is a heavyweight C++ LSM dependency that conflicts with the single-static-binary ethos and adds write amplification + compaction tail latency. Adapt: design a pluggable storage-provider trait but back it with a leaner embedded engine (e.g. a Rust LSM like sled/fjall, or a custom log-structured store) and keep keys/metadata in RAM. Borrow the column-family separation of data vs expires and the keys-in-RAM/values-on-flash split. |
| MVCC snapshots for non-blocking SCAN/KEYS and BGSAVE | KeyDB | **borrow** | Maintains point-in-time snapshots (in-memory snapshot of the dict; rocksdb::Snapshot for FLASH) so KEYS/SCAN and background save can iterate a consistent view without blocking concurrent writers. | Non-blocking iteration and snapshotted persistence are exactly what a high-throughput cache needs to avoid latency cliffs on SCAN/KEYS/save. Rust's ownership model and structures like persistent/immutable trees or epoch-based reclamation (crossbeam-epoch) make copy-on-write snapshots natural. Borrow the concept; implement via COW + epoch GC rather than fork()-based COW where possible. |
| Active-active multi-master replication with ephemeral UUID loop prevention | KeyDB | **reject** | active-replica yes + multi-master yes + multiple replicaof lines build ring/mesh topologies; each node has an ephemeral per-process UUID to break replication loops; conflicts resolved by last-operation-wins. | The loop-prevention UUID is fine, but the conflict model is fundamentally weak: docs admit identical-key conflicts are 'undefined,' deletes leave tombstone gaps, and there is no causal ordering or true CRDT. For IronCache's single-node-to-multi-node path, reject KeyDB's approximate LWW and instead design from CRDT/causal foundations (e.g. per-key Lamport/HLC timestamps with explicit tombstone handling, or proper CRDT registers) so multi-master is correct by construction rather than best-effort. |
| Subkey expiry (EXPIREMEMBER / EXPIREMEMBERAT) | KeyDB | **adapt** | Sets independent TTLs on individual members of sets/hashes, with TTL/PTTL extended to query member-level expiry; active + passive expiration cycles remove expired subkeys. | A genuine capability Redis historically lacked and a real differentiator. IronCache should consider native per-element TTLs as a first-class feature (Redis 7.4 later added hash-field TTLs, so it is converging). Adapt the command surface but design the expiry index for efficiency (e.g. a per-shard timer wheel or min-heap of member expirations) rather than KeyDB's expireset approach, which it itself reverted in 6.3.4. |
| CPU-based load shedding (overload-protect-percent) and RDB-less fast sync | KeyDB | **borrow** | overload-protect-percent sheds/disconnects clients when CPU exceeds a threshold; enable-keydb-fastsync performs full replica sync without staging an RDB file on disk. | Both are pragmatic operability wins. Admission control / load shedding under CPU saturation protects tail latency and is a good fit for IronCache's efficiency story. Diskless/streamed full sync avoids the fork+RDB write amplification path. Borrow both as design requirements for the replication and overload-handling subsystems. |

## Implications for IronCache

- Pursue real multicore command execution (KeyDB's core insight) but reject its single-shared-dict-plus-one-spinlock topology; prototype a per-core sharded keyspace where single-key ops are lock-free and only multi-key/cross-shard commands coordinate, to beat KeyDB on throughput-per-core.
- Borrow KeyDB's adaptive ticket-spinlock-then-futex pattern for any unavoidable shared structure: FIFO-fair, SMT-friendly (PAUSE/YIELD), and contention-aware (futex fallback). In Rust, evaluate parking_lot vs a custom ticket lock over futex/WaitOnAddress and benchmark against KeyDB's tuning constants.
- Design FLASH-style RAM-over-disk tiering as a pluggable storage-provider trait, but back it with a lean embedded Rust engine (not RocksDB) to preserve the single-static-binary goal and avoid LSM compaction tail latency; keep all keys + metadata in RAM, values on flash, with separate data/expire indexes.
- Make non-blocking SCAN/KEYS and snapshotted background persistence a hard requirement, implemented via copy-on-write + epoch-based reclamation (crossbeam-epoch) rather than fork()-COW, to eliminate latency cliffs.
- Treat KeyDB's approximate last-write-wins multi-master as an anti-pattern: if IronCache offers active-active, build it on causal/HLC timestamps or CRDTs with explicit tombstone semantics so identical-key conflicts and deletes are well-defined, not 'undefined.'
- Adopt per-element TTLs (EXPIREMEMBER-style) as a first-class feature but back them with an efficient per-shard timer wheel / min-heap, noting KeyDB itself reverted its expireset structure in 6.3.4.
- Bake in operability primitives KeyDB added late: CPU-based admission control / load shedding and diskless streamed full replica sync, designed in from M1 rather than bolted on.
- Use KeyDB as the cautionary governance case study in the Vision EPIC: a strong single-vendor fork (Snap) that stalled (no release since v6.3.4 in Oct 2023, no commit since Mar 2024) and was overtaken by Valkey; argue IronCache needs an open, community-tracked roadmap and must track upstream Redis/Valkey wire features (RESP3, functions, hash-field TTLs) that KeyDB never adopted.

## Research papers and primary sources

- **Scaling Memcache at Facebook** (NSDI 2013). Lessons scaling a multithreaded in-memory cache (memcached) including lock granularity, lease mechanisms, and replication/consistency tradeoffs at scale. [source](https://www.usenix.org/conference/nsdi13/technical-sessions/presentation/nishtala) Relevance: Directly informs IronCache's choice between KeyDB's shared-keyspace locking and a sharded model, and how to reason about cache consistency under replication.
- **MICA: A Holistic Approach to Fast In-Memory Key-Value Storage** (NSDI 2014). Partition the keyspace across cores (exclusive read/write or concurrent read) with lossy/lossless indexes and kernel-bypass networking to maximize per-core KV throughput. [source](https://www.usenix.org/conference/nsdi14/technical-sessions/presentation/lim) Relevance: The strongest argument for IronCache's per-core sharded keyspace alternative to KeyDB's single shared dict + spinlock.
- **Memshare: a Dynamic Multi-tenant Key-value Cache** (USENIX ATC 2017). Log-structured memory management and dynamic memory partitioning to improve hit-rate and memory efficiency across tenants. [source](https://www.usenix.org/conference/atc17/technical-sessions/presentation/cidon) Relevance: Feeds IronCache's memory-efficiency and eviction design, complementary to KeyDB's RocksDB FLASH tiering approach.
- **WiscKey: Separating Keys from Values in SSD-conscious Storage** (FAST 2016). Separate keys (in the LSM tree) from values (in a separate value log) to drastically cut write amplification on SSD. [source](https://www.usenix.org/conference/fast16/technical-sessions/presentation/lu) Relevance: Directly relevant to designing a leaner alternative to KeyDB's RocksDB FLASH tier with keys-in-RAM/values-on-flash.
- **Conflict-free Replicated Data Types (CRDTs)** (SSS 2011). Data types whose concurrent updates always converge without coordination, providing principled strong eventual consistency. [source](https://hal.inria.fr/inria-00609399/document) Relevance: The correct foundation for IronCache active-active, replacing KeyDB's admittedly 'undefined' last-write-wins multi-master.

## Open questions

- Does KeyDB's per-keyspace spinlock actually scale linearly past ~4 server-threads on modern many-core CPUs, or does the shared dict become the bottleneck? Need independent benchmarks since vendor numbers use only ~7 threads.
- How much RAM does KeyDB's FLASH key cache consume per key (keys-in-RAM model), and what is the practical RAM:flash ratio ceiling before hit-rate collapses? Docs are vague on the numeric ratio.
- What is RocksDB's compaction-induced tail-latency profile under sustained write-heavy FLASH workloads, and can a Rust LSM (fjall/sled) or custom log-structured store match throughput with smaller binary size and fewer dependencies?
- Exactly how does KeyDB reconcile concurrent writes to the same key across multiple server-threads vs across masters — is intra-node ordering total under the spinlock while inter-node is only LWW, and where are the consistency gaps?
- Is KeyDB effectively end-of-life? Snap maintains it internally but with no public release since Oct 2023 — should IronCache cite it as historical prior art rather than a living competitor, and instead benchmark against Valkey's multithreading work?
- Does KeyDB's MVCC snapshot for BGSAVE avoid fork() entirely (true in-process COW) or still rely on copy-on-write fork semantics, and what is the memory overcommit behavior (v6.3.4 fixed an overcommit-during-fork bug)?

## Proposed issues (seeds for the tracker)

- **[decision, M1]** Decide keyspace concurrency model: shared dict + fine-grained lock (KeyDB) vs per-core sharded keyspace (MICA). Choose between KeyDB-style shared keyspace with adaptive locking and a per-core sharded keyspace that keeps single-key ops lock-free. This decision is foundational to the throughput-per-core goal.
- **[research, M0]** Research: independent benchmark of KeyDB shared-keyspace scaling vs core count. Reproduce KeyDB's multithreading benchmarks on a modern many-core box to find where the per-keyspace spinlock stops scaling, so IronCache can target a higher ceiling.
- **[design, M2]** Design: adaptive ticket-spinlock-then-futex primitive in Rust. Borrow KeyDB's fastlock pattern (FIFO ticket, PAUSE/YIELD spin, futex/WaitOnAddress fallback) for any unavoidable shared structure; evaluate parking_lot vs a custom implementation and pin tuning constants.
- **[design, M2]** Design: pluggable RAM-over-flash storage provider without RocksDB. Specify a storage-provider trait for tiering cold values to disk while keeping keys/metadata in RAM, backed by a lean embedded Rust engine to preserve the single-static-binary goal (vs KeyDB's RocksDB dependency).
- **[decision, M1]** Decision: reject last-write-wins multi-master; adopt CRDT/HLC foundation for active-active. KeyDB's multi-master is admittedly 'undefined' on identical-key conflicts and leaks tombstones on deletes. Decide on a principled causal/HLC or CRDT model for IronCache's single-to-multi-node path.
- **[design, M2]** Design: non-blocking SCAN/KEYS and snapshotted persistence via COW + epoch GC. Borrow KeyDB's MVCC-snapshot idea for non-blocking iteration and background save, implemented with copy-on-write structures and crossbeam-epoch reclamation instead of fork()-COW.
- **[design, M2]** Design: native per-element TTLs (EXPIREMEMBER) with an efficient expiry index. Offer per-member TTLs on collections like KeyDB's EXPIREMEMBER (and Redis 7.4 hash-field TTLs), backed by a per-shard timer wheel/min-heap rather than KeyDB's reverted expireset.
- **[non-goal, M0]** Non-goal: tracking KeyDB feature parity; track Valkey/Redis wire features instead. Declare KeyDB historical prior art (dormant since 2024, Redis-6-based) and a non-goal for parity; instead track RESP3, functions, and hash-field TTLs from upstream Redis/Valkey.
- **[task, M0]** Task: capture KeyDB governance case study in the Vision EPIC. Document KeyDB's trajectory (EQ-Alpha to Snap acquisition to stall after v6.3.4, overtaken by Valkey) as the cautionary argument for IronCache's open, community-tracked roadmap.
- **[design, M1]** Design: CPU-based admission control and diskless streamed full sync. Bake in KeyDB's late additions overload-protect-percent (load shedding under CPU saturation) and enable-keydb-fastsync (RDB-less full sync) as first-class requirements from the architecture spec.
