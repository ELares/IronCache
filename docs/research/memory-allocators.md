## Summary

The state of the art in cache-server memory allocation is dominated by three thread-caching, size-class allocators — jemalloc (Redis/Valkey, FreeBSD, formerly Rust std), mimalloc (Dragonfly, .NET), and snmalloc (research/Microsoft) — plus Google's tcmalloc. All four share a common skeleton: per-thread/per-CPU caches to make the fast path lock-free, fixed size classes to bound internal fragmentation, and large OS allocations (chunks/arenas/segments) carved into slabs to bound external fragmentation. The decisive differences for a cache are (a) how aggressively dirty pages are returned to the OS (RSS minimization), (b) the per-object metadata overhead, and (c) whether the allocator can be introspected to drive online defragmentation. jemalloc 5.x wins on (a) and (c): its dirty/muzzy page decay (opt.dirty_decay_ms default 10000ms; muzzy disabled) and its background_thread purger keep RSS close to live bytes, and Redis ships a forked jemalloc 5.3.0 that exposes je_get_defrag_hint() (guarded by JEMALLOC_FRAG_HINT, built with --with-lg-quantum=3 for 8-byte spacing) so the server can ask "is the run holding this pointer sparse?" and relocate live objects during active defrag.

For IronCache the central tension is "max throughput per core, minimal memory" — these pull in opposite directions. mimalloc optimizes throughput and tail latency via extreme free-list sharding (one free list per ~64KiB page, plus separate thread-local and concurrent/atomic free lists so cross-thread frees are a single CAS) and Dragonfly chose it precisely for its predictable performance and clean segment layout; Microsoft's own numbers show mimalloc ~13% faster than jemalloc on Lean and 2.5x on sh6bench. But Redis/Valkey stay on jemalloc chiefly because of the defrag hint and proven RSS-decay behavior under churny, long-lived, eviction-driven workloads — exactly IronCache's domain. snmalloc's message-passing remote-free design (objects deallocated by a non-owning thread are batched into 2^6 buckets and shipped back lock-free; just 64 bits of metadata per 64KiB slab) is the most memory-efficient metadata layout published, and is attractive if IronCache adopts a thread-per-core / shared-nothing shard model where cross-shard frees are common.

The fragmentation accounting that matters is RSS vs used_memory: Redis defines mem_fragmentation_ratio = RSS / used_memory, and warns that the OS does not reclaim freed heap automatically, so a server that fills to 5GB then frees 2GB still shows ~5GB RSS while reporting ~3GB used. Ratios slightly above 1.0 are normal allocator overhead; high ratios after a peak-then-free indicate external fragmentation that active-defrag is designed to fix (it relocates objects within jemalloc runs using the hint, defaults: activedefrag no, threshold-lower 10%, threshold-upper 100%, ignore-bytes 100MB, cycle-min 1% CPU, cycle-max 25% CPU). Ratios below 1.0 mean the OS has swapped — a separate pathology.

Transparent Huge Pages are a footgun for any forking, COW-heavy server: Redis explicitly recommends `echo never > /sys/kernel/mm/transparent_hugepage/enabled` because after fork() for RDB/AOF, touching a few thousand 2MiB THP pages copies almost the whole heap, causing large latency spikes and up to ~2x memory. jemalloc's own opt.thp default is "default" (no change) and opt.metadata_thp is "disabled", reflecting the same caution. IronCache, as a single static binary, should ship a hardened default: pick one allocator, set decay/purge defaults that favor low RSS, disable THP for the heap (or at minimum for the fork path), and surface a fragmentation_ratio metric.

Per-object overhead minimization is the other half of "minimal memory." Redis attacks this above the allocator with compact encodings (listpack/intset, with je_malloc_usable_size-aware sizing) and shared small-integer objects, and the --with-lg-quantum=3 build gives 8-byte size-class granularity so a 24-byte object isn't rounded to 32. IronCache should combine an inline-small-string / pointer-tagging value representation (avoid a heap allocation and a 16-byte header for values <= ~14-22 bytes) with an 8-byte-granularity allocator and usable-size-aware capacity, so that the dominant small-KV case costs one allocation with near-zero slack. The Rust ecosystem makes all four allocators available as #[global_allocator] drop-ins: tikv-jemallocator 0.7.0 (bundling jemalloc 5.3.1), mimalloc 0.1.52, and snmalloc-rs 0.7.4 — but only jemalloc currently exposes the defrag hint and the rich mallctl introspection IronCache needs for accounting and online defrag.

## Mechanisms: borrow, adapt, or reject

| Mechanism | System | Stance | What it does | Rationale for IronCache |
| --- | --- | --- | --- | --- |
| Dirty/muzzy page decay purging (madvise-based RSS reclamation) | jemalloc 5.3.1 | **borrow** | Tracks freed-but-still-mapped pages as dirty, then over a decay window (dirty_decay_ms default 10s) progressively returns them to the OS via madvise(MADV_DONTNEED) (or an intermediate MADV_FREE muzzy state). This is the primary lever keeping RSS close to live bytes after deletions/evictions. | Exactly the mechanism a minimal-memory cache needs: it bounds RSS after eviction storms without per-free syscall overhead. IronCache should expose decay_ms as a tunable and default it aggressively (e.g. lower than 10s) since cache workloads churn hard; pair with a background purge thread so purging stays off the request path. |
| je_get_defrag_hint() + active defragmentation | Redis 8.8.0 (forked jemalloc 5.3.0) / src/defrag.c | **borrow** | The allocator answers whether the run/slab backing a given pointer is sparsely used; Redis scans the keyspace and reallocates+copies objects flagged as living in sparse runs, letting jemalloc free the now-empty runs. Cycle is CPU-throttled (1%-25%) and gated on fragmentation thresholds (10%-100%) and a 100MB ignore floor. | External fragmentation is the dominant RSS bloat source for long-lived caches with mixed object sizes; without online defrag the only fix is a restart. IronCache must ship this. Since upstream jemalloc does not expose the hint by default, IronCache should either carry the JEMALLOC_FRAG_HINT patch on tikv-jemalloc-sys or design a slab allocator that exposes an equivalent is-this-slab-sparse query natively. |
| Per-thread cache (tcache) with bounded size classes | jemalloc 5.3.1 / tcmalloc | **adapt** | Each thread keeps a small magazine of recently-freed blocks per size class (jemalloc tcache_max default 32KiB), serving alloc/free without touching the arena lock; only spillover hits the shared arena. | Borrow the lock-free fast path, but in a thread-per-core / shard-per-core IronCache the thread cache becomes a per-shard cache with no cross-thread contention at all. Cap tcache size tightly because idle per-thread caches inflate RSS — a real cost when minimizing memory across many cores. |
| Free-list sharding (per-page local + concurrent free lists) | mimalloc v3.3.2 | **adapt** | One free list per ~64KiB page instead of per size class, with a separate atomic free list so non-owning threads free via a single CAS. Improves locality, shrinks the fast path, and naturally batches cross-thread frees. | The locality and single-CAS remote-free are excellent for throughput-per-core. IronCache should adopt page-local free lists in any custom slab allocator, but a strict shard-per-core model can avoid the concurrent free list entirely (or use it only for cross-shard migration), trimming metadata. |
| Message-passing batched remote deallocation | snmalloc 0.7.4 / ISMM 2019 | **adapt** | Frees performed by a non-owning thread are batched into a fixed 2^6-bucket array keyed by the destination allocator address and shipped back in bulk without locks, with only 64 bits of metadata per 64KiB slab. | If IronCache uses a shared-nothing shard model, cross-shard frees (e.g. a value migrated during rebalancing, or a connection thread freeing a shard-owned object) are the producer/consumer pattern snmalloc targets. Borrow the batched-return idea; the ultra-low 64-bit/slab metadata is also a direct win for minimal per-object overhead. Adapt rather than borrow wholesale because IronCache likely controls thread topology and can often avoid remote frees by design. |
| Transparent Huge Page avoidance for the heap/fork path | Redis / jemalloc | **borrow** | Redis instructs operators to set THP to never; jemalloc keeps opt.thp=default and opt.metadata_thp=disabled. THP causes COW amplification after fork() (2MiB granularity) and unpredictable RSS. | Any forking persistence path (snapshotting) makes THP a latency and memory hazard. IronCache should either avoid fork-based snapshots (data-structure-level COW, or io_uring async serialization) or madvise(MADV_NOHUGEPAGE) the heap. As a single static binary it can also detect and warn when system THP=always at startup, like Redis. |
| Background purge thread | Redis (jemalloc-bg-thread=yes) / jemalloc 5.3.1 (default off) | **borrow** | Offloads dirty/muzzy page purging (madvise) to dedicated jemalloc worker threads so the application thread never blocks on reclamation syscalls. | Redis flips the upstream default to ON precisely because synchronous purging on the request path causes latency spikes. IronCache, optimizing throughput-per-core, must keep purging off the hot path — enable a background purger (or a dedicated reclaim core) by default. |
| Fine-grained size classes (--with-lg-quantum=3) | Redis (forked jemalloc) | **borrow** | Builds jemalloc with 8-byte quantum instead of 16-byte, adding intermediate small size classes so e.g. a 24-byte object rounds to 24 not 32, cutting internal fragmentation/slop on small allocations. | Small KV entries dominate cache memory; tighter size classes directly reduce per-object slop. IronCache should use 8-byte-granularity size classes for small objects (and be usable-size-aware when sizing buffers). Low cost, direct memory win. |
| maxmemory accounting + sampled-LRU/LFU eviction | Redis 8.8.0 | **adapt** | Tracks used_memory against maxmemory and, when exceeded under an eviction policy (allkeys-lru/lfu etc.), evicts approximate-LRU/LFU keys sampled with maxmemory-samples (default 5). Default policy is noeviction (errors on write). | maxmemory must be measured against actual allocator-attributed bytes (via mallctl/malloc_usable_size), not naive object sizes, or accounting drifts from RSS. IronCache should account using allocator introspection and choose a more cache-appropriate default eviction policy than noeviction (e.g. allkeys-lru) given its target use case — a decision worth an explicit issue. |
| Inline small values / pointer tagging (shared integers, embedded strings) | Redis (shared int objects, embstr/listpack) — generalized | **borrow** | Avoids a separate heap allocation and object header for small values by inlining them into the key/entry slot or sharing immutable small-integer objects, eliminating both allocator overhead and pointer-chasing. | The single biggest per-object win for a minimal-footprint cache: a small string/integer value should cost zero extra allocations and no 16-byte header. IronCache should design its value representation with NaN-boxing / pointer tagging / inline small strings (e.g. SSO up to ~22 bytes) from day one, since retrofitting the value layout later is expensive. |

## Implications for IronCache

- Default to jemalloc via tikv-jemallocator 0.7.0 (jemalloc 5.3.1) as the #[global_allocator] for the M2 prototype, because it is the only mainstream Rust allocator that exposes both rich mallctl introspection (for memory accounting) and the je_get_defrag_hint() path (for online defrag) that a long-lived cache needs.
- Carry (or upstream) the JEMALLOC_FRAG_HINT patch on tikv-jemalloc-sys so IronCache can implement active defragmentation; if that proves unmaintainable, scope a native slab allocator that exposes an equivalent is-this-slab-sparse query — treat this as a fork-vs-build decision.
- Ship aggressive RSS-favoring defaults: enable jemalloc background_thread (upstream default is OFF), lower dirty_decay_ms below 10s for churny eviction workloads, and expose decay/purge as tunables with a documented latency/RSS trade-off.
- Expose a mem_fragmentation_ratio (RSS/used_memory) metric and a peak-tracking variant, with a Redis-compatible MEMORY DOCTOR-style report, so operators can see external fragmentation and whether active-defrag is needed.
- Design the value representation up front for minimal per-object overhead: inline small strings (SSO), pointer tagging / NaN-boxing for small integers and floats, and a shared-immutable-small-integer cache; combine with 8-byte size-class granularity and malloc_usable_size-aware buffer sizing.
- Decide THP/fork stance explicitly: prefer non-fork snapshotting (async io_uring serialization or data-structure-level COW) to avoid THP COW amplification and the ~2x memory spike; if fork is used, madvise(MADV_NOHUGEPAGE) the heap and warn at startup when system THP=always (mirroring Redis).
- If IronCache adopts thread-per-core / shard-per-core, evaluate snmalloc-style batched remote-free and its 64-bit-per-slab metadata to minimize both contention and metadata overhead; otherwise keep per-shard caches and avoid cross-thread frees by construction.
- Benchmark all three Rust allocators (tikv-jemallocator 0.7.0, mimalloc 0.1.52, snmalloc-rs 0.7.4) under a realistic cache workload (mixed small KV, eviction churn, fragmentation after peak-then-free) measuring throughput/core, p99.9 latency, and steady-state RSS — vendor benchmarks (e.g. mimalloc Lean/sh6bench) are not representative.

## Research papers and primary sources

- **Mimalloc: Free List Sharding in Action** (APLAS 2019). Extreme free-list sharding — one free list per ~64KiB page plus separate thread-local and atomic/concurrent free lists — yields a tiny, branch-light fast path and single-CAS cross-thread frees. [source](https://www.microsoft.com/en-us/research/wp-content/uploads/2019/06/mimalloc-tr-v1.pdf) Relevance: Directly informs IronCache slab/page free-list layout for maximizing throughput-per-core with good locality.
- **snmalloc: A Message Passing Allocator** (ISMM 2019). Batched, lock-free message-passing return of remotely-freed objects to their owning allocator, with only 64 bits of metadata per 64KiB slab. [source](https://dl.acm.org/doi/10.1145/3315573.3329980) Relevance: Blueprint for cross-shard deallocation and ultra-low metadata overhead in a shared-nothing IronCache shard model.
- **A Scalable Concurrent malloc(3) Implementation for FreeBSD (jemalloc)** (BSDCan 2006 (Jason Evans)). Per-arena allocation with thread caches and size-class binning to reduce lock contention and fragmentation; the design lineage behind today jemalloc 5.x. [source](https://www.bsdcan.org/2006/papers/jemalloc.pdf) Relevance: Foundational rationale for the arena/tcache/size-class model IronCache will inherit if it uses jemalloc, and for tuning narenas vs RSS.
- **Mesh: Compacting Memory Management for C/C++ Applications** (PLDI 2019). Compacts the heap without moving objects in the virtual address space by remapping physically-backed pages whose live objects do not overlap (virtual-aliasing meshing), reducing RSS without changing pointers. [source](https://dl.acm.org/doi/10.1145/3314221.3314582) Relevance: An alternative to pointer-rewriting active-defrag: IronCache could reclaim fragmented physical pages without the je_get_defrag_hint object-copy machinery, avoiding pointer invalidation.
- **Beyond malloc efficiency to fleet efficiency: a hugepage-aware memory allocator (TCMalloc/Temeraire)** (OSDI 2021). Hugepage-aware allocation (Temeraire) that packs allocations to keep hugepages either fully used or fully empty, improving CPU via fewer TLB misses while controlling RSS at fleet scale. [source](https://www.usenix.org/conference/osdi21/presentation/hunter) Relevance: Counterpoint to THP is always bad: shows how a hugepage-aware allocator can get TLB benefits without COW bloat — relevant if IronCache wants huge-page throughput gains while bounding fragmentation.

## Open questions

- Does maintaining a JEMALLOC_FRAG_HINT patch on tikv-jemalloc-sys across jemalloc upstream bumps cost more than building a native slab allocator with a first-class sparseness query? (fork-vs-build)
- What dirty_decay_ms / purge-delay default minimizes steady-state RSS for cache churn without inducing latency spikes from madvise/page-fault storms? Needs empirical sweep on a representative workload.
- For a thread-per-core IronCache, can cross-shard frees be eliminated by construction (so neither mimalloc concurrent free list nor snmalloc message passing is needed), or are connection-thread-to-shard frees unavoidable?
- Can Mesh-style physical-page compaction (no pointer rewriting) replace or complement object-copying active-defrag, given IronCache controls its own data structures?
- How should maxmemory be accounted to track RSS closely — sum of malloc_usable_size, jemalloc stats.allocated via mallctl, or a hybrid — and what is the accounting error under heavy fragmentation?
- Is there a measurable throughput win from hugepage-aware allocation (Temeraire-style) that justifies the COW/fork complexity for IronCache snapshot path, or should the heap stay on 4KiB pages?
- What is the exact current redis.conf default for hash-max-listpack-entries (docs show 512, config suggests 128)? Needs verification against config.c before adopting as a comparison baseline.

## Proposed issues (seeds for the tracker)

- **[decision, M1]** Decision: select default global allocator (tikv-jemallocator vs mimalloc vs snmalloc-rs vs custom slab). Choose IronCache #[global_allocator] (and any per-shard allocator) weighing introspection/defrag-hint support, RSS-decay behavior, and throughput-per-core. Document why; lean toward tikv-jemallocator 0.7.0 (jemalloc 5.3.1) for M2 unless benchmarks say otherwise.
- **[research, M0]** Research: benchmark jemalloc/mimalloc/snmalloc under a cache workload (throughput/core, p99.9, steady-state RSS, post-peak fragmentation). Build a representative harness (small mixed KV, eviction churn, fill-to-peak-then-free) and measure all three Rust allocator bindings; vendor microbenchmarks are not representative. Output feeds the allocator-selection decision.
- **[design, M1]** Design: online defragmentation strategy (jemalloc frag-hint object copy vs Mesh-style page remap vs native slab sparseness query). Specify how IronCache reclaims external fragmentation for long-lived data: port JEMALLOC_FRAG_HINT/je_get_defrag_hint, adopt Mesh-style physical-page remapping, or expose a native is-this-slab-sparse query. Include CPU-throttle and threshold defaults modeled on Redis (10%-100%, 1%-25% CPU).
- **[design, M1]** Design: compact value representation — inline small strings, pointer tagging/NaN-boxing, shared small integers, 8-byte size classes. Define IronCache per-object layout to minimize allocator overhead: SSO for short values, tagged pointers for small ints/floats, a shared-immutable-int cache, and malloc_usable_size-aware sizing on 8-byte-granularity size classes. Retrofitting later is costly, so settle the value ABI early.
- **[decision, M1]** Decision: THP and fork/snapshot stance (MADV_NOHUGEPAGE heap, non-fork serialization, startup THP=always warning). Decide how IronCache snapshots without COW amplification: avoid fork (async io_uring serialization or data-structure COW), or madvise(MADV_NOHUGEPAGE) the heap and warn when system THP=always, mirroring Redis echo never guidance.
- **[design, M2]** Design: memory accounting + maxmemory enforcement against allocator-attributed bytes. Account used_memory from allocator introspection (jemalloc mallctl stats.allocated / malloc_usable_size) rather than naive object sizes so maxmemory tracks RSS; pick a cache-appropriate default eviction policy (likely allkeys-lru, not noeviction) and expose mem_fragmentation_ratio.
- **[task, M2]** Task: tune purge/decay and background-thread defaults for low RSS without latency spikes. Sweep dirty_decay_ms/purge-delay and enable a background purge thread by default (upstream jemalloc default is off; Redis flips it on); document the RSS-vs-latency curve and choose IronCache defaults.
- **[meta, M0]** Meta: maintain a pinned prior-art table of allocator versions/defaults (jemalloc 5.3.x, mimalloc v3.x, snmalloc 0.7.x, Rust bindings). Keep a living table of upstream versions and key defaults (decay, tcache, narenas, THP, Redis active-defrag config) with source URLs and accessed dates, refreshed each milestone to prevent stale numbers in the vision EPIC and README.
