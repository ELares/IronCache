## Summary

"Keeping the Redis contracts" decomposes into three orthogonal surfaces that IronCache must treat separately: (1) the wire serialization (RESP2/RESP3), (2) the request/response interaction model (pipelining, inline commands, MULTI/EXEC, Pub/Sub push semantics, client-side-caching invalidation), and (3) the command surface (~240 core commands across ~20 groups, plus their exact reply types and error string conventions). The RESP3 spec (authored by Salvatore Sanfilippo, accepted, versions May 2018–Mar 2023) is small and stable: 15 base types, each identified by a single leading byte, length-prefixed for binary safety, CRLF-terminated. Critically, RESP3 is opt-in: a connection starts in RESP2 and only upgrades when the client sends HELLO 3; an unknown HELLO yields "-ERR unknown command" and NOPROTO is returned for unsupported versions. This negotiation contract is the single most load-bearing compatibility primitive — getting HELLO and per-connection protocol state right is what lets every unmodified client (redis-py, ioredis, Jedis, go-redis, StackExchange.Redis, redis-cli) connect.

The protocol itself is cheap to implement and cheap to parse. RESP is deliberately length-prefixed rather than delimiter-scanned, so a correct scalar parser is already near-optimal. Kevin Montrose's 2025 experiment building a Vector512/SWAR nearly-branchless RESP parser concluded it did NOT beat Garnet's scalar parser that special-cases common GET/SET shapes — Garnet held a 10–20% advantage across most workloads and the SIMD code was "disgustingly un-maintainable." The lesson for IronCache: the throughput bottleneck is not CRLF scanning but allocation, syscall batching, and command dispatch; invest in zero-copy slicing over the read buffer, a perfect-hash command dispatch table, and io_uring/readv batching rather than SIMD parsing theater. RESP3's streamed types ($?, *?, ~?, %? with a "." terminator) and attributes (|) are rarely used by clients and can be deferred.

The interaction model carries more semantic weight than the bytes. Pipelining is implicit (read N requests, write N replies in order) and is the highest-leverage throughput feature — it must be correct from day one. Inline commands (first byte != '*', space-split) exist mainly for telnet/redis-cli probing and a few health checks; cheap to support, low priority. MULTI/EXEC/DISCARD/WATCH give a queue-then-atomically-apply model with optimistic locking and explicitly NO rollback (runtime errors inside EXEC do not undo prior commands; EXEC returns an array, or null if a WATCHed key changed). This non-ACID semantics is a contract clients depend on — IronCache must replicate the exact behavior including dirty-CAS abort, not "improve" it. Pub/Sub (SUBSCRIBE/PSUBSCRIBE/PUBLISH plus sharded SSUBSCRIBE/SPUBLISH/SUNSUBSCRIBE, Redis 7.0+) and keyspace notifications (notify-keyspace-events flag string, __keyspace@db__ / __keyevent@db__ channels, off by default for CPU cost) are layered on the same push machinery.

Client-side caching is the most protocol-coupled feature and the one most worth getting right for a "most efficient cache" pitch, because it moves hot reads off the server entirely. CLIENT TRACKING ON [REDIRECT id] [BCAST] [OPTIN|OPTOUT] [NOLOOP] [PREFIX p] drives two designs: default mode keeps a server-side invalidation table (key -> client-id set, O(tracked_keys x clients) memory, evicts by sending spurious invalidations when full) versus BCAST mode (prefix subscriptions, near-zero server memory, non-overlapping prefixes). Invalidations arrive as RESP3 push messages on the same connection, or in RESP2 via a SUBSCRIBE to __redis__:invalidate on a second connection referenced by REDIRECT. The invalidation table is single-namespace across all DBs. This is a strong "borrow" — but IronCache should lead with BCAST + RESP3 push (bounded memory, single connection, race-free) and treat the per-client tracking table as the costly fallback.

Scripting splits into ephemeral EVAL/EVALSHA/SCRIPT LOAD (Lua 5.1, SHA1-keyed, lost on restart/failover, client must re-load) and Redis Functions (Redis 7.0+: FUNCTION LOAD with #!lua name=lib shebang, FCALL/FCALL_RO, register_function, persisted to AOF and replicated, no-writes flag for replicas). Both block the whole server during execution. For IronCache, an embedded Lua engine is a large surface with real semantic risk (KEYS/ARGV contract, redis.call error propagation, RESP-type conversion via redis.setresp). It is the cleanest candidate to TIER LATE or replace with a narrower native-atomic-op story, since most cache workloads never script. ACL/AUTH (Redis 6.0+: default user "on nopass ~* &* +@all", ACL SETUSER with +cmd/@cat/~keypattern/&channel/>password stored as SHA-256, requirepass legacy) and TLS are table-stakes for production but orthogonal to the cache hot path; AUTH/HELLO AUTH and a single-password mode are M1, full ACL categories M2+. Recommended tiering: Tier 0 (RESP2+RESP3 framing, HELLO, PING, AUTH, pipelining, inline, RESET, CLIENT basics) → Tier 1 (the ~80/20 data commands: strings/GET-SET-MSET-INCR/GETEX, expiry TTL/EXPIRE/PERSIST, generic DEL/EXISTS/TYPE/SCAN, hashes, lists, sets, sorted sets) → Tier 2 (Pub/Sub + sharded, transactions, client-side caching) → Tier 3 (keyspace notifications, full ACL/TLS, bitmaps/HLL/geo/streams) → Tier 4 / explicit non-goal candidates (Lua scripting/Functions, modules, cluster bus).

## Mechanisms: borrow, adapt, or reject

| Mechanism | System | Stance | What it does | Rationale for IronCache |
| --- | --- | --- | --- | --- |
| HELLO-based per-connection protocol negotiation | Redis RESP2/RESP3 | **borrow** | Connection starts in RESP2; client sends HELLO [3] [AUTH user pass] to negotiate RESP3 and receive a server-info map. Unknown HELLO -> -ERR unknown command (fallback RESP2); unsupported version -> -NOPROTO. | This is the load-bearing compatibility primitive that lets every unmodified client connect. IronCache must implement HELLO exactly, track protocol per-connection, and return a correct server-info map (server name, version, proto, id, mode, role). |
| Length-prefixed binary-safe framing (single-byte type markers, CRLF terminators) | Redis RESP | **borrow** | Every value is a type-byte + length-prefixed/CRLF-terminated payload, avoiding delimiter scanning of payload bytes. | Cheap, unambiguous, and already near-optimal to parse. Enables zero-copy slicing over the socket read buffer; the framing format is fixed by the contract and not something to redesign. |
| SIMD / Vector512 / SWAR nearly-branchless RESP request parser | Kevin Montrose experiment / Garnet comparison | **reject** | Uses vectorized digit-position bitmaps, SWAR integer parsing, and perfect-hash command identification to parse RESP with minimal branches. | Montrose's own benchmarks show it LOSES to Garnet's scalar parser that special-cases GET/SET (Garnet ~10-20% faster) and the code is 'disgustingly un-maintainable.' For IronCache the real wins are allocation avoidance, syscall batching (io_uring/readv), and dispatch — not SIMD CRLF scanning. Keep a clean scalar parser with hot-path special-casing instead. |
| Implicit pipelining (read N requests, reply N in order) | Redis request/response model | **borrow** | Client streams multiple commands without waiting; server replies in the same order, amortizing RTT. | Highest-leverage throughput feature and a hard client expectation. Must be correct from M0/M1; pairs naturally with batched writev/io_uring submission. |
| Inline commands (first byte != '*', space-split args) | Redis request/response model | **adapt** | Accepts plain text commands for telnet/redis-cli probing without RESP array framing. | Trivial to support and used by redis-cli/health checks, but binary-unsafe and single-command. Implement a minimal parser (PING, simple GET/SET) and document it as a probing convenience, not a primary path. |
| MULTI/EXEC/DISCARD/WATCH optimistic transactions with no rollback | Redis transactions | **borrow** | Queue commands, then apply atomically at EXEC; WATCH provides dirty-CAS abort (EXEC -> null). Runtime errors do not roll back already-applied commands. | Clients depend on the exact semantics including no-rollback and null-on-WATCH-conflict. Must replicate behavior precisely; the single-threaded-equivalent execution boundary maps cleanly onto a per-shard execution lock in IronCache. |
| Pub/Sub + sharded Pub/Sub over RESP3 push | Redis Pub/Sub (SUBSCRIBE/PSUBSCRIBE/SSUBSCRIBE) | **adapt** | Channel and pattern subscriptions; sharded variant (7.0+) confines messages to the owning shard in Cluster mode. Messages delivered as RESP3 push (>) or RESP2 multi-bulk. | Borrow the command surface and push delivery for compatibility, but Pub/Sub is fire-and-forget with no persistence. For a cache it is Tier 2; sharded Pub/Sub only matters once IronCache is multi-node. Reuse the same push machinery as client-side-caching invalidation. |
| CLIENT TRACKING default mode (server-side per-key invalidation table) | Redis client-side caching | **adapt** | Server records key -> set-of-client-ids for every read; on write sends targeted invalidations. Table is bounded and evicts by sending spurious invalidations when full. | Powerful but O(tracked_keys x clients) memory — at odds with IronCache's minimal-memory goal. Support for compatibility but make BCAST + RESP3 push the recommended/default path and the tracking table a costly opt-in with a hard memory cap. |
| CLIENT TRACKING BCAST + RESP3 invalidation push | Redis client-side caching | **borrow** | Prefix-based broadcast invalidation with near-zero server memory; non-overlapping prefixes; invalidations as same-connection RESP3 push messages. | Bounded memory, single race-free connection, directly serves the 'most efficient cache' pitch by moving hot reads to clients. This is the flagship client-side-caching mode IronCache should lead with. |
| RESP2 invalidation via __redis__:invalidate + REDIRECT | Redis client-side caching | **adapt** | RESP2 clients put invalidations on a Pub/Sub channel consumed by a second connection identified by CLIENT TRACKING REDIRECT id; FLUSHALL sends a null invalidation. | Needed for RESP2-only clients but adds a second-connection race window. Implement for compatibility but steer clients to RESP3 single-connection; reuse the exact __redis__:invalidate channel name and null-on-flush semantics. |
| EVAL/EVALSHA ephemeral Lua scripting + SCRIPT LOAD | Redis programmability | **reject** | Runs SHA1-cached Lua 5.1 scripts atomically with KEYS/ARGV; cache lost on flush/restart/failover. | Large, semantically risky surface (redis.call error propagation, RESP-type conversion, sandboxing) that most cache workloads never use. Declare a non-goal for the initial milestones; if demand appears, prefer a narrow set of native atomic ops over embedding a full Lua VM. |
| Redis Functions (FUNCTION LOAD / FCALL / libraries, persisted+replicated) | Redis programmability | **reject** | Named persistent Lua libraries replicated with data; FCALL_RO + no-writes flag for replicas. | Even heavier than EVAL and depends on durable replication semantics IronCache will not have at M0-M2. Non-goal initially; revisit only after core + clustering ship. |
| ACL users/categories/key-and-channel patterns + AUTH/requirepass | Redis security (ACL, 6.0+) | **adapt** | Per-user command/category/key-pattern/channel-pattern permissions with SHA-256 passwords; default user 'on nopass ~* &* +@all'; legacy requirepass. | Production needs AUTH and at least a single-password/default-user model early (M1). Full ACL category and selector machinery (~20 @categories, %R/%W, selectors) is large and orthogonal to the hot path — implement HELLO AUTH + requirepass + a minimal ACL in M1, full ACL in M2+. |
| TLS termination | Redis security | **adapt** | Optional TLS on the listener for encrypted client connections. | Table-stakes for production but orthogonal to protocol semantics. As a single static binary, IronCache should embed rustls behind a config flag rather than relying on stunnel sidecars; can land slightly after plaintext + AUTH. |
| RESP3 streamed strings/aggregates ($?, *?, ~?, %?) and attributes (\|) | Redis RESP3 spec | **adapt** | Unknown-length streamed payloads terminated by '.', plus out-of-band attribute metadata before a reply. | Rarely emitted by Redis and rarely consumed by clients; a parser must tolerate them but IronCache need not generate streamed types or attributes initially. Defer generation; ensure the parser does not choke if a client sends them. |

## Implications for IronCache

- Define an explicit compatibility tiering and publish it in the README so users know exactly what 'Redis-compatible' means: Tier 0 = wire+connection (RESP2/RESP3 framing, HELLO, AUTH, PING, RESET, pipelining, inline, CLIENT ID/SETNAME/GETNAME); Tier 1 = the 80/20 data surface (strings GET/SET/SETEX/SETNX/GETEX/MSET/MGET/INCR/DECR/APPEND/STRLEN, expiry EXPIRE/PEXPIRE/TTL/PTTL/PERSIST, generic DEL/UNLINK/EXISTS/TYPE/KEYS/SCAN/RENAME/COPY, hashes, lists, sets, sorted sets); Tier 2 = Pub/Sub + sharded, MULTI/EXEC/WATCH, CLIENT TRACKING; Tier 3 = keyspace notifications, full ACL, TLS, bitmaps/HLL/geo/streams; Tier 4/non-goals = Lua scripting/Functions, modules, cluster bus.
- Treat HELLO + per-connection protocol state as a single-owner state machine. Each connection holds: protocol (2|3), authenticated user, selected DB, subscribed channels/patterns, tracking config, MULTI queue. Get this object right and most compatibility falls out.
- Build a perfect-hash (or FST) command dispatch table keyed on the uppercased command token, with hot-path special-casing for GET/SET/MGET/INCR (Garnet's winning strategy) rather than a SIMD parser. Generate the table from a machine-readable command spec so reply-type and arity validation are data-driven.
- Make zero-copy the parser default: parse RESP requests as &[u8] slices into the connection read buffer, defer to owned allocation only when a value must outlive the request. Pair with batched writev / io_uring submission for replies — this is where pipelining throughput actually comes from.
- Lead client-side caching with BCAST + RESP3 same-connection push (bounded prefix memory, race-free) as the documented recommended mode; implement default per-client tracking and the RESP2 __redis__:invalidate+REDIRECT path for compatibility but cap the tracking table memory hard and expose a metric for invalidation-table size.
- Reuse one internal 'server push' channel abstraction for all out-of-band delivery: Pub/Sub messages, sharded Pub/Sub, keyspace notifications, and client-side-caching invalidations. They all reduce to RESP3 push (>) on RESP3 connections and multi-bulk on RESP2.
- Replicate transaction semantics exactly, including NO rollback and null-on-WATCH-conflict. Model EXEC as 'acquire the shard execution boundary, replay queued commands, collect per-command replies into an array.' Do not 'improve' the contract — clients rely on it.
- Ship AUTH + requirepass + HELLO AUTH and a minimal single-/default-user ACL in M1; defer the full ~20 @category + %R/%W + selector machinery to M2. Store passwords as SHA-256 to match ACL GETUSER output expectations.
- Embed TLS (rustls) behind a config flag to honor the single-static-binary goal, rather than requiring an stunnel/envoy sidecar; keep it orthogonal to the protocol layer.
- Make the parser tolerant of RESP3 features IronCache does not itself emit (attributes |, streamed types $?/*?/~?/%? with '.' terminator, big numbers) so that aggressive clients negotiating RESP3 never trip the connection, even if IronCache never generates those types initially.
- Match Redis error-string conventions precisely (leading uppercase token: ERR, WRONGTYPE, NOPROTO, NOAUTH, NOPERM, EXECABORT, MOVED/ASK later) because some clients pattern-match on these prefixes; treat the error catalog as part of the wire contract.
- Note that modern official clients (redis-py 8, node-redis 6) now negotiate RESP3 by default, so IronCache's RESP3 path is a first-class, frequently-exercised code path from day one — not an optional extra.

## Research papers and primary sources

- **A (Nearly) Branchless RESP Request Parser** (Kevin Montrose blog, 2025). Builds a Vector512/SWAR nearly-branchless RESP parser (vectorized digit bitmaps, branchless integer parsing, perfect-hash command identification) and benchmarks it against Garnet's scalar parser. [source](https://kevinmontrose.com/2025/09/15/a-nearly-branchless-resp-request-parser/) Relevance: Directly answers IronCache's 'should we SIMD the RESP parser?' question with a negative result: the SIMD/branchless parser was slower than Garnet's scalar GET/SET-special-cased parser and was unmaintainable. Steers IronCache toward dispatch/allocation/syscall optimization instead.
- **Introducing Garnet: a faster cache-store (Microsoft Research)** (Microsoft Research blog / Garnet project, 2024). A RESP-compatible cache-store that adopts the RESP wire protocol so unmodified Redis clients work, and beats peers on throughput/latency via a network/storage architecture rather than protocol tricks. [source](https://www.microsoft.com/en-us/research/blog/introducing-garnet-an-open-source-next-generation-faster-cache-store-for-accelerating-applications-and-services/) Relevance: Demonstrates the exact thesis IronCache is built on — adopt RESP for client compatibility, then win on the engine. Garnet's command-special-casing parser is the performance baseline to match/beat.
- **RESP3 Specification** (redis/redis-specifications (Sanfilippo), 2018-2023). The canonical, accepted RESP3 protocol spec: 15 types with single-byte markers, opt-in via HELLO to preserve RESP2 compatibility, plus streamed types and attributes. [source](https://github.com/redis/redis-specifications/blob/master/protocol/RESP3.md) Relevance: The authoritative contract IronCache's serialization layer must implement; defines exactly what RESP3 compatibility means and why negotiation must be opt-in.
- **Dragonfly: a modern replacement for Redis and Memcached** (dragonflydb/dragonfly, 2022-present). A multi-threaded, shared-nothing cache implementing RESP2/RESP3 and ~240 Redis commands, treating ~240 commands as the meaningful-compatibility threshold. [source](https://github.com/dragonflydb/dragonfly) Relevance: Validates IronCache's ~240-command 80/20 target and the RESP2+RESP3 dual-protocol requirement as the bar for 'drop-in Redis replacement.'

## Open questions

- What exactly is IronCache's RESP3 emission policy: should it emit native RESP3 aggregate types (map/set/double) for commands like HGETALL/CONFIG GET/SMEMBERS when proto=3, or always emit RESP2-shaped replies and only differ on null? (Redis differs per-command; emitting RESP2 shapes under RESP3 risks subtle client breakage.)
- Should the tracking table for default-mode client-side caching live per-shard or be a global structure, and what is its hard memory cap and eviction policy (spurious-invalidation eviction like Redis, or LRU) given IronCache's minimal-memory goal?
- Is multi-DB / SELECT supported, and if so does client-side-caching invalidation use Redis's single-cross-DB-namespace behavior or a per-DB namespace? This affects both the tracking design and SELECT semantics.
- What is the precise scope of Lua: confirm scripting/Functions as an explicit non-goal for M0-M2, or carve out a tiny set of server-side atomic primitives (e.g., compare-and-set, atomic counters with TTL) that cover the common scripting use cases without a Lua VM?
- How faithfully must error strings match Redis (exact wording vs just the leading uppercase token)? Need a test corpus of clients that pattern-match on error text to set the bar.
- For multi-node IronCache, does sharded Pub/Sub (SSUBSCRIBE/SPUBLISH) and MOVED/ASK redirection get implemented, and does that pull in the cluster-bus protocol — or does IronCache use a different topology that still presents the Redis Cluster client contract?
- Which CLIENT subcommands are in Tier 0 (ID, SETNAME, GETNAME, NO-EVICT, NO-TOUCH, INFO, LIST, KILL, UNPAUSE) versus deferred, given that several clients issue CLIENT SETINFO/SETNAME automatically on connect?
